uv — 차세대 Python 패키지 관리자

핵심 개념: uv란 무엇인가

uv는 Astral사가 Rust로 개발한 올인원 Python 패키지 관리자이다. pip, pip-tools, venv, poetry, pyenv의 기능을 하나로 통합하면서도, Rust의 병렬 처리 능력 덕분에 pip 대비 10~100배 빠른 의존성 해석과 설치 속도를 제공한다. MCP 공식 문서에서도 uv를 기본 도구로 권장하고 있다.

uv vs pip vs poetry 비교

pip + venv (기존)
# 가상환경 수동 생성
python3 -m venv .venv
source .venv/bin/activate

# 패키지 설치
pip install mcp httpx

# 의존성 고정
pip freeze > requirements.txt

가상환경 관리, 의존성 해석, 잠금 파일 생성이 모두 분리되어 있다. Python 버전 관리는 별도 도구(pyenv)가 필요하다.

uv (권장)
# 프로젝트 초기화 (가상환경 자동 생성)
uv init mcp-demo
cd mcp-demo

# 패키지 설치 + 잠금 파일 자동 생성
uv add mcp httpx

# 실행 (가상환경 자동 활성화)
uv run server.py

가상환경 생성, 의존성 해석, 잠금 파일, Python 버전 관리까지 전부 자동이다. 속도는 pip의 10~100배.

설치

curl -LsSf https://astral.sh/uv/install.sh | sh
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
brew install uv

설치 후 터미널을 재시작하고 버전을 확인한다.

uv --version
# 출력 예시: uv 0.7.x

핵심 명령어 레퍼런스

명령어설명예시
uv init 새 프로젝트 초기화. pyproject.toml, .python-version, .venv/ 자동 생성 uv init mcp-demo
uv add 패키지를 의존성에 추가하고 설치. pyproject.toml과 uv.lock 자동 갱신 uv add "mcp[cli]" httpx
uv remove 패키지를 의존성에서 제거 uv remove httpx
uv run 가상 환경 내에서 명령 실행. 환경을 수동으로 활성화할 필요 없다 uv run server.py
uv lock 의존성을 해석하여 uv.lock 잠금 파일을 생성/갱신 uv lock
uv sync uv.lock 기준으로 가상 환경을 동기화 (CI/CD에서 유용) uv sync
uv python install 특정 Python 버전을 설치 (pyenv 없이 Python 버전 관리 가능) uv python install 3.12
Deep Dive: uv의 가상 환경 자동 관리

uv init을 실행하면 프로젝트 루트에 .venv/ 디렉토리가 자동으로 생성된다. 이후 uv run을 사용하면 이 가상 환경이 자동으로 활성화된 상태에서 명령이 실행된다. source .venv/bin/activate를 수동으로 입력할 필요가 전혀 없다. 만약 .venv/가 삭제되었거나 Python 버전이 변경되었다면, uv sync가 자동으로 가상 환경을 재생성한다.

프로젝트 초기화

uv를 사용하여 MCP 서버 프로젝트를 처음부터 만들어보겠다.

  1. 1
    프로젝트 생성 
    # 새 프로젝트 디렉토리 생성 및 초기화
uv init mcp-demo
cd mcp-demo

    uv init은 pyproject.toml, .python-version, main.py, .venv/를 자동으로 생성한다.

  2. 2
    MCP SDK 설치 
    # MCP SDK + CLI 도구 설치
uv add "mcp[cli]"

    "mcp[cli]"는 MCP Python SDK와 함께 mcp dev, mcp install 등의 CLI 명령을 포함한다. 대괄호 안의 cli는 extras(선택적 의존성 그룹)를 의미한다.

  3. 3
    추가 패키지 설치 (선택) 
    # HTTP 요청이 필요한 도구를 만들 경우
uv add httpx

# 데이터 검증이 필요한 경우 (MCP SDK에 이미 포함되어 있지만 직접 사용할 때)
uv add pydantic
  4. 4
    pyproject.toml 확인

    설치가 완료되면 pyproject.toml에 의존성이 기록된다.

    [project]
name = "mcp-demo"
version = "0.1.0"
description = ""
readme = "README.md"
requires-python = ">=3.12"
dependencies = [
    "mcp[cli]>=1.12.0",
    "httpx>=0.28.0",
]
Deep Dive: pyproject.toml 구조

pyproject.toml은 PEP 621에 의해 표준화된 Python 프로젝트 설정 파일이다. [project] 테이블에는 프로젝트 이름, 버전, Python 버전 요구사항, 의존성이 들어간다. uv add를 실행하면 dependencies 리스트가 자동으로 갱신되며, 동시에 uv.lock 파일이 생성되어 정확한 버전이 고정(pin)된다. 이 잠금 파일을 Git에 커밋하면 팀원 모두가 동일한 환경을 재현할 수 있다.

MCP Inspector 설치 및 사용법

핵심 개념: MCP Inspector란

MCP Inspector는 MCP 서버를 시각적으로 테스트할 수 있는 웹 기반 개발 도구이다. 서버에 등록된 Tools, Resources, Prompts를 탐색하고, 직접 호출하여 결과를 확인할 수 있다. LLM 없이도 서버의 동작을 검증할 수 있으므로, 개발 중 가장 먼저 사용하게 되는 디버깅 도구이다.

실행 방법

Inspector는 별도 설치 없이 npx로 바로 실행하거나, MCP CLI의 dev 명령으로 서버와 함께 실행할 수 있다.

# MCP CLI의 dev 명령으로 서버 + Inspector 동시 실행
uv run mcp dev server.py

# 출력 예시:
# Starting MCP Inspector...
# Proxy server listening on port 6277
# 🔍 MCP Inspector: http://localhost:5173

이 방법이 가장 간편하다. 서버 프로세스와 Inspector가 함께 시작되며, 브라우저에서 http://localhost:5173을 열면 즉시 테스트할 수 있다.

# Inspector만 독립적으로 실행
npx @modelcontextprotocol/inspector

# 이미 실행 중인 서버에 연결할 때 유용하다
# Inspector UI에서 Transport 설정을 변경하여 연결 가능

Inspector UI 구성 요소

Inspector는 크게 다음과 같은 영역으로 구성된다.

영역설명
연결 상태 표시 상단에 서버 연결 상태(Connected/Disconnected)를 표시한다. Transport 유형(stdio/SSE/Streamable HTTP)과 서버 이름이 나타난다.
Tools 탭 서버에 등록된 도구 목록을 보여준다. 도구를 선택하면 입력 스키마가 표시되고, 파라미터를 직접 입력하여 실행할 수 있다. 결과는 JSON 형태로 하단에 출력된다.
Resources 탭 서버가 노출하는 리소스 목록을 보여준다. URI를 클릭하면 리소스의 내용을 읽어올 수 있다. 리소스 템플릿의 경우 URI 파라미터를 입력할 수 있는 폼이 제공된다.
Prompts 탭 서버에 등록된 프롬프트 템플릿 목록이다. 인자를 입력하면 생성되는 메시지 배열을 미리 확인할 수 있다.
Notifications / Logs 서버와 클라이언트 사이에 오가는 JSON-RPC 메시지를 실시간으로 모니터링한다. 디버깅에 매우 유용하다.
TIP

Inspector의 Notifications 패널은 서버가 보내는 모든 JSON-RPC 메시지를 원본 그대로 보여준다. 프로토콜 레벨에서 어떤 데이터가 오가는지 확인하고 싶을 때 이 패널을 활용하면 된다.

MCP 서버 등록

설정 파일을 열고 다음과 같이 작성한다. 파일이 없으면 새로 생성한다.

{
  "mcpServers": {
    "mcp-demo": {
      "command": "uv",
      "args": [
        "run",
        "--directory", "/absolute/path/to/mcp-demo",
        "server.py"
      ]
    }
  }
}
WARNING

--directory 경로는 반드시 절대 경로를 사용해야 한다. ~이나 상대 경로는 Claude Desktop에서 인식하지 못한다. macOS 예시: /Users/username/projects/mcp-demo

각 필드의 의미는 다음과 같다.

필드설명
mcpServers MCP 서버 목록을 담는 최상위 객체이다. 키 이름이 서버 식별자가 된다.
command 서버를 시작할 명령어이다. uv로 실행하므로 "uv"를 지정한다.
args 명령어에 전달할 인자 배열이다. --directory로 프로젝트 디렉토리를 지정하면 해당 디렉토리의 가상 환경이 사용된다.

환경 변수 전달

API 키 등 환경 변수가 필요한 서버의 경우 env 필드를 추가한다.

{
  "mcpServers": {
    "weather-server": {
      "command": "uv",
      "args": [
        "run",
        "--directory", "/absolute/path/to/weather-server",
        "server.py"
      ],
      "env": {
        "WEATHER_API_KEY": "your-api-key-here"
      }
    }
  }
}

Deep Dive: Claude Desktop의 MCP 서버 관리 방식

Claude Desktop은 설정 파일에 등록된 각 MCP 서버를 별도의 자식 프로세스로 실행한다. 서버와는 stdio(표준 입출력) 전송 방식으로 통신하며, 서버의 stdout으로 JSON-RPC 메시지를 주고받는다. 따라서 서버 코드에서 print()를 사용하면 프로토콜 메시지가 깨질 수 있다. 디버깅용 출력은 반드시 stderr로 보내야 한다 (import sys; print("debug", file=sys.stderr)).

권장 프로젝트 구조

MCP 서버 프로젝트가 커질수록 파일을 체계적으로 관리하는 것이 중요하다. 아래는 권장하는 디렉토리 구조이다.

mcp-demo/
├── .python-version          # Python 버전 (uv가 관리)
├── .venv/                   # 가상 환경 (uv가 자동 생성, .gitignore에 추가)
├── pyproject.toml           # 프로젝트 메타데이터 및 의존성
├── uv.lock                  # 의존성 잠금 파일 (Git에 커밋)
├── server.py                # MCP 서버 엔트리포인트
├── tools/                   # 도구 모듈 (기능별 분리)
│   ├── __init__.py
│   ├── math_tools.py
│   └── api_tools.py
├── resources/               # 리소스 모듈
│   ├── __init__.py
│   └── config_resources.py
├── prompts/                 # 프롬프트 모듈
│   ├── __init__.py
│   └── review_prompts.py
└── tests/                   # 테스트
    ├── __init__.py
    └── test_tools.py
파일/디렉토리역할
server.py FastMCP 인스턴스를 생성하고 mcp.run()을 호출하는 진입점이다. 도구/리소스/프롬프트를 이 파일에 직접 정의하거나, 별도 모듈에서 import한다.
tools/ 도구 함수들을 기능별로 분리한다. 서버가 커지면 한 파일에 모든 도구를 두기 어렵다.
uv.lock 정확한 의존성 버전을 고정하는 잠금 파일이다. 반드시 Git에 커밋하여 환경 재현성을 보장한다.
.venv/ uv가 자동 생성하는 가상 환경이다. .gitignore에 추가하여 버전 관리에서 제외한다.

설치 검증

모든 설치가 완료되었는지 한 번에 확인하는 스크립트이다. check_env.py 파일을 만들고 실행해보자.

"""MCP 개발 환경 검증 스크립트"""
import sys
import shutil


def check(name: str, condition: bool, detail: str = ""):
    status = "OK" if condition else "FAIL"
    msg = f"[{status}] {name}"
    if detail:
        msg += f" - {detail}"
    print(msg)
    return condition


all_ok = True

# 1. Python 버전
py_ver = sys.version_info
all_ok &= check(
    "Python 버전",
    py_ver >= (3, 10),
    f"{py_ver.major}.{py_ver.minor}.{py_ver.micro}"
)

# 2. MCP SDK
try:
    import mcp
    all_ok &= check("MCP SDK", True, f"v{mcp.__version__}")
except ImportError:
    all_ok &= check("MCP SDK", False, "설치 필요: uv add 'mcp[cli]'")
