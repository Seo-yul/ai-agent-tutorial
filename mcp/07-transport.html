7.1 Transport란?

핵심 개념: Transport 계층

Transport는 MCP 아키텍처에서 가장 아래쪽에 위치하는 물리적 메시지 전달 계층이다. 애플리케이션(도구 호출, 리소스 읽기 등)이 만들어 낸 JSON-RPC 메시지를 네트워크나 프로세스 간 파이프를 통해 상대편으로 실어 나르는 역할을 한다. Transport는 메시지의 의미를 해석하지 않으며, 오직 바이트를 전달하는 데만 관심을 가진다.

MCP의 계층 구조를 이해하면 Transport의 위치가 명확해진다. 최상위에는 MCP 서버의 비즈니스 로직(도구, 리소스, 프롬프트)이 있고, 중간에 JSON-RPC 프로토콜이 메시지 포맷을 정의하며, 최하위에서 Transport가 메시지를 물리적으로 운반한다. 

graph TB
    subgraph APP ["Application 계층"]
        direction LR
        T["Tools"] ~~~ R["Resources"] ~~~ P["Prompts"]
    end
    subgraph PROTO ["Protocol 계층"]
        JSONRPC["JSON-RPC 2.0\n메시지 포맷 정의\n(Request / Response / Notification)"]
    end
    subgraph TRANS ["Transport 계층"]
        direction LR
        STDIO["stdio\n(stdin/stdout)"] ~~~ HTTP["Streamable HTTP\n(HTTP POST + SSE)"]
    end
    APP --> PROTO --> TRANS

MCP Python SDK는 두 가지 Transport를 기본 제공한다.

Transport통신 채널대표 사용처
stdio 프로세스의 stdin / stdout 파이프 Claude Desktop, IDE 플러그인, 로컬 CLI
Streamable HTTP HTTP POST + Server-Sent Events 원격 서버, 웹 서비스, 클라우드 배포
WARNING

이전에 지원되던 sse Transport(transport="sse")는 deprecated 되었다. 모든 SSE 기반 통신은 streamable-http로 마이그레이션해야 한다.

7.2 JSON-RPC 메시지 구조

MCP의 모든 통신은 JSON-RPC 2.0 규격을 따른다. Transport가 운반하는 것은 결국 이 JSON-RPC 메시지이므로, 메시지 구조를 정확히 이해해야 디버깅과 커스텀 클라이언트 구현이 가능하다. JSON-RPC에는 세 가지 메시지 유형이 있다.

Request (요청)

클라이언트가 서버에 작업을 요청할 때 보내는 메시지이다. id 필드가 반드시 포함되며, 서버는 동일한 id로 응답한다.

→ Request — 클라이언트 → 서버 
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "add",
    "arguments": { "a": 5, "b": 3 }
  }
}

Response (응답)

서버가 요청에 대해 반환하는 메시지이다. 성공 시 result, 실패 시 error 필드를 포함한다.

성공 응답 (result)
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "8"
      }
    ]
  }
}
에러 응답 (error)
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32602,
    "message": "Invalid params",
    "data": "Missing required argument: a"
  }
}

Notification (알림)

응답을 기대하지 않는 단방향 메시지이다. id 필드가 없다는 것이 핵심적인 차이이다.

↓ Notification — 서버 → 클라이언트 (응답 없음) 
{
  "jsonrpc": "2.0",
  "method": "notifications/tools/list_changed",
  "params": {}
}
Deep Dive: JSON-RPC 에러 코드

JSON-RPC 2.0은 표준 에러 코드를 정의하며, MCP에서도 동일하게 사용된다.

코드이름의미
-32700Parse errorJSON 파싱 실패 (잘못된 JSON)
-32600Invalid Request유효한 JSON이지만 JSON-RPC 규격에 맞지 않음
-32601Method not found요청한 메서드가 존재하지 않음
-32602Invalid params메서드 파라미터가 올바르지 않음
-32603Internal error서버 내부 오류
-32000 ~ -32099Server error구현에 따른 커스텀 서버 에러

도구 실행 자체의 에러(예: 계산 실패)는 JSON-RPC 에러가 아니라, result 안에 isError: true로 반환된다는 점에 주의한다.

메시지 흐름 전체 예시

클라이언트가 서버에 연결하여 도구를 호출하는 전체 흐름을 살펴보자.

sequenceDiagram
    participant C as Client
    participant S as Server

    C->>S: initialize (id:1)
    S-->>C: result: capabilities (id:1)
    C->>S: notifications/initialized
    Note right of S: Notification이므로 응답 없음

    C->>S: tools/list (id:2)
    S-->>C: result: [add, multiply, ...] (id:2)

    C->>S: tools/call "add" (id:3)
    S-->>C: result: "8" (id:3)

7.3 stdio Transport — 심화

핵심 개념: stdio Transport

stdio Transport는 클라이언트가 서버를 자식 프로세스(child process)로 실행하고, 해당 프로세스의 stdin/stdout 파이프를 통해 JSON-RPC 메시지를 줄 단위(newline-delimited)로 교환하는 방식이다. 네트워크가 필요 없고, 프로세스 격리로 보안이 단순하다.

내부 작동 원리

  1. 1
    프로세스 시작 클라이언트(예: Claude Desktop)가 설정된 command와 args로 서버 프로세스를 spawn한다.
  2. 2
    파이프 연결 클라이언트의 출력이 서버의 stdin으로, 서버의 stdout이 클라이언트의 입력으로 연결된다.
  3. 3
    메시지 교환 각 JSON-RPC 메시지는 한 줄의 JSON 문자열로 전송된다. 줄바꿈(\n)이 메시지 구분자 역할을 한다.
  4. 4
    종료 클라이언트가 stdin을 닫거나 프로세스를 종료(kill)하면 연결이 끊어진다. 
sequenceDiagram
    participant Host as 클라이언트 (Host)
    participant Proc as 서버 프로세스

    Host->>Proc: spawn("uv", ["run", "server.py"])
    Note over Host,Proc: stdin/stdout 파이프 연결

    Host->>Proc: stdin: {"jsonrpc":"2.0","id":1,"method":"initialize",...}
    Proc-->>Host: stdout: {"jsonrpc":"2.0","id":1,"result":{...}}

    Host->>Proc: stdin: {"jsonrpc":"2.0","id":2,"method":"tools/call",...}
    Proc-->>Host: stdout: {"jsonrpc":"2.0","id":2,"result":{...}}

    Note over Proc: stderr: 디버그 로그 (프로토콜에 영향 없음)

    Host->>Proc: close stdin / kill process

왜 print()를 쓰면 안 되는가

WARNING: stdout 오염 문제

stdio Transport에서 stdout은 프로토콜 전용 채널이다. 서버 코드에서 print()를 호출하면 JSON-RPC 메시지가 아닌 텍스트가 stdout에 섞여 들어가고, 클라이언트는 이를 파싱하려다 실패하여 연결이 즉시 끊어진다.

잘못된 예 (print 사용)
@mcp.tool()
def calculate(expr: str) -> str:
    # stdout에 출력 → 프로토콜 오염!
    print(f"계산 요청: {expr}")
    return str(eval(expr))
올바른 예 (stderr 또는 logging)
import sys
import logging

logger = logging.getLogger(__name__)

@mcp.tool()
def calculate(expr: str) -> str:
    # stderr로 출력 (프로토콜에 영향 없음)
    print(f"계산 요청: {expr}", file=sys.stderr)
    # 또는 logging 모듈 사용
    logger.info(f"계산 요청: {expr}")
    return str(eval(expr))

기본 서버 코드

    """인사 메시지를 반환한다"""

# stdio로 실행 (기본값)
mcp.run()

7.4 Streamable HTTP Transport — 심화

핵심 개념: Streamable HTTP Transport

Streamable HTTP Transport는 HTTP POST로 요청을 보내고, 서버가 SSE(Server-Sent Events) 또는 일반 JSON 응답으로 결과를 반환하는 방식이다. 표준 HTTP 인프라(로드 밸런서, 리버스 프록시, CDN)를 그대로 활용할 수 있어 원격 배포에 적합하다.

내부 작동 원리

Streamable HTTP는 다음과 같은 흐름으로 동작한다.

  1. 1
    초기화 요청 클라이언트가 POST /mcp로 initialize 메시지를 보낸다. 서버는 Mcp-Session-Id 헤더와 함께 capabilities를 응답한다.
  2. 2
    세션 유지 이후 모든 요청에 Mcp-Session-Id 헤더를 포함하여 세션을 식별한다.
  3. 3
    요청/응답 클라이언트가 POST /mcp로 JSON-RPC 메시지를 보내면, 서버는 SSE 스트림 또는 단일 JSON으로 응답한다. SSE를 사용하면 중간 진행 상황이나 다수의 결과를 스트리밍으로 전달할 수 있다.
  4. 4
    서버 알림 수신 (선택) 클라이언트가 GET /mcp로 SSE 스트림을 열어두면, 서버가 비동기적으로 Notification을 푸시할 수 있다. 
sequenceDiagram
    participant C as Client
    participant S as Server (HTTP)

    C->>S: POST /mcp {initialize}
    S-->>C: 200 OK + Mcp-Session-Id: abc123
    Note over C: 이후 모든 요청에 세션 ID 포함

    C->>S: POST /mcp {tools/list}
Mcp-Session-Id: abc123
    S-->>C: 200 OK (JSON 또는 SSE)

    C->>S: POST /mcp {tools/call}
Mcp-Session-Id: abc123
    S-->>C: SSE stream (진행 상황 + 최종 결과)

    opt 서버 알림 수신
        C->>S: GET /mcp (SSE 스트림 열기)
        S-->>C: event: message (Notification)
    end
Deep Dive: SSE(Server-Sent Events)와의 관계

SSE는 HTTP 응답을 끊지 않고 서버가 클라이언트에 이벤트를 지속적으로 보내는 웹 표준이다. Streamable HTTP Transport는 Content-Type: text/event-stream으로 응답하여, 하나의 HTTP 요청 안에서 여러 JSON-RPC 메시지를 순차적으로 전송할 수 있다. 이를 통해 도구 실행 중간에 로그 메시지를 보내거나, 긴 작업의 진행 상황을 실시간으로 알릴 수 있다.

SSE 스트림 내의 각 이벤트는 다음 형식을 따른다: 

event: message
data: {"jsonrpc":"2.0","id":3,"result":{"content":[{"type":"text","text":"8"}]}}

event: message
data: {"jsonrpc":"2.0","method":"notifications/message","params":{"level":"info","data":"처리 완료"}}

기본 서버 코드

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("my-server")

@mcp.tool()
def hello(name: str) -> str:
    """인사 메시지를 반환한다"""
    return f"안녕, {name}!"

@mcp.tool()
def add(a: float, b: float) -> str:
    """두 수를 더한다"""
    return str(a + b)

# Streamable HTTP로 실행
mcp.run(transport="streamable-http")

위 코드를 실행하면 http://0.0.0.0:8000/mcp에 MCP 엔드포인트가 열린다.

설정 옵션 상세

# 모든 옵션을 명시적으로 지정
mcp.run(
    transport="streamable-http",
    host="0.0.0.0",          # 바인딩할 호스트 (기본: "0.0.0.0")
    port=8000,               # 바인딩할 포트 (기본: 8000)
    path="/mcp",             # MCP 엔드포인트 경로 (기본: "/mcp")
    log_level="info",        # 로깅 레벨 (기본: "info")
    stateless_http=False,    # Stateless 모드 (기본: False)
)
옵션기본값설명

port8000리스닝 포트 번호

log_level"info"Uvicorn 로깅 레벨 ("debug", "info", "warning", "error")
stateless_httpFalseTrue로 설정하면 세션 없이 동작 (7.5절 참고)

Mcp-Session-Id와 세션 관리

Streamable HTTP는 기본적으로 Stateful하다. 서버는 initialize 응답에 Mcp-Session-Id 헤더를 포함하고, 클라이언트는 이후 모든 요청에 이 헤더를 포함해야 한다. 세션이 유효하지 않으면 서버는 404 Not Found를 반환한다.

← Response Headers — initialize 응답 
HTTP/1.1 200 OK
Content-Type: text/event-stream
Mcp-Session-Id: a1b2c3d4-e5f6-7890-abcd-ef1234567890

7.5 Stateless HTTP 모드

핵심 개념: Stateless 모드

Stateless 모드에서는 서버가 세션을 유지하지 않는다. Mcp-Session-Id 헤더를 발급하지 않으며, 모든 요청은 독립적으로 처리된다. 이 모드는 수평 확장(scale-out)과 서버리스 환경에 최적화되어 있다. 

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("stateless-server")

@mcp.tool()
def translate(text: str, target_lang: str) -> str:
    """텍스트를 번역한다 (상태 비유지)"""
    # 각 요청이 독립적이므로 세션 컨텍스트에 의존하지 않는다
    return f"[{target_lang}] {text} (번역됨)"

@mcp.tool()
def calculate(expression: str) -> str:
    """수식을 계산한다"""
    try:
        result = eval(expression, {"__builtins__": {}})
        return f"{expression} = {result}"
    except Exception as e:
        return f"계산 오류: {e}"

# Stateless HTTP 모드로 실행
mcp.run(
    transport="streamable-http",
    host="0.0.0.0",
    port=8000,
    stateless_http=True,
)

서버리스 환경 적합성

Stateless 모드는 AWS Lambda, Google Cloud Run, Vercel Functions 등 요청이 올 때마다 인스턴스가 생성되는 서버리스 환경에 적합하다. 각 요청이 완전히 독립적이므로, 어떤 인스턴스가 요청을 처리해도 동일한 결과를 보장한다.

서버의 모든 도구가 외부 API 호출이나 단순 계산처럼 상태에 의존하지 않는다면 Stateless 모드가 운영 부담을 크게 줄여준다. 반대로 대화 이력이나 리소스 구독이 필요하다면 반드시 Stateful 모드를 사용한다.

7.6 Transport 선택 가이드

어떤 Transport를 선택할지는 배포 환경과 요구사항에 따라 결정한다.

stdio
  • 로컬 전용 (같은 머신)
  • 네트워크 설정 불필요
  • 1:1 연결 (클라이언트 1개)
  • 프로세스 격리로 보안 단순
  • Claude Desktop, Cursor, VS Code
  • 설정 복잡도: 매우 낮음
Streamable HTTP
  • 원격 접근 가능 (다른 머신)
  • 표준 HTTP 인프라 활용
  • 다중 클라이언트 동시 접속
  • HTTPS + 인증 구성 필요
  • 웹 앱, API 서버, 클라우드
  • 설정 복잡도: 중간

    Q{"서버를 어디서 실행하는가?"}

TIP

개발 중에는 stdio를 사용하고, 프로덕션 배포 시에는 Streamable HTTP로 전환하는 전략이 일반적이다. MCP 서버 코드를 수정할 필요 없이 mcp.run()의 transport 파라미터만 바꾸면 된다.

7.7 프로덕션 배포

Streamable HTTP Transport를 프로덕션 환경에 배포할 때는 HTTPS, 리버스 프록시, 컨테이너화 등 추가 인프라 구성이 필요하다. 이 절에서는 실무에서 자주 사용되는 배포 패턴을 다룬다.

배포 아키텍처

graph LR
    Client["MCP 클라이언트\n(HTTPS)"]
    Nginx["Nginx\n(리버스 프록시\n+ TLS 종단)"]
    MCP["MCP 서버\n(Docker 컨테이너\nHTTP :8000)"]

    Client -->|"HTTPS :443"| Nginx -->|"HTTP :8000"| MCP

이 패턴에서 TLS(HTTPS) 종단은 Nginx가 담당하고, MCP 서버는 평문 HTTP로만 동작한다. 이렇게 하면 인증서 관리를 Nginx에 집중할 수 있고, MCP 서버 코드는 전혀 변경할 필요가 없다.

Docker 배포

FROM python:3.12-slim

WORKDIR /app

# 의존성 설치
COPY pyproject.toml uv.lock ./
RUN pip install uv && uv sync --frozen

# 서버 코드 복사
COPY server.py .

# MCP 서버 실행
EXPOSE 8000
CMD ["uv", "run", "server.py"]
services:
  mcp-server:
    build: .
    ports:
      - "8000:8000"
    environment:
      - API_KEY=${API_KEY}
      - DATABASE_URL=${DATABASE_URL}
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/mcp"]
      interval: 30s
      timeout: 10s
      retries: 3

  nginx:
    image: nginx:alpine
    ports:
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf
      - ./certs:/etc/nginx/certs
    depends_on:
      - mcp-server

Nginx 리버스 프록시 설정

SSE 스트리밍을 지원하려면 Nginx에서 버퍼링을 비활성화해야 한다.

server {
    listen 443 ssl;
    server_name mcp.example.com;

    ssl_certificate     /etc/nginx/certs/cert.pem;
    ssl_certificate_key /etc/nginx/certs/key.pem;

    location /mcp {
        proxy_pass http://mcp-server:8000;

        # SSE 스트리밍 지원을 위해 버퍼링 비활성화
        proxy_buffering off;
        proxy_cache off;

        # SSE 연결을 위한 타임아웃 설정
        proxy_read_timeout 300s;
        proxy_send_timeout 300s;

        # 필요한 헤더 전달
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE를 위한 Connection 헤더
        proxy_set_header Connection '';
        proxy_http_version 1.1;
        chunked_transfer_encoding off;
    }
}
WARNING

proxy_buffering off는 SSE 스트리밍에 필수적이다. 이 설정이 없으면 Nginx가 서버 응답을 버퍼에 모아둔 후 한꺼번에 전송하므로, 클라이언트가 실시간 이벤트를 받지 못한다.

인증 미들웨어

프로덕션 환경에서는 MCP 서버에 인증을 추가해야 한다. MCP Python SDK는 OAuth 2.1 기반 인증을 내장 지원하며, 자세한 내용은 Chapter 11. OAuth 인증에서 다룬다. 여기서는 간단한 Bearer 토큰 검증 패턴만 소개한다. 

from starlette.applications import Starlette
from starlette.middleware import Middleware
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import JSONResponse
import os

class AuthMiddleware(BaseHTTPMiddleware):
    """간단한 Bearer 토큰 인증 미들웨어"""
    async def dispatch(self, request: Request, call_next):
        # health check 등은 인증 제외
        if request.url.path == "/health":
            return await call_next(request)

        auth_header = request.headers.get("Authorization", "")
        expected_token = os.environ.get("MCP_AUTH_TOKEN", "")

        if not auth_header.startswith("Bearer ") or \
           auth_header[7:] != expected_token:
            return JSONResponse(
                {"error": "Unauthorized"},
                status_code=401,
            )

        return await call_next(request)

# ASGI 통합 시 미들웨어 적용 (Ch13 참고)
# app = Starlette(middleware=[Middleware(AuthMiddleware)])
Deep Dive: CORS 설정

웹 브라우저에서 MCP 서버에 직접 접근하는 경우(예: 웹 기반 MCP 클라이언트), CORS 설정이 필요하다. MCP Python SDK의 Streamable HTTP 서버는 Starlette/ASGI 기반이므로, CORSMiddleware를 추가하면 된다. 

from starlette.middleware.cors import CORSMiddleware

# ASGI 앱에 CORS 미들웨어 추가
app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://your-frontend.com"],
    allow_methods=["GET", "POST", "DELETE"],
    allow_headers=["Content-Type", "Mcp-Session-Id", "Authorization"],
    expose_headers=["Mcp-Session-Id"],
)

expose_headers에 "Mcp-Session-Id"를 반드시 포함해야 브라우저가 이 헤더를 읽을 수 있다.

7.8 다음 단계

이 장에서는 Transport 계층의 내부 구조와 JSON-RPC 메시지 흐름, 그리고 프로덕션 배포 전략을 학습했다. 핵심 정리는 다음과 같다.

  • JSON-RPC 2.0 — Request(id 포함), Response(result/error), Notification(id 없음)의 3가지 메시지 유형
  • stdio — 로컬 프로세스 간 통신, stdout 오염 금지, Claude Desktop/IDE에 최적
  • Streamable HTTP — HTTP POST + SSE, 세션 관리(Mcp-Session-Id), 원격 배포 가능
  • Stateless 모드 — 세션 없이 동작, 수평 확장과 서버리스 환경에 적합
  • 프로덕션 배포 — Docker + Nginx 리버스 프록시, TLS 종단, 인증 미들웨어

다음 장에서는 Python으로 MCP 클라이언트를 직접 구현한다. 서버에 연결하여 도구를 호출하고, LLM과 결합하여 에이전트 루프를 구축하며, Sampling 기능으로 서버가 LLM을 역으로 호출하는 패턴까지 다룬다.
