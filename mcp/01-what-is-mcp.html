MCP란 무엇인가

Model Context Protocol의 개념, 아키텍처, 그리고 프로토콜 내부 구조를 깊이 이해한다

MCP란?

핵심 개념: MCP (Model Context Protocol)

MCP는 LLM 애플리케이션과 외부 데이터 소스/도구 사이의 통신을 표준화하는 개방형 프로토콜이다. JSON-RPC 2.0을 기반으로 하며, 클라이언트-서버 아키텍처를 통해 LLM이 파일 시스템, 데이터베이스, API 등 외부 세계에 구조적으로 접근할 수 있게 한다.

Anthropic은 2024년 11월 MCP를 오픈 소스로 공개했다. 현재 프로토콜 최신 버전은 2025-03-26이며, 이 튜토리얼은 Python SDK v1.12.x를 기준으로 한다.

"AI를 위한 USB-C" — 비유를 넘어서

MCP를 흔히 "AI를 위한 USB-C"라고 비유한다. USB-C가 하나의 커넥터로 충전, 데이터 전송, 영상 출력을 모두 처리하듯, MCP는 하나의 프로토콜로 LLM이 다양한 외부 서비스와 소통하게 한다. 이 비유는 직관적이지만, MCP의 본질을 완전히 전달하지는 못한다.

USB-C는 물리적 커넥터에 불과하다. MCP는 그보다 훨씬 높은 수준에서 동작한다. 프로토콜 협상(capability negotiation), 구조화된 메시지 교환, 양방향 통신, 보안 제어까지 포함하는 완전한 애플리케이션 레벨 프로토콜이다. 굳이 비유하자면 USB-C보다는 HTTP에 가깝다. HTTP가 웹 브라우저와 서버 사이의 통신을 표준화한 것처럼, MCP는 LLM 애플리케이션과 외부 도구 사이의 통신을 표준화한다.

MCP가 해결하는 근본적 문제

LLM은 본질적으로 텍스트 입력을 받아 텍스트 출력을 생성하는 모델이다. 훈련 데이터에 포함된 지식은 가지고 있지만, 실시간 정보에 접근하거나 외부 시스템에 작업을 수행할 수 있는 내재적 메커니즘이 없다. 파일을 읽을 수도, 데이터베이스를 쿼리할 수도, API를 호출할 수도 없다.

이 문제를 해결하기 위해 각 LLM 제공자가 독자적인 도구 호출(function calling) 방식을 만들었지만, 이는 제공자마다 API가 다르고, 매번 새로운 연동 코드를 작성해야 하는 파편화 문제를 낳았다. MCP는 바로 이 파편화를 해결하는 표준이다.

INFO

MCP 사양(specification)은 spec.modelcontextprotocol.io에서 확인할 수 있다. TypeScript SDK, Python SDK, Java SDK, Kotlin SDK, C# SDK가 공식으로 제공되며, 이 튜토리얼에서는 Python SDK를 사용한다.

왜 MCP가 필요한가?

N×M 문제 → N+M 해결

N개의 LLM 애플리케이션과 M개의 외부 도구가 있다고 하자. 표준 프로토콜이 없으면 각 앱이 각 도구에 대한 연동 코드를 따로 작성해야 한다. 이는 N × M개의 통합 코드를 의미한다. 앱이 4개이고 도구가 4개라면 16개의 연동 코드가 필요하다.

MCP를 도입하면 각 도구는 MCP 서버를 하나만 구현하면 되고, 각 LLM 앱은 MCP 클라이언트를 하나만 구현하면 된다. 결과적으로 N + M개의 통합만 필요하다. 위 예시에서 16개가 8개로 줄어든다. 규모가 커질수록 이 효과는 기하급수적으로 커진다.

기존 접근법과의 비교

MCP 이전에도 LLM에 외부 도구를 연결하려는 시도는 여러 가지 있었다. 각각의 한계를 살펴보자.

접근법 방식 한계
Function Calling LLM 제공자(OpenAI, Anthropic 등)가 자체 API에 도구 호출 기능을 내장 제공자마다 API가 다르다. 벤더 종속(vendor lock-in). 서버 측 표준이 없다.
Custom API 래퍼 각 외부 서비스에 대해 직접 REST/GraphQL 래퍼를 작성 매번 새로 작성해야 한다. 표준이 없어 재사용이 불가능하다.
LangChain Tools LangChain 프레임워크의 Tool 추상화를 사용 LangChain 프레임워크에 종속된다. 다른 프레임워크에서 재사용할 수 없다.
Function Calling (기존)
  • LLM 제공자마다 다른 API 스펙
  • 서버 측 표준 없음 — 각 도구를 직접 연동
  • 단방향: LLM → 도구 호출만 가능
  • 벤더 종속: OpenAI에서 Anthropic으로 전환 시 전면 재작성
  • 도구 발견(discovery) 메커니즘 없음
MCP (표준 프로토콜)
  • 벤더 독립적 오픈 프로토콜
  • 서버가 도구를 표준 인터페이스로 노출
  • 양방향 통신: 서버도 LLM에 요청 가능 (Sampling)
  • LLM 제공자 변경 시 MCP 서버 코드 변경 불필요
  • 프로토콜 레벨의 도구 발견: tools/list
TIP

MCP와 Function Calling은 경쟁 관계가 아니라 보완 관계이다. 실제로 MCP 클라이언트 내부에서는 LLM의 Function Calling을 활용하여 어떤 MCP 도구를 호출할지 결정한다. MCP는 Function Calling 위에 올라가는 표준화 계층이다.

MCP 아키텍처

MCP 아키텍처는 Host, Client, Server의 3계층(3-tier)으로 구성된다. 이 구조를 정확히 이해하는 것이 MCP 개발의 출발점이다. 

graph TD
    subgraph Host ["Host (LLM 애플리케이션)"]
        LLM["LLM Engine"]
        ClientA["MCP Client A"]
        ClientB["MCP Client B"]
        ClientC["MCP Client C"]
        LLM --- ClientA
        LLM --- ClientB
        LLM --- ClientC
    end
    ClientA <-->|"stdio"| ServerA["MCP Server A\n(파일 시스템)"]
    ClientB <-->|"stdio"| ServerB["MCP Server B\n(데이터베이스)"]
    ClientC <-->|"HTTP"| ServerC["MCP Server C\n(원격 API)"]

Host (호스트)

사용자가 직접 상호작용하는 LLM 애플리케이션이다. Host는 LLM 엔진을 내장하고 있으며, 하나 이상의 MCP Client를 내부에서 관리한다. 사용자의 자연어 요청을 LLM에 전달하고, LLM의 도구 호출 의사결정을 MCP Client를 통해 실제 서버 호출로 변환하는 역할을 한다.

대표적인 Host 애플리케이션:

  • Claude Desktop — Anthropic의 공식 데스크톱 앱
  • Cursor — AI 코딩 IDE
  • VS Code + GitHub Copilot — Copilot의 MCP 확장
  • Windsurf — Codeium의 AI IDE
  • Zed — 고성능 코드 에디터
  • 직접 구현한 앱 — Python SDK의 ClientSession을 사용하여 직접 만든 애플리케이션

Client (클라이언트)

Host 내부에서 동작하는 프로토콜 커넥터이다. 하나의 Client는 하나의 Server와 1:1 연결을 유지한다. 주요 책임은 다음과 같다:

  • 프로토콜 협상: 연결 시 initialize 요청을 통해 서버와 지원 기능을 교환한다
  • 메시지 라우팅: LLM의 도구 호출 요청을 JSON-RPC 메시지로 변환하여 서버에 전달한다
  • Capability 관리: 서버가 지원하는 기능(Tools, Resources, Prompts 등)을 추적한다
  • 전송 계층 관리: stdio 또는 HTTP 연결을 생성하고 유지한다

Server (서버)

외부 도구나 데이터를 MCP 프로토콜로 노출하는 경량 프로세스이다. 파일 시스템, 데이터베이스, 웹 API 등 다양한 리소스를 LLM이 사용할 수 있도록 구조화하여 제공한다.

TIP

이 튜토리얼에서 우리가 주로 만들게 되는 것이 바로 MCP Server이다. Python SDK의 FastMCP 클래스를 사용하면 데코레이터 몇 줄로 서버를 구축할 수 있다. Client 구현은 Chapter 8에서 다룬다.

배포 토폴로지: 로컬 vs 원격

MCP 서버는 두 가지 방식으로 배포할 수 있다. 전송 계층(transport)에 따라 구분된다.

로컬 서버 (stdio)
  • Host가 서버 프로세스를 직접 실행 (subprocess)
  • 표준 입출력(stdin/stdout)으로 통신
  • 설치가 간단하고 보안 걱정이 적다
  • 사용자 컴퓨터의 로컬 리소스 접근에 적합
  • 예: 파일 시스템 서버, 로컬 DB 서버
원격 서버 (Streamable HTTP)
  • 독립적인 웹 서버로 배포
  • HTTP + SSE(Server-Sent Events)로 통신
  • 여러 클라이언트가 동시 접속 가능
  • 인증/인가 체계 필요 (OAuth 2.1)
  • 예: SaaS API 서버, 공유 도구 서버

MCP의 6가지 핵심 기능

MCP는 6가지 핵심 기능(primitives + capabilities)을 정의한다. 초기에는 Tools, Resources, Prompts 3가지 프리미티브만 알려졌지만, 현재 사양에서는 서버와 클라이언트 양방향으로 6가지 기능이 정의되어 있다.

기능 방향 제공 주체 설명
Tools Server → Client Server LLM이 호출하는 함수. 부수 효과(side effect)를 가질 수 있다.
Resources Server → Client Server 읽기 전용 데이터를 URI로 노출한다. 파일, DB 레코드, 설정 값 등.
Prompts Server → Client Server 재사용 가능한 프롬프트 템플릿. 사용자가 선택하여 적용한다.
Sampling Client → Server Client 서버가 클라이언트의 LLM에게 텍스트 생성(completion)을 요청한다.
Roots Client → Server Client 클라이언트가 서버에게 파일 시스템 루트 경로를 알려준다.
Elicitation Client → Server Client 서버가 사용자에게 추가 정보를 요청한다 (대화형 폼).

Server가 제공하는 기능 (Server → Client)

Tools — LLM이 호출하는 함수

Tools는 MCP의 가장 핵심적인 기능이다. LLM이 사용자의 요청을 처리하기 위해 능동적으로 호출하는 함수이다. 파일 쓰기, API 호출, 계산 수행 등 부수 효과를 가질 수 있으므로 사용자 확인(human-in-the-loop)이 권장된다. 상세 내용은 Chapter 4: Tools 심화에서 다룬다.

Resources — 읽기 전용 데이터

Resources는 서버가 URI 형태로 노출하는 데이터이다. file:///path/to/doc.md, postgres://db/users 같은 URI를 통해 접근하며, 부수 효과 없이 데이터를 읽기만 한다. LLM이 아닌 애플리케이션(Host)이 접근 시점을 제어한다. 상세 내용은 Chapter 5: Resources에서 다룬다.

Prompts — 재사용 프롬프트 템플릿

서버가 미리 정의한 프롬프트 템플릿이다. 코드 리뷰, 번역, 요약 등 반복적인 작업을 위한 최적화된 프롬프트를 서버가 제공하고, 사용자가 명시적으로 선택하여 사용한다. 상세 내용은 Chapter 6: Prompts에서 다룬다.

Client가 제공하는 기능 (Client → Server)

Sampling — 서버가 LLM에게 요청

Sampling은 MCP의 독특한 기능이다. 일반적으로 LLM이 도구를 호출하지만, Sampling을 사용하면 서버가 역으로 클라이언트의 LLM에게 텍스트 생성을 요청할 수 있다. 이를 통해 "에이전트 루프"를 구현할 수 있다 — 예를 들어 서버가 코드를 분석한 뒤 LLM에게 개선 사항을 요청하고, 그 결과를 다시 처리하는 식이다. Human-in-the-loop이 적용되어 사용자 승인 없이는 실행되지 않는다.

Roots — 파일 시스템 루트

클라이언트가 서버에게 작업 대상 파일 시스템 경로를 알려주는 기능이다. 예를 들어 IDE가 현재 열린 프로젝트의 루트 디렉토리를 서버에 전달하면, 서버는 해당 경로를 기준으로 파일 작업을 수행한다. Roots는 보안 경계를 설정하는 데에도 사용된다 — 서버는 제공된 루트 외부에 접근하지 않아야 한다.

Elicitation — 사용자에게 정보 요청

서버가 작업 중 추가 정보가 필요할 때 사용자에게 직접 질문하는 기능이다. 예를 들어 데이터베이스 마이그레이션 서버가 "정말 이 테이블을 삭제할까요?"라고 사용자에게 확인을 요청할 수 있다. Elicitation은 2025-03-26 사양에서 추가된 비교적 새로운 기능이다.

WARNING

Tools는 외부 시스템에 변경을 가할 수 있으므로 (예: 파일 삭제, 이메일 전송), 반드시 사용자 확인 과정을 거치도록 설계해야 한다. Resources는 읽기 전용이므로 비교적 안전하다. Sampling과 Elicitation은 프로토콜 차원에서 human-in-the-loop이 강제된다.

MCP 프로토콜 내부 — JSON-RPC 2.0

Deep Dive: JSON-RPC 2.0 기반 프로토콜

MCP의 모든 통신은 JSON-RPC 2.0 규격을 따른다. JSON-RPC는 JSON을 사용하는 경량 원격 프로시저 호출(RPC) 프로토콜로, 전송 계층에 독립적이다. MCP는 이 위에 자체적인 메서드명과 파라미터 구조를 정의하여 LLM-도구 통신에 특화된 프로토콜을 구축했다.

JSON-RPC 2.0에는 3가지 메시지 유형이 있다:

  • Request: id를 포함하며 응답을 기대하는 메시지
  • Response: Request에 대한 응답. 같은 id를 포함한다
  • Notification: id가 없으며 응답을 기대하지 않는 일방 메시지

실제 프로토콜 메시지 예제

MCP 통신에서 오가는 실제 JSON-RPC 메시지를 살펴보자. 이 메시지들은 전송 계층(stdio 또는 HTTP) 위에서 교환된다.

1) initialize — 연결 초기화 및 Capability 협상

클라이언트가 서버에 연결할 때 가장 먼저 보내는 메시지이다. 양측이 지원하는 기능(capabilities)과 프로토콜 버전을 교환한다.

→ REQUEST: initialize
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-03-26",
    "capabilities": {
      "sampling": {},
      "roots": {
        "listChanged": true
      }
    },
    "clientInfo": {
      "name": "my-ai-app",
      "version": "1.0.0"
    }
  }
}
← RESPONSE
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-03-26",
    "capabilities": {
      "tools": {
        "listChanged": true
      },
      "resources": {
        "subscribe": true,
        "listChanged": true
      },
      "prompts": {
        "listChanged": true
      },
      "logging": {}
    },
    "serverInfo": {
      "name": "weather-server",
      "version": "0.1.0"
    }
  }
}

2) notifications/initialized — 초기화 완료 알림

클라이언트가 initialize 응답을 받은 후, 서버에게 "초기화가 완료되었다"는 Notification을 보낸다. 이 메시지에는 id가 없다 — 응답을 기대하지 않기 때문이다.

→ NOTIFICATION: notifications/initialized
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}

3) tools/list — 사용 가능한 도구 목록 조회

클라이언트가 서버에 등록된 모든 도구의 목록을 요청한다. LLM은 이 목록을 보고 어떤 도구를 호출할지 결정한다.

→ REQUEST: tools/list
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/list"
}
← RESPONSE
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "tools": [
      {
        "name": "get_weather",
        "description": "주어진 도시의 현재 날씨를 조회한다",
        "inputSchema": {
          "type": "object",
          "properties": {
            "city": {
              "type": "string",
              "description": "도시 이름"
            }
          },
          "required": ["city"]
        }
      }
    ]
  }
}

4) tools/call — 도구 실행

LLM이 특정 도구를 호출하기로 결정하면, 클라이언트가 서버에 tools/call 요청을 보낸다.

→ REQUEST: tools/call
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": {
      "city": "Seoul"
    }
  }
}
← RESPONSE
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "서울의 현재 기온은 24°C이며, 맑은 날씨이다."
      }
    ]
  }
}

Capability Negotiation 상세

initialize 단계에서 클라이언트와 서버는 각자 지원하는 기능을 선언한다. 이를 Capability Negotiation이라 한다. 양측은 상대방이 선언한 기능만 사용할 수 있다.

Client Capabilities
  • sampling: 서버의 LLM 완성 요청을 수락할 수 있다
  • roots: 파일 시스템 루트를 제공할 수 있다
  • elicitation: 서버의 사용자 질문 요청을 처리할 수 있다

서버는 클라이언트가 해당 capability를 선언한 경우에만 sampling/createMessage, roots/list, elicitation/create 요청을 보낼 수 있다.

Server Capabilities
  • tools: 호출 가능한 도구를 제공한다
  • resources: 읽기 전용 데이터를 제공한다
  • prompts: 프롬프트 템플릿을 제공한다
  • logging: 구조화된 로그를 전송할 수 있다

클라이언트는 서버가 해당 capability를 선언한 경우에만 tools/list, resources/read, prompts/get 등의 요청을 보낼 수 있다.

Deep Dive: 프로토콜 버전 호환성

initialize 요청에서 클라이언트는 자신이 지원하는 protocolVersion을 보낸다. 서버는 해당 버전을 지원하면 같은 버전으로 응답하고, 지원하지 않으면 자신이 지원하는 가장 가까운 버전으로 응답한다. 클라이언트가 서버의 응답 버전을 수용할 수 없으면 연결을 종료한다.

현재까지 공개된 프로토콜 버전은 다음과 같다:

  • 2024-11-05 — 최초 공개 버전
  • 2025-03-26 — 현재 최신 버전. Elicitation, Streamable HTTP 등 추가

MCP 통신 흐름 — 전체 라이프사이클

MCP 연결의 전체 생명 주기를 단계별로 살펴보자. 연결 수립부터 종료까지 5단계로 구성된다.

  1. 1
    연결 수립 (Transport Layer)

    클라이언트가 서버와 통신 채널을 연다. stdio 방식이라면 서버 프로세스를 실행하여 stdin/stdout을 연결하고, Streamable HTTP라면 서버의 MCP 엔드포인트에 HTTP 연결을 맺는다. 아직 프로토콜 메시지는 교환되지 않은 상태이다.

  2. 2
    Initialize (Capability Negotiation)

    클라이언트가 initialize Request를 보낸다. 프로토콜 버전, 클라이언트 정보, 지원하는 기능(capabilities)을 선언한다. 서버는 자신의 정보와 기능을 응답한다. 이 교환을 통해 양측이 "무엇을 할 수 있는지"를 합의한다.

  3. 3
    Initialized Notification

    클라이언트가 notifications/initialized Notification을 보내 초기화가 완료되었음을 알린다. 이 시점부터 양측이 정상적인 프로토콜 메시지를 주고받을 수 있다.

  4. 4
    기능 사용 (Operation Phase)

    실제 작업이 수행되는 단계이다. 클라이언트는 tools/list, tools/call, resources/read, prompts/get 등의 요청을 보낸다. 서버도 (해당 capability가 협상된 경우) 클라이언트에 sampling/createMessage, elicitation/create 등의 요청을 보낼 수 있다. 이 단계는 연결이 끊어질 때까지 계속된다.

  5. 5
    연결 종료 (Shutdown)

    클라이언트가 연결을 닫는다. stdio 방식이라면 서버 프로세스의 stdin을 닫거나 프로세스를 종료하고, HTTP 방식이라면 연결을 끊는다. 서버는 자원을 정리(cleanup)하고 종료한다.

핵심을 정리하면, MCP는 위 파이프라인의 가운데 단계 — LLM과 외부 도구 사이의 통신 — 를 표준화하는 프로토콜이다. LLM이 어떤 도구를 호출할지 결정하는 것은 LLM 자체의 역할이고, MCP는 그 결정을 실제 도구 호출로 변환하는 통신 규격을 제공한다.

다음 단계

이 챕터에서 MCP의 핵심 개념을 깊이 살펴보았다. 정리하면:

  • MCP는 LLM과 외부 도구 사이의 통신을 표준화하는 개방형 프로토콜이다
  • N×M 문제를 N+M으로 해결하여 통합 비용을 획기적으로 줄인다
  • Host → Client → Server 3계층 아키텍처로 동작한다
  • Tools, Resources, Prompts, Sampling, Roots, Elicitation의 6가지 기능을 제공한다
  • JSON-RPC 2.0 기반으로 구조화된 메시지를 교환한다
  • Capability Negotiation을 통해 양측의 기능을 안전하게 합의한다

다음 챕터에서는 실제 개발을 위한 환경을 설정한다. Python과 uv를 설치하고 MCP Python SDK를 사용할 프로젝트를 구성할 것이다.
