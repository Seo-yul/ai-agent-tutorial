Context 및 Logging

ServerRequestContext의 모든 기능, 구조화된 로깅, 진행률 보고, 그리고 외부 모니터링 연동

12.1 ServerRequestContext란?

핵심 개념: Context와 ServerRequestContext

MCP 서버에서 도구, 리소스, 프롬프트 핸들러가 호출될 때 현재 요청에 대한 모든 맥락 정보를 담고 있는 객체가 주입된다. FastMCP를 사용할 때는 Context 클래스가, 저수준 Server API를 사용할 때는 ServerRequestContext가 이 역할을 한다. Context는 ServerRequestContext를 내부에 래핑한 고수준 파사드(Facade)로, 로깅, 진행률 보고, 세션 접근, HTTP 요청 정보 등 핸들러가 필요한 거의 모든 기능을 노출한다.

Context가 제공하는 기능을 다이어그램으로 정리하면 다음과 같다. 

graph TD
    CTX["Context (FastMCP 고수준 래퍼)"]

    CTX --> LOG["로깅
debug / info / warning / error"]
    CTX --> PROG["진행률 보고
report_progress(current, total)"]
    CTX --> SESS["세션 접근
ctx.session → ServerSession"]
    CTX --> REQ["HTTP 요청
ctx.request → Starlette Request"]
    CTX --> META["요청 메타
request_id, client_id"]
    CTX --> RES["리소스 알림
send_resource_updated /
send_resource_list_changed"]

    SESS --> LOW["저수준 API
send_log_message /
send_progress_notification"]

    subgraph Internal ["내부 구조"]
        SRC["ServerRequestContext
(ContextVar로 관리)"]
        CTX -.->|래핑| SRC
    end
Deep Dive: ServerRequestContext의 내부 구조

ServerRequestContext는 Python의 contextvars.ContextVar를 통해 현재 비동기 태스크에 바인딩된다. SDK 내부에서 각 요청을 처리할 때 request_ctx.set(ctx)로 컨텍스트를 설정하고, 핸들러 실행이 끝나면 자동으로 정리된다. 이 덕분에 동시에 여러 요청이 처리되더라도 각 핸들러는 자신만의 격리된 컨텍스트에 접근할 수 있다. ServerRequestContext의 주요 속성은 session(현재 클라이언트 세션), request_id(JSON-RPC 요청 ID), meta(progressToken 등 메타 정보)이다.

12.2 Context 자동 주입

FastMCP는 핸들러 함수의 타입 힌트를 검사하여 Context 타입의 매개변수를 자동으로 주입한다. 이 매개변수는 LLM에 노출되는 도구 스키마에 포함되지 않으므로, LLM이 이 인자를 채워 넣을 필요가 전혀 없다. 

from mcp.server.fastmcp import FastMCP, Context

mcp = FastMCP("Injection Demo")

@mcp.tool()
async def greet(name: str, ctx: Context) -> str:
    """사용자에게 인사한다"""
    # name → LLM이 제공하는 인자 (스키마에 포함)
    # ctx  → FastMCP가 자동 주입 (스키마에서 제외)
    await ctx.info(f"{name}에게 인사 중")
    return f"안녕, {name}!"

위 도구가 LLM에 노출되는 JSON Schema는 다음과 같다. ctx는 보이지 않는다.

← tools/list 응답 (발췌)
{
  "name": "greet",
  "description": "사용자에게 인사한다",
  "inputSchema": {
    "type": "object",
    "properties": {
      "name": { "type": "string" }
    },
    "required": ["name"]
  }
}
Deep Dive: 자동 주입이 동작하는 원리

FastMCP는 inspect.signature()를 사용해 핸들러의 모든 매개변수를 순회한다. 각 매개변수의 어노테이션이 Context 클래스이거나 그 서브클래스이면, 해당 매개변수를 도구 입력 스키마 생성에서 제외하고 호출 시점에 현재 요청의 Context 인스턴스를 주입한다. 매개변수 이름은 ctx일 필요가 없다 — context, c 등 어떤 이름이든 타입 힌트만 Context이면 동작한다.

WARNING

타입 힌트 없이 ctx라는 이름만 사용하면 자동 주입이 되지 않는다. 반드시 ctx: Context처럼 타입 어노테이션을 명시해야 한다.

12.3 로깅 심화

4단계 편의 메서드

FastMCP의 Context는 가장 자주 사용하는 4개 로그 레벨에 대해 편의 메서드를 제공한다. 각 메서드는 내부적으로 MCP 프로토콜의 notifications/message를 클라이언트에 전송한다. 

from mcp.server.fastmcp import FastMCP, Context

mcp = FastMCP("Logging Demo")

@mcp.tool()
async def process_data(ctx: Context, data: str) -> str:
    """데이터를 처리한다"""
    await ctx.debug("변수 상태 점검 — 개발 환경에서만 유용")
    await ctx.info("처리 시작 — 일반적인 진행 상황")
    await ctx.warning("메모리 사용량 80% 초과 — 주의 필요")
    await ctx.error("외부 API 호출 실패 — 즉시 확인 필요")
    return f"처리 완료: {data}"

구조화된 데이터 로깅

문자열 대신 딕셔너리를 전달하면 JSON 형태로 직렬화되어 클라이언트에 전송된다. 이를 통해 클라이언트 측에서 로그를 파싱하여 대시보드에 표시하거나 특정 필드로 필터링하는 등 고급 처리가 가능하다. 

@mcp.tool()
async def analyze(ctx: Context, query: str) -> str:
    """쿼리를 분석한다"""
    # 구조화된 로그: 딕셔너리 전달
    await ctx.info({
        "action": "query_analysis",
        "query_length": len(query),
        "timestamp": "2025-06-25T10:30:00Z",
        "metadata": {
            "source": "user_input",
            "language": "ko"
        }
    })
    return "분석 완료"

RFC 5424 — 8단계 로그 레벨

MCP 프로토콜은 RFC 5424 (Syslog)에 기반한 8가지 로그 레벨을 정의한다. FastMCP의 4가지 편의 메서드는 이 중 일부만 커버하며, 나머지 레벨은 저수준 API를 통해 사용할 수 있다.

레벨 심각도 설명 Context 메서드 사용 예시
emergency 0 (최고) 시스템 사용 불가 저수준 API 서버 완전 중단, 데이터 손실
alert 1 즉각 조치 필요 저수준 API DB 연결 끊김, 인증 시스템 장애
critical 2 심각한 상태 저수준 API 디스크 공간 부족, OOM 임박
error 3 에러 발생 ctx.error() 파일 읽기 실패, API 호출 실패
warning 4 경고 상황 ctx.warning() deprecated API 사용, 임계치 초과
notice 5 정상이지만 중요한 이벤트 저수준 API 설정 변경, 서버 재시작
info 6 정보성 메시지 ctx.info() 작업 시작/완료, 처리 현황
debug 7 (최저) 디버깅용 상세 정보 ctx.debug() 변수 값, 내부 상태 추적

emergency, alert, critical, notice 레벨은 저수준 API의 send_log_message를 통해 사용한다. 

@mcp.tool()
async def critical_operation(ctx: Context) -> str:
    """모든 로그 레벨을 시연한다"""
    # FastMCP 편의 메서드 (4가지)
    await ctx.debug("디버그: 내부 변수 값 확인")
    await ctx.info("정보: 작업 정상 시작")
    await ctx.warning("경고: 메모리 사용량 80% 초과")
    await ctx.error("에러: 외부 API 호출 실패")

    # 저수준 API로 나머지 레벨 사용
    session = ctx.session
    if session:
        await session.send_log_message(
            level="notice",
            data="알림: 설정 파일 변경 감지",
        )
        await session.send_log_message(
            level="critical",
            data="심각: 핵심 서비스 응답 없음",
        )
        await session.send_log_message(
            level="alert",
            data="긴급: 데이터베이스 연결 끊김",
        )
        await session.send_log_message(
            level="emergency",
            data="비상: 시스템 사용 불가",
        )
    return "모든 레벨 전송 완료"

notifications/message 프로토콜 메시지

로그 메서드를 호출하면 내부적으로 다음과 같은 JSON-RPC notification이 클라이언트에 전송된다.

→ Notification: notifications/message
{
  "jsonrpc": "2.0",
  "method": "notifications/message",
  "params": {
    "level": "info",
    "logger": "mcp_server",
    "data": {
      "action": "query_analysis",
      "query_length": 42
    }
  }
}
INFO

data 필드는 문자열 또는 JSON 객체 모두 가능하다. logger 필드는 로그를 보낸 서버를 식별하는 선택적 문자열로, 저수준 API의 send_log_message(logger="my_module")에서 지정할 수 있다.

클라이언트 측 로그 수신

서버에서 보낸 로그를 클라이언트에서 수신하려면 ClientSession에 notification handler를 등록해야 한다. 다음은 로그 메시지를 수신하여 출력하는 클라이언트 예제다. 

from mcp import ClientSession
from mcp.client.streamable_http import streamablehttp_client
from mcp.types import LoggingMessageNotification

async def main():
    async with streamablehttp_client("http://localhost:8000/mcp") as (
        read_stream, write_stream, _
    ):
        async with ClientSession(read_stream, write_stream) as session:
            await session.initialize()

            # 로그 수신 핸들러 등록
            async def on_log(notification: LoggingMessageNotification):
                params = notification.params
                print(f"[{params.level}] {params.data}")

            # notification_handler를 통해 로그 수신
            session.logging_notification_handler = on_log

            # 도구를 호출하면 서버가 보내는 로그가 핸들러로 전달된다
            result = await session.call_tool("process_data", {"data": "테스트"})
            print(f"결과: {result}")

logging/setLevel 구현

클라이언트는 logging/setLevel 요청을 보내 서버의 로그 전송 레벨을 동적으로 변경할 수 있다. 예를 들어, 클라이언트가 warning으로 설정하면 서버는 warning 이상 (warning, error, critical, alert, emergency)의 로그만 전송한다.

→ Request: logging/setLevel
{
  "jsonrpc": "2.0",
  "id": 5,
  "method": "logging/setLevel",
  "params": {
    "level": "warning"
  }
}
← Response
{
  "jsonrpc": "2.0",
  "id": 5,
  "result": {}
}

클라이언트 코드에서 로그 레벨을 변경하는 방법은 다음과 같다.

# 클라이언트 측: 로그 레벨 변경
await session.set_logging_level("warning")

# 이후 서버는 warning 이상의 로그만 전송한다
# debug, info 로그는 필터링되어 전송되지 않는다
Deep Dive: 로그 레벨 필터링의 동작 원리

logging/setLevel이 호출되면 서버 내부의 ServerSession이 최소 로그 레벨을 기록해 둔다. 이후 send_log_message()가 호출될 때마다 해당 레벨이 최소 레벨 이상인지 확인하고, 미달이면 실제 notification을 전송하지 않는다. 이는 서버 측에서 필터링이 이루어지므로 네트워크 트래픽도 절약된다. FastMCP의 ctx.debug() 등 편의 메서드는 내부적으로 send_log_message()를 호출하므로 이 필터링의 영향을 동일하게 받는다.

12.4 진행률 보고

오래 걸리는 작업의 진행 상황을 실시간으로 클라이언트에 알리려면 ctx.report_progress(current, total)를 사용한다. 클라이언트는 이 정보를 프로그레스 바 등으로 시각화할 수 있다. 

import asyncio
from mcp.server.fastmcp import FastMCP, Context

mcp = FastMCP("Progress Demo")

@mcp.tool()
async def batch_process(ctx: Context, items: list[str]) -> str:
    """배치 처리한다"""
    results = []
    total = len(items)

    for i, item in enumerate(items):
        await ctx.report_progress(i, total)
        await ctx.info(f"처리 중: {item}")
        await asyncio.sleep(0.1)  # 실제 처리 시뮬레이션
        results.append(item.upper())

    # 최종 완료 보고
    await ctx.report_progress(total, total)
    return ", ".join(results)

progressToken 조건

진행률 보고는 클라이언트가 tools/call 요청의 _meta.progressToken에 값을 포함했을 때만 실제로 전송된다. 토큰이 없으면 report_progress는 아무 동작도 하지 않고 조용히 무시된다.

→ Request: tools/call (progressToken 포함)
{
  "jsonrpc": "2.0",
  "id": 7,
  "method": "tools/call",
  "params": {
    "name": "batch_process",
    "arguments": { "items": ["a", "b", "c"] },
    "_meta": {
      "progressToken": "progress-token-001"
    }
  }
}
→ Notification: notifications/progress
{
  "jsonrpc": "2.0",
  "method": "notifications/progress",
  "params": {
    "progressToken": "progress-token-001",
    "progress": 2,
    "total": 3
  }
}
WARNING

total은 선택적(optional)이다. 전체 작업량을 미리 알 수 없는 경우 report_progress(current)처럼 total 없이 호출할 수 있다. 이 경우 클라이언트는 스피너(spinner) 형태로 표시하거나, 퍼센티지 없이 진행 수만 표시한다.

12.5 Low-Level API

FastMCP를 사용하지 않는 저수준 서버에서는 request_ctx.get()으로 현재 요청의 ServerRequestContext에 직접 접근한다. 

from mcp.server import Server
from mcp.server.lowlevel.server import request_ctx
from mcp.types import (
    CallToolResult,
    TextContent,
    Tool,
)

server = Server("low-level-example")

@server.call_tool()
async def handle_tool(name: str, arguments: dict) -> list[TextContent]:
    """저수준 도구 핸들러"""
    # 현재 요청 컨텍스트 획득
    ctx = request_ctx.get()

    # 세션을 통해 로그 전송
    await ctx.session.send_log_message(
        level="info",
        data=f"도구 '{name}' 처리 시작",
        logger="tool_handler",
    )

    # 진행률 알림 전송 (progressToken이 있을 때만)
    if ctx.meta and ctx.meta.progressToken:
        await ctx.session.send_progress_notification(
            progress_token=ctx.meta.progressToken,
            progress=0.5,
            total=1.0,
        )

    if name == "greet":
        user = arguments.get("name", "세계")
        await ctx.session.send_log_message(
            level="debug",
            data=f"인사 대상: {user}",
            logger="tool_handler",
        )
        return [TextContent(type="text", text=f"안녕, {user}!")]

    await ctx.session.send_log_message(
        level="error",
        data=f"알 수 없는 도구: {name}",
        logger="tool_handler",
    )
    return [TextContent(type="text", text=f"알 수 없는 도구: {name}")]

@server.list_tools()
async def list_tools() -> list[Tool]:
    return [
        Tool(
            name="greet",
            description="인사한다",
            inputSchema={
                "type": "object",
                "properties": {
                    "name": {"type": "string", "description": "이름"},
                },
                "required": ["name"],
            },
        )
    ]
저수준 API (Server)
  • request_ctx.get()으로 컨텍스트 획득
  • ctx.session.send_log_message()로 로그 전송
  • ctx.session.send_progress_notification()으로 진행률
  • 로그 레벨 8단계 모두 직접 사용 가능
  • logger 필드 직접 지정 가능
고수준 API (FastMCP Context)
  • 타입 힌트로 자동 주입
  • ctx.debug/info/warning/error() 편의 메서드
  • ctx.report_progress()로 진행률
  • 4개 레벨 편의 메서드 + 저수준 접근 가능
  • ctx.session으로 저수준 API 접근

12.6 HTTP 헤더 접근

Streamable HTTP 전송을 사용할 때, ctx.request를 통해 원본 HTTP 요청의 Starlette Request 객체에 접근할 수 있다. 이를 활용하면 인증 토큰 추출, 클라이언트 IP 확인, 커스텀 헤더 읽기 등이 가능하다. 

import json
from mcp.server.fastmcp import FastMCP, Context

mcp = FastMCP("Header Example")

@mcp.tool()
async def whoami(ctx: Context) -> str:
    """현재 요청의 인증 정보를 확인한다"""
    if not ctx.request:
        return "HTTP 요청이 아님 (stdio 전송)"

    # Bearer 토큰 추출
    auth_header = ctx.request.headers.get("authorization", "")
    if not auth_header.startswith("Bearer "):
        await ctx.warning("인증 토큰 없음")
        return "인증되지 않은 요청"

    token = auth_header[7:]
    await ctx.info(f"토큰 확인: {token[:8]}...")

    # 클라이언트 IP
    client_ip = ctx.request.client.host if ctx.request.client else "unknown"
    await ctx.debug(f"클라이언트 IP: {client_ip}")

    # 커스텀 헤더
    tenant_id = ctx.request.headers.get("x-tenant-id", "default")

    return json.dumps({
        "token_prefix": token[:8],
        "client_ip": client_ip,
        "tenant_id": tenant_id,
    }, indent=2)


@mcp.tool()
async def echo_headers(ctx: Context) -> str:
    """모든 요청 헤더를 반환한다"""
    if not ctx.request:
        return "헤더 정보 없음 (stdio 전송)"

    headers = dict(ctx.request.headers)
    # 민감한 헤더 마스킹
    if "authorization" in headers:
        headers["authorization"] = headers["authorization"][:15] + "..."
    return json.dumps(headers, indent=2, ensure_ascii=False)
WARNING

ctx.request는 Streamable HTTP 전송에서만 사용 가능하다. stdio 전송에서는 None이므로 반드시 None 체크를 해야 한다. 두 전송 모두 지원하는 도구를 만들려면 if ctx.request: 분기를 항상 작성한다.

12.7 리소스 변경 알림

서버가 리소스를 제공하는 경우, 리소스의 내용이 변경되거나 리소스 목록 자체가 변경되었을 때 클라이언트에 알릴 수 있다. 이를 통해 클라이언트는 캐시를 무효화하고 최신 데이터를 다시 요청할 수 있다.

OpenTelemetry 트레이싱 통합

OpenTelemetry를 사용하면 각 도구 호출을 span으로 추적하고, 분산 시스템에서의 요청 흐름을 시각화할 수 있다. 

# 필요한 패키지 설치
pip install opentelemetry-api opentelemetry-sdk opentelemetry-exporter-otlp
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.resources import Resource
from mcp.server.fastmcp import FastMCP, Context
import time

# OpenTelemetry 초기화
resource = Resource.create({"service.name": "mcp-server"})
provider = TracerProvider(resource=resource)
processor = BatchSpanProcessor(OTLPSpanExporter(endpoint="http://otel-collector:4317"))
provider.add_span_processor(processor)
trace.set_tracer_provider(provider)
tracer = trace.get_tracer("mcp-server")

mcp = FastMCP("Traced Server")

@mcp.tool()
async def search(ctx: Context, query: str) -> str:
    """데이터를 검색한다"""
    with tracer.start_as_current_span(
        "tool.search",
        attributes={
            "mcp.tool": "search",
            "mcp.query_length": len(query),
        }
    ) as span:
        await ctx.info(f"검색 시작: {query}")
        start = time.time()

        # 쿼리 파싱 단계
        with tracer.start_as_current_span("search.parse_query"):
            parsed = query.strip().lower()
            span.set_attribute("search.parsed_query", parsed)

        # DB 조회 단계
        with tracer.start_as_current_span("search.execute"):
            results = [f"결과-{i}" for i in range(3)]
            span.set_attribute("search.result_count", len(results))

        elapsed = time.time() - start
        span.set_attribute("mcp.duration_ms", elapsed * 1000)
        await ctx.info(f"검색 완료: {len(results)}건, {elapsed*1000:.1f}ms")

        return str(results)

Prometheus 메트릭 수집

도구 호출 횟수, 응답 시간, 에러율 등의 메트릭을 Prometheus로 수집하여 Grafana 대시보드에서 실시간 모니터링할 수 있다. 

from prometheus_client import Counter, Histogram, start_http_server
from mcp.server.fastmcp import FastMCP, Context
import time
import functools

# Prometheus 메트릭 정의
TOOL_CALLS = Counter(
    "mcp_tool_calls_total",
    "MCP 도구 호출 총 횟수",
    ["tool_name", "status"],
)
TOOL_DURATION = Histogram(
    "mcp_tool_duration_seconds",
    "MCP 도구 처리 시간 (초)",
    ["tool_name"],
    buckets=[0.01, 0.05, 0.1, 0.5, 1.0, 5.0],
)

mcp = FastMCP("Metrics Server")

def with_metrics(tool_name: str):
    """도구에 Prometheus 메트릭을 자동 수집하는 데코레이터"""
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(*args, **kwargs):
            start = time.time()
            try:
                result = await func(*args, **kwargs)
                TOOL_CALLS.labels(tool_name=tool_name, status="success").inc()
                return result
            except Exception as e:
                TOOL_CALLS.labels(tool_name=tool_name, status="error").inc()
                raise
            finally:
                TOOL_DURATION.labels(tool_name=tool_name).observe(
                    time.time() - start
                )
        return wrapper
    return decorator

@mcp.tool()
@with_metrics("query_data")
async def query_data(ctx: Context, sql: str) -> str:
    """데이터를 쿼리한다"""
    await ctx.info(f"쿼리 실행: {sql}")
    return "쿼리 결과"

    return result

12.9 멀티테넌시

하나의 MCP 서버가 여러 테넌트(사용자/조직)를 동시에 서비스할 때, 요청별 컨텍스트 격리와 사용자별 로깅이 필수적이다. HTTP 헤더에서 테넌트 정보를 추출하고, 로그에 테넌트 ID를 포함시키는 패턴을 구현해 보자. 

import structlog
from mcp.server.fastmcp import FastMCP, Context

logger = structlog.get_logger("mcp_server")
mcp = FastMCP("Multi-tenant Server")

def get_tenant_info(ctx: Context) -> dict:
    """HTTP 헤더에서 테넌트 정보를 추출한다"""
    if not ctx.request:
        return {"tenant_id": "local", "user_id": "anonymous"}

    return {
        "tenant_id": ctx.request.headers.get("x-tenant-id", "default"),
        "user_id": ctx.request.headers.get("x-user-id", "anonymous"),
        "client_ip": ctx.request.client.host if ctx.request.client else "unknown",
    }

@mcp.tool()
async def tenant_query(ctx: Context, query: str) -> str:
    """테넌트별로 격리된 쿼리를 실행한다"""
    tenant = get_tenant_info(ctx)

    # 테넌트별 구조화 로깅
    log = logger.bind(**tenant)
    log.info("query_start", query=query)

    # MCP 로그에도 테넌트 정보 포함
    await ctx.info({
        "event": "query_start",
        **tenant,
        "query": query,
    })

    # 테넌트별 데이터 소스 분기
    if tenant["tenant_id"] == "org-alpha":
        results = "Alpha 조직 전용 데이터"
    elif tenant["tenant_id"] == "org-beta":
        results = "Beta 조직 전용 데이터"
    else:
        results = "기본 데이터"

    log.info("query_complete", result_length=len(results))
    await ctx.info(f"쿼리 완료: {len(results)}자")

    return results

@mcp.tool()
async def tenant_stats(ctx: Context) -> str:
    """현재 테넌트의 사용 통계를 반환한다"""
    tenant = get_tenant_info(ctx)
    await ctx.info({
        "event": "stats_request",
        **tenant,
    })
    return f"테넌트 '{tenant['tenant_id']}' 통계: 도구 호출 15회, 평균 응답 120ms"
Deep Dive: ContextVar를 통한 요청 격리

Python의 contextvars.ContextVar는 비동기 태스크마다 독립적인 값을 유지한다. MCP SDK가 request_ctx를 ContextVar로 관리하기 때문에, 서로 다른 테넌트의 요청이 동시에 처리되더라도 각 핸들러는 자신의 요청 컨텍스트에만 접근한다. 별도의 스레드 로컬 변수나 명시적 잠금 없이도 안전한 멀티테넌시가 보장되는 것이다. 단, 전역 변수(딕셔너리, 리스트 등)를 테넌트 간에 공유할 때는 명시적인 키 분리가 필요하다.

12.10 실전 예제: 프로덕션급 요청 추적 서버

지금까지 배운 모든 기능을 통합한 프로덕션급 MCP 서버를 구현한다. 요청 ID 발급, 처리 시간 측정, 에러 추적, 통계 수집, OpenTelemetry span, Prometheus 메트릭, 구조화 로깅을 모두 포함하는 완전한 예제다. 

"""
프로덕션급 요청 추적 MCP 서버

실행:
  pip install mcp[cli] opentelemetry-api opentelemetry-sdk structlog
  python production_server.py
"""
import time
import uuid
import asyncio
import functools
from dataclasses import dataclass, field
from typing import Any

import structlog
from opentelemetry import trace

from mcp.server.fastmcp import FastMCP, Context

# ── 구조화 로깅 설정 ─────────────────────────────────
structlog.configure(
    processors=[
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ],
    wrapper_class=structlog.stdlib.BoundLogger,
    logger_factory=structlog.PrintLoggerFactory(),
)
server_logger = structlog.get_logger("mcp_production")

# ── OpenTelemetry 트레이서 ───────────────────────────
tracer = trace.get_tracer("mcp-production-server")

# ── 통계 저장소 ──────────────────────────────────────
@dataclass
class ToolStats:
    total_calls: int = 0
    total_time: float = 0.0
    errors: int = 0
    last_called: str = ""

    @property
    def avg_time_ms(self) -> float:
        if self.total_calls == 0:
            return 0.0
        return (self.total_time / self.total_calls) * 1000


stats_store: dict[str, ToolStats] = {}

# ── 추적 데코레이터 ─────────────────────────────────
def tracked(tool_name: str):
    """도구 호출을 종합적으로 추적하는 데코레이터.
    요청 ID, 처리 시간, 에러, OTel span, 구조화 로그를 자동 수집한다."""
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(ctx: Context, *args, **kwargs):
            request_id = str(uuid.uuid4())[:8]
            start = time.time()
            log = server_logger.bind(
                request_id=request_id,
                tool=tool_name,
            )

            # 통계 초기화
            if tool_name not in stats_store:
                stats_store[tool_name] = ToolStats()

            with tracer.start_as_current_span(
                f"tool.{tool_name}",
                attributes={
                    "mcp.tool": tool_name,
                    "mcp.request_id": request_id,
                },
            ) as span:
                # MCP + 구조화 로그 동시 발행
                await ctx.info({
                    "event": "request_start",
                    "request_id": request_id,
                    "tool": tool_name,
                })
                log.info("request_start")

                try:
                    result = await func(ctx, *args, **kwargs)
                    elapsed = time.time() - start

                    # 통계 업데이트
                    s = stats_store[tool_name]
                    s.total_calls += 1
                    s.total_time += elapsed
                    s.last_called = time.strftime("%Y-%m-%d %H:%M:%S")

                    # span 속성
                    span.set_attribute("mcp.duration_ms", elapsed * 1000)
                    span.set_attribute("mcp.status", "success")

                    await ctx.info({
                        "event": "request_end",
                        "request_id": request_id,
                        "elapsed_ms": round(elapsed * 1000, 2),
                        "status": "success",
                    })
                    log.info("request_end", elapsed_ms=round(elapsed * 1000, 2))

                    return result

                except Exception as e:
                    elapsed = time.time() - start
                    s = stats_store[tool_name]
                    s.total_calls += 1
                    s.total_time += elapsed
                    s.errors += 1

                    span.set_attribute("mcp.status", "error")
                    span.set_attribute("mcp.error", str(e))
                    span.record_exception(e)

                    await ctx.error({
                        "event": "request_error",
                        "request_id": request_id,
                        "error": str(e),
                        "elapsed_ms": round(elapsed * 1000, 2),
                    })
                    log.error("request_error", error=str(e))
                    raise

        return wrapper
    return decorator


# ── MCP 서버 ─────────────────────────────────────────
mcp = FastMCP("Production Tracker")

@mcp.tool()
@tracked("search_database")
async def search_database(ctx: Context, query: str) -> str:
    """데이터베이스를 검색한다"""
    await ctx.report_progress(1, 3)
    await ctx.debug(f"쿼리 파싱: {query}")

    await ctx.report_progress(2, 3)
    await ctx.info("데이터베이스 검색 실행 중...")
    await asyncio.sleep(0.2)

    results = [
        {"id": 1, "title": f"{query} 관련 문서 1"},
        {"id": 2, "title": f"{query} 관련 문서 2"},
    ]

    await ctx.report_progress(3, 3)
    return str(results)

@mcp.tool()
@tracked("transform_data")
async def transform_data(ctx: Context, data: str, operation: str) -> str:
    """데이터를 변환한다"""
    operations = {
        "upper": str.upper,
        "lower": str.lower,
        "reverse": lambda s: s[::-1],
        "strip": str.strip,
    }

    if operation not in operations:
        raise ValueError(f"지원하지 않는 연산: {operation}")

    result = operations[operation](data)
    await ctx.info(f"변환 완료: {operation}")
    return result

@mcp.tool()
@tracked("get_stats")
async def get_stats(ctx: Context) -> str:
    """도구별 호출 통계를 반환한다"""
    if not stats_store:
        return "기록된 통계 없음"

    lines = ["=== 도구 호출 통계 ===\n"]
    for name, s in stats_store.items():
        lines.append(
            f"[{name}]\n"
            f"  총 호출: {s.total_calls}회\n"
            f"  평균 응답: {s.avg_time_ms:.2f}ms\n"
            f"  에러: {s.errors}회\n"
            f"  마지막 호출: {s.last_called}\n"
        )

    return "\n".join(lines)


if __name__ == "__main__":
    mcp.run(transport="streamable-http")

    S-->>C: result: [검색 결과]
TIP

프로덕션에서는 stats_store를 메모리 대신 Redis에 저장하면 서버 재시작 시에도 통계가 유지되고 여러 인스턴스 간에 통계를 공유할 수 있다. 또한 OpenTelemetry Collector를 Jaeger나 Tempo에 연결하면 도구 호출의 전체 트레이스를 시각적으로 분석할 수 있다.

INFO

tracked 데코레이터 패턴은 횡단 관심사(cross-cutting concern)를 깔끔하게 분리한다. 새로운 도구를 추가할 때 @tracked("tool_name") 한 줄만 추가하면 요청 추적, 에러 처리, 통계, 트레이싱이 모두 자동으로 적용된다.
