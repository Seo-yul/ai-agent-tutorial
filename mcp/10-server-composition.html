서버 컴포지션이란?

핵심 개념: 서버 컴포지션

MCP 서버가 커지면 하나의 서버에 모든 도구, 리소스, 프롬프트를 넣는 것은 유지보수가 어려워진다. 서버 컴포지션은 관심사별로 여러 MCP 서버를 분리하고, 이를 조합하여 하나의 통합 서비스를 구성하는 패턴이다. 마이크로서비스 아키텍처처럼 각 서버는 독립적으로 개발, 테스트, 배포할 수 있다.

서버 컴포지션이 필요한 세 가지 핵심 이유:

  • 관심사 분리 — 각 서버가 하나의 도메인(결제, 사용자, 알림 등)에 집중한다
  • 독립적 개발 — 서버별로 별도 팀이 개발하고, 의존성을 최소화한다
  • 재사용성 — 동일한 서버 모듈을 여러 애플리케이션에서 조합하여 사용한다

FastMCP 내장 컴포지션 API

FastMCP는 서버 컴포지션을 위한 두 가지 핵심 API를 제공한다: mount()와 import_server(). 이 둘은 비슷해 보이지만 동작 방식이 근본적으로 다르다.

mount() — 서버를 하위 서버로 마운트

mount()는 자식 서버를 별도의 독립 서버로 유지하면서 부모 서버에 연결한다. 자식 서버의 도구, 리소스, 프롬프트에 prefix가 자동으로 붙는다. 

from mcp.server.fastmcp import FastMCP

# 자식 서버 정의
users_server = FastMCP("Users")

@users_server.tool()
def get_user(user_id: int) -> dict:
    """사용자 정보를 조회한다"""
    return {"id": user_id, "name": "홍길동"}

@users_server.resource("users://list")
def list_users() -> str:
    """사용자 목록"""
    return "홍길동, 김철수, 이영희"


products_server = FastMCP("Products")

@products_server.tool()
def search_products(query: str) -> list[dict]:
    """상품을 검색한다"""
    return [{"id": 1, "name": query}]


# 부모 서버에 마운트
main_server = FastMCP("Main")
main_server.mount("users", users_server)
main_server.mount("products", products_server)

# 결과: 도구명이 "users_get_user", "products_search_products"가 된다
# 리소스 URI도 "users+users://list"로 변환된다

import_server() — 서버의 내용을 가져오기

import_server()는 자식 서버의 도구, 리소스, 프롬프트를 현재 서버에 통합한다. 자식 서버 자체는 사라지고, 그 내용만 부모 서버에 합쳐진다. 

from mcp.server.fastmcp import FastMCP

# 공통 유틸리티 서버
utils_server = FastMCP("Utils")

@utils_server.tool()
def format_date(date_str: str) -> str:
    """날짜를 포맷팅한다"""
    return date_str

@utils_server.tool()
def generate_id() -> str:
    """고유 ID를 생성한다"""
    import uuid
    return str(uuid.uuid4())


# 메인 서버에 import (prefix 적용)
main_server = FastMCP("Main")
main_server.import_server("utils", utils_server)

# 결과: 도구명이 "utils_format_date", "utils_generate_id"가 된다
# 하지만 이들은 main_server의 도구로 직접 등록된 것이다

mount vs import_server 비교

mount()
  • 자식 서버가 독립적으로 유지된다
  • 자식 서버의 Lifespan이 별도로 관리된다
  • 자식 서버에 대한 변경이 부모에 자동 반영된다
  • 런타임에 자식 서버를 동적으로 추가/제거 가능
  • 도구 호출 시 자식 서버의 컨텍스트에서 실행
import_server()
  • 자식 서버의 내용이 부모 서버에 복사/통합된다
  • import 후 자식 서버 객체는 더 이상 필요 없다
  • import 시점의 스냅샷 — 이후 변경은 반영되지 않음
  • import는 서버 시작 전 한 번만 수행
  • 도구 호출 시 부모 서버의 컨텍스트에서 실행
특성 mount() import_server()
자식 서버 독립성 유지 소멸 (내용만 복사)
동적 변경 반영 O X (스냅샷)
Lifespan 관리 자식 서버 자체 Lifespan 부모 서버 Lifespan에 통합
사용 시점 런타임에 동적 조합 시작 전 정적 조합
적합한 용도 마이크로서비스, 플러그인 시스템 코드 정리, 모듈 분리
TIP

코드 정리 목적으로 파일을 분리하고 싶을 때는 import_server()를, 독립적인 서비스를 조합할 때는 mount()를 사용하자. 대부분의 경우 mount()가 더 유연하다.

ASGI 기반 마운트

HTTP 전송을 사용하는 경우, 각 FastMCP 서버를 Starlette ASGI 앱으로 변환하여 서로 다른 경로에 마운트할 수 있다. 이 방식은 각 서버가 완전히 독립적인 HTTP 엔드포인트를 가진다. 

import contextlib
from starlette.applications import Starlette
from starlette.routing import Mount
from mcp.server.fastmcp import FastMCP

# 각 서버를 stateless HTTP 모드로 생성
echo_mcp = FastMCP(name="EchoServer", stateless_http=True, json_response=True)

@echo_mcp.tool()
def echo(message: str) -> str:
    """메시지를 그대로 반환한다"""
    return f"Echo: {message}"

math_mcp = FastMCP(name="MathServer", stateless_http=True, json_response=True)

@math_mcp.tool()
def add(a: int, b: int) -> int:
    """두 수를 더한다"""
    return a + b

# Lifespan에서 모든 세션 매니저를 시작/종료
@contextlib.asynccontextmanager
async def lifespan(app):
    async with contextlib.AsyncExitStack() as stack:
        await stack.enter_async_context(echo_mcp.session_manager.run())
        await stack.enter_async_context(math_mcp.session_manager.run())
        yield

# ASGI 앱으로 마운트
app = Starlette(
    routes=[
        Mount("/echo", echo_mcp.streamable_http_app()),
        Mount("/math", math_mcp.streamable_http_app()),
    ],
    lifespan=lifespan,
)

경로 설정

기본적으로 streamable_http_app()은 /mcp 경로에서 요청을 수신한다. 따라서 Mount("/echo", ...)로 마운트하면 실제 엔드포인트는 /echo/mcp가 된다. 

# streamable_http_path를 "/"로 설정하면 마운트 경로가 곧 엔드포인트
echo_mcp.settings.streamable_http_path = "/"
math_mcp.settings.streamable_http_path = "/"

# 결과: /echo, /math가 각각 엔드포인트가 된다
설정 Echo 엔드포인트 Math 엔드포인트
기본값 (/mcp) /echo/mcp /math/mcp
streamable_http_path = "/" /echo /math
streamable_http_path = "/api" /echo/api /math/api
Deep Dive: FastMCP mount() vs ASGI Mount

FastMCP.mount()와 Starlette의 Mount()는 다른 수준에서 동작한다. FastMCP.mount()는 MCP 프로토콜 수준에서 도구/리소스를 prefix로 연결하는 것이고, Starlette Mount()는 HTTP 라우팅 수준에서 ASGI 앱을 경로에 바인딩하는 것이다. FastMCP.mount()를 사용하면 단일 HTTP 엔드포인트에서 여러 서버의 도구를 모두 노출할 수 있고, ASGI Mount()를 사용하면 각 서버가 완전히 별개의 엔드포인트를 가진다.

Lifespan 관리

여러 MCP 서버를 조합할 때 각 서버의 초기화와 종료 순서가 중요하다. AsyncExitStack 패턴을 사용하면 모든 서버를 안전하게 관리할 수 있다. 

import contextlib
import logging

logger = logging.getLogger(__name__)


@contextlib.asynccontextmanager
async def lifespan(app):
    """앱의 라이프사이클을 관리한다

    AsyncExitStack은 등록된 역순으로 종료를 보장한다.
    따라서 의존되는 서버를 먼저 등록하고, 의존하는 서버를 나중에 등록한다.
    """
    async with contextlib.AsyncExitStack() as stack:
        # 1. 기반 서비스 (DB, 캐시 등)
        await stack.enter_async_context(db_mcp.session_manager.run())
        logger.info("DB 서버 시작 완료")

        # 2. 비즈니스 서비스 (DB에 의존)
        await stack.enter_async_context(api_mcp.session_manager.run())
        await stack.enter_async_context(chat_mcp.session_manager.run())
        logger.info("비즈니스 서버 시작 완료")

        # 3. 관리 서비스 (모든 서버에 의존)
        await stack.enter_async_context(admin_mcp.session_manager.run())
        logger.info("관리 서버 시작 완료")

        yield  # 서버 운영 중

        # 종료 시: admin → chat → api → db 순으로 정리
        logger.info("모든 MCP 서버 종료 시작")

서버 간 데이터 공유

분리된 서버들이 공통 리소스(DB 연결, 캐시, 설정 등)를 공유해야 하는 경우가 많다. 세 가지 패턴을 살펴보자.

패턴 1: 공유 Python 객체

같은 프로세스 내에서 실행되는 서버들은 Python 객체를 직접 참조할 수 있다:

from dataclasses import dataclass, field
from mcp.server.fastmcp import FastMCP


@dataclass
class SharedState:
    """여러 서버가 공유하는 상태"""
    cache: dict = field(default_factory=dict)
    request_count: int = 0


# 공유 상태 객체
shared = SharedState()


# 서버 A: 캐시에 쓰기
server_a = FastMCP("Writer")

@server_a.tool()
def write_cache(key: str, value: str) -> str:
    """캐시에 값을 저장한다"""
    shared.cache[key] = value
    shared.request_count += 1
    return f"저장 완료: {key}={value}"


# 서버 B: 캐시에서 읽기
server_b = FastMCP("Reader")

@server_b.tool()
def read_cache(key: str) -> str:
    """캐시에서 값을 읽는다"""
    shared.request_count += 1
    return shared.cache.get(key, "없음")

패턴 2: DB 연결 공유

import contextlib
from starlette.applications import Starlette
from starlette.routing import Mount
from mcp.server.fastmcp import FastMCP

# DB 연결 풀 (예시)
db_pool = None

users_mcp = FastMCP("Users", stateless_http=True)
orders_mcp = FastMCP("Orders", stateless_http=True)


@users_mcp.tool()
def get_user(user_id: int) -> dict:
    """사용자를 조회한다"""
    # 공유 DB 풀 사용
    conn = db_pool.acquire()
    try:
        return conn.execute("SELECT * FROM users WHERE id = ?", (user_id,))
    finally:
        db_pool.release(conn)


@orders_mcp.tool()
def get_orders(user_id: int) -> list[dict]:
    """사용자의 주문을 조회한다"""
    conn = db_pool.acquire()
    try:
        return conn.execute("SELECT * FROM orders WHERE user_id = ?", (user_id,))
    finally:
        db_pool.release(conn)


@contextlib.asynccontextmanager
async def lifespan(app):
    global db_pool
    # DB 풀 초기화 — 모든 서버가 공유
    db_pool = await create_pool("postgresql://localhost/mydb")

    async with contextlib.AsyncExitStack() as stack:
        await stack.enter_async_context(users_mcp.session_manager.run())
        await stack.enter_async_context(orders_mcp.session_manager.run())
        yield

    # DB 풀 종료
    await db_pool.close()

패턴 3: Lifespan을 통한 의존성 주입

from contextlib import asynccontextmanager
from mcp.server.fastmcp import FastMCP


@asynccontextmanager
async def users_lifespan(server: FastMCP):
    """Users 서버의 Lifespan — DB 연결 초기화"""
    import aiosqlite
    db = await aiosqlite.connect("users.db")
    # 서버 컨텍스트에 DB 연결을 저장
    yield {"db": db}
    await db.close()


users_mcp = FastMCP("Users", lifespan=users_lifespan)


@users_mcp.tool()
def get_user(user_id: int, ctx=None) -> dict:
    """사용자를 조회한다"""
    # ctx.request_context.lifespan_context에서 DB 접근
    db = ctx.request_context.lifespan_context["db"]
    return {"id": user_id, "name": "홍길동"}

에러 격리

서버 컴포지션에서 하나의 서버 장애가 다른 서버에 미치는 영향을 이해하고, 격리하는 방법을 살펴보자.

ASGI 마운트의 에러 격리

ASGI 기반으로 각 서버를 별도 경로에 마운트하면, 각 서버는 독립적인 요청 처리 파이프라인을 가진다. 한 서버에서 발생한 예외는 해당 서버의 응답에만 영향을 미친다: 

unstable_mcp = FastMCP("Unstable", stateless_http=True)

@unstable_mcp.tool()
def risky_operation() -> str:
    """불안정한 작업을 수행한다"""
    raise RuntimeError("서버 장애 발생!")  # 이 에러는 이 서버에만 영향

stable_mcp = FastMCP("Stable", stateless_http=True)

@stable_mcp.tool()
def safe_operation() -> str:
    """안정적인 작업을 수행한다"""
    return "정상 동작"  # unstable_mcp의 장애와 무관

app = Starlette(
    routes=[
        Mount("/unstable", unstable_mcp.streamable_http_app()),
        Mount("/stable", stable_mcp.streamable_http_app()),
    ],
    lifespan=lifespan,
)
# /unstable에서 에러가 발생해도 /stable은 정상 동작한다

FastMCP mount()의 에러 전파

FastMCP.mount()로 연결된 서버는 하나의 MCP 세션 안에서 동작한다. 따라서 자식 서버의 도구에서 발생한 예외는 해당 tools/call 응답에만 에러로 표시되며, 다른 도구 호출에는 영향을 주지 않는다. MCP 프로토콜이 도구 호출 단위로 에러를 격리하기 때문이다.

WARNING

자식 서버의 Lifespan에서 예외가 발생하면 부모 서버 전체의 시작이 실패할 수 있다. Lifespan 코드에는 적절한 예외 처리를 반드시 추가하자.

프록시/게이트웨이 패턴

원격에서 실행 중인 여러 MCP 서버를 하나의 진입점으로 묶어 제공하는 패턴이다. 게이트웨이 서버가 클라이언트의 요청을 받아 적절한 원격 서버로 라우팅한다.

"""gateway_server.py — 원격 MCP 서버들을 프록시하는 게이트웨이"""

from mcp.server.fastmcp import FastMCP
from mcp.client.streamable_http import streamable_http_client
from mcp.client.session import ClientSession

gateway = FastMCP("Gateway")

# 원격 서버 URL 매핑
REMOTE_SERVERS = {
    "users": "http://10.0.1.1:8001/mcp",
    "analytics": "http://10.0.1.2:8002/mcp",
    "storage": "http://10.0.1.3:8003/mcp",
}


@gateway.tool()
async def proxy_call(
    server: str,
    tool_name: str,
    arguments: dict,
) -> dict:
    """원격 MCP 서버의 도구를 호출한다

    Args:
        server: 대상 서버 (users, analytics, storage)
        tool_name: 호출할 도구 이름
        arguments: 도구 인자
    """
    url = REMOTE_SERVERS.get(server)
    if not url:
        raise ValueError(f"알 수 없는 서버: {server}")

    async with streamable_http_client(url) as (r, w, _):
        async with ClientSession(r, w) as session:
            await session.initialize()
            result = await session.call_tool(tool_name, arguments)
            return {
                "server": server,
                "tool": tool_name,
                "result": [c.text for c in result.content if hasattr(c, "text")],
            }


@gateway.tool()
async def list_remote_tools(server: str) -> list[dict]:
    """원격 서버의 도구 목록을 조회한다"""
    url = REMOTE_SERVERS.get(server)
    if not url:
        raise ValueError(f"알 수 없는 서버: {server}")

    async with streamable_http_client(url) as (r, w, _):
        async with ClientSession(r, w) as session:
            await session.initialize()
            tools = await session.list_tools()
            return [
                {"name": t.name, "description": t.description}
                for t in tools.tools
            ]
TIP

프록시 패턴은 인증, 로깅, 속도 제한 등 교차 관심사(cross-cutting concerns)를 게이트웨이에서 일괄 처리할 수 있다는 장점이 있다. 원격 서버들은 내부 네트워크에서 인증 없이 운영하고, 게이트웨이에서만 외부 인증을 수행하는 구조가 가능하다.

"""microservice_app.py — 마이크로서비스 MCP 아키텍처"""

import contextlib
from datetime import datetime
from starlette.applications import Starlette
from starlette.routing import Mount
from mcp.server.fastmcp import FastMCP


# ============================================================
# API 서버: 데이터 조회 관련 도구
# ============================================================

api_mcp = FastMCP(name="APIServer", stateless_http=True, json_response=True)
api_mcp.settings.streamable_http_path = "/"


@api_mcp.tool()
def get_users() -> list[dict]:
    """등록된 사용자 목록을 조회한다"""
    return [
        {"id": 1, "name": "홍길동", "role": "admin"},
        {"id": 2, "name": "김철수", "role": "user"},
        {"id": 3, "name": "이영희", "role": "user"},
    ]


@api_mcp.tool()
def get_products(category: str = "all") -> list[dict]:
    """상품 목록을 조회한다"""
    products = [
        {"id": 101, "name": "노트북", "category": "electronics", "price": 1500000},
        {"id": 102, "name": "마우스", "category": "electronics", "price": 35000},
        {"id": 103, "name": "프로그래밍 책", "category": "books", "price": 32000},
    ]
    if category != "all":
        products = [p for p in products if p["category"] == category]
    return products


# ============================================================
# Chat 서버: 대화 관련 도구
# ============================================================

chat_mcp = FastMCP(name="ChatServer", stateless_http=True, json_response=True)
chat_mcp.settings.streamable_http_path = "/"


@chat_mcp.tool()
def send_message(channel: str, message: str) -> dict:
    """채널에 메시지를 전송한다"""
    return {
        "status": "sent",
        "channel": channel,
        "message": message,
        "timestamp": datetime.now().isoformat(),
    }


@chat_mcp.tool()
def get_channels() -> list[str]:
    """사용 가능한 채널 목록을 반환한다"""
    return ["general", "random", "dev", "design", "marketing"]


@chat_mcp.tool()
def get_recent_messages(channel: str, limit: int = 5) -> list[dict]:
    """채널의 최근 메시지를 조회한다"""
    return [
        {
            "user": "홍길동",
            "message": f"{channel} 채널 메시지 {i}",
            "time": "10:00",
        }
        for i in range(1, limit + 1)
    ]


# ============================================================
# Admin 서버: 관리 도구
# ============================================================

admin_mcp = FastMCP(name="AdminServer", stateless_http=True, json_response=True)
admin_mcp.settings.streamable_http_path = "/"


@admin_mcp.tool()
def get_server_status() -> dict:
    """서버 상태를 조회한다"""
    return {
        "status": "healthy",
        "uptime": "72h 15m",
        "cpu_usage": "23%",
        "memory_usage": "45%",
        "active_connections": 127,
    }


@admin_mcp.tool()
def get_logs(level: str = "info", limit: int = 10) -> list[dict]:
    """서버 로그를 조회한다"""
    return [
        {
            "level": level,
            "message": f"샘플 {level} 로그 #{i}",
            "timestamp": datetime.now().isoformat(),
        }
        for i in range(1, limit + 1)
    ]


# ============================================================
# 앱 조합
# ============================================================

@contextlib.asynccontextmanager
async def lifespan(app):
    """모든 MCP 서버의 라이프사이클을 관리한다"""
    async with contextlib.AsyncExitStack() as stack:
        await stack.enter_async_context(api_mcp.session_manager.run())
        await stack.enter_async_context(chat_mcp.session_manager.run())
        await stack.enter_async_context(admin_mcp.session_manager.run())
        yield


app = Starlette(
    routes=[
        Mount("/api", api_mcp.streamable_http_app()),
        Mount("/chat", chat_mcp.streamable_http_app()),
        Mount("/admin", admin_mcp.streamable_http_app()),
    ],
    lifespan=lifespan,
)

"""각 서버에 독립적으로 연결하는 클라이언트"""
import asyncio
from mcp.client.streamable_http import streamable_http_client
from mcp.client.session import ClientSession


async def main():
    endpoints = {
        "API": "http://localhost:8000/api",
        "Chat": "http://localhost:8000/chat",
        "Admin": "http://localhost:8000/admin",
    }

    for name, url in endpoints.items():
        async with streamable_http_client(url) as (r, w, _):
            async with ClientSession(r, w) as session:
                await session.initialize()
                tools = await session.list_tools()
                print(f"\n[{name} Server] 도구 {len(tools.tools)}개:")
                for tool in tools.tools:
                    print(f"  - {tool.name}: {tool.description}")


if __name__ == "__main__":
    asyncio.run(main())
TIP

각 MCP 서버를 별도의 파이썬 모듈(servers/api.py, servers/chat.py 등)로 분리한 후 메인 앱에서 import하여 마운트하면 코드 구조가 훨씬 깔끔해진다. 팀이 성장하면 각 서버를 별도 리포지토리로 분리하고, 패키지로 설치하여 사용할 수도 있다.
