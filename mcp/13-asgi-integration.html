Chapter 13. Starlette/ASGI 통합

기존 웹 애플리케이션에 MCP 서버를 통합한다

13.1 왜 ASGI 통합인가?

실무에서는 이미 운영 중인 웹 애플리케이션이 있는 경우가 많다. 이때 MCP 서버를 별도 프로세스로 띄우는 대신, 기존 ASGI 앱에 MCP 엔드포인트를 마운트하면 하나의 서버에서 REST API와 MCP를 동시에 제공할 수 있다.

ASGI 통합의 주요 이점은 다음과 같다.

  • 단일 배포 — 하나의 프로세스와 포트로 모든 것을 관리한다.
  • 자원 공유 — 데이터베이스 연결 풀, 캐시 등을 REST API와 MCP가 공유할 수 있다.
  • 미들웨어 재사용 — 인증, CORS, 로깅 등의 미들웨어를 한 번만 설정하면 된다.
  • 운영 단순화 — 모니터링, 로드 밸런싱, 배포 파이프라인이 하나로 통일된다.

13.2 기본 마운트

Starlette 앱에 MCP 서버를 마운트하는 가장 기본적인 방법을 살펴보자. FastMCP의 streamable_http_app() 메서드가 반환하는 ASGI 앱을 Starlette의 Mount에 연결하면 된다. 

import contextlib
from starlette.applications import Starlette
from starlette.routing import Mount, Route
from starlette.responses import JSONResponse
from mcp.server.fastmcp import FastMCP

# MCP 서버 생성
mcp = FastMCP("API Server", json_response=True)

@mcp.tool()
def hello(name: str) -> str:
    """인사한다"""
    return f"안녕하세요, {name}님!"

# 일반 REST API 핸들러
async def health_check(request):
    return JSONResponse({"status": "ok"})

async def api_users(request):
    return JSONResponse({"users": ["김철수", "이영희"]})

# lifespan으로 MCP 세션 매니저를 관리한다
@contextlib.asynccontextmanager
async def lifespan(app):
    async with mcp.session_manager.run():
        yield

app = Starlette(
    routes=[
        Route("/health", health_check),
        Route("/api/users", api_users),
        Mount("/mcp", app=mcp.streamable_http_app()),
    ],
    lifespan=lifespan,
)
INFO

lifespan 핸들러는 앱 시작 시 MCP 세션 매니저를 초기화하고, 앱 종료 시 정리하는 역할을 한다. 이를 설정하지 않으면 MCP 엔드포인트가 정상적으로 동작하지 않는다.

13.3 uvicorn으로 실행

위에서 만든 앱을 server.py로 저장한 후, uvicorn으로 실행한다. 

# 개발 모드 (자동 리로드 활성화)
uvicorn server:app --host 0.0.0.0 --port 8000 --reload

# 프로덕션 모드
uvicorn server:app --host 0.0.0.0 --port 8000 --workers 1

실행 후 접근 가능한 엔드포인트는 다음과 같다.

경로 메서드 설명
GET /health GET REST API — 헬스 체크
GET /api/users GET REST API — 사용자 목록
POST /mcp/mcp POST MCP — 요청 전송 (Streamable HTTP)
GET /mcp/mcp GET MCP — 스트림 연결 (Streamable HTTP)
DELETE /mcp/mcp DELETE MCP — 세션 종료

REST API 엔드포인트를 curl로 테스트할 수 있다.

# 헬스 체크
curl http://localhost:8000/health
# 출력: {"status": "ok"}

# 사용자 목록
curl http://localhost:8000/api/users
# 출력: {"users": ["김철수", "이영희"]}
WARNING

MCP 엔드포인트의 경로가 /mcp/mcp인 이유는, Mount("/mcp", ...)로 마운트하고 FastMCP의 기본 streamable_http_path가 "/mcp"이기 때문이다. 경로를 변경하려면 다음 섹션의 경로 커스터마이징을 참고한다.

13.4 경로 커스터마이징

기본적으로 FastMCP의 Streamable HTTP 경로는 /mcp다. 이를 변경하면 마운트 경로와 조합하여 최종 엔드포인트 경로를 원하는 대로 설정할 수 있다. 

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("Custom Path Server")

# streamable_http_path를 "/"로 설정하면
# Mount 경로가 곧 MCP 엔드포인트 경로가 된다
mcp.settings.streamable_http_path = "/"

# 이제 Mount("/mcp", ...) → POST /mcp 가 MCP 엔드포인트
# Mount("/api/v1/mcp", ...) → POST /api/v1/mcp 가 MCP 엔드포인트

다양한 경로 조합 예시는 다음과 같다.

Mount 경로 streamable_http_path 최종 MCP 엔드포인트
/mcp /mcp (기본값) /mcp/mcp
/mcp / /mcp
/api/v1/mcp / /api/v1/mcp
/ /mcp /mcp
TIP

일반적으로 streamable_http_path = "/"로 설정하고, Mount 경로에서 전체 경로를 관리하는 것이 직관적이다.

13.5 미들웨어 활용

Starlette의 미들웨어를 사용하면 REST API와 MCP 엔드포인트 모두에 공통 기능(CORS, 인증, 요청 로깅 등)을 적용할 수 있다. 

import contextlib
from starlette.applications import Starlette
from starlette.routing import Mount, Route
from starlette.responses import JSONResponse
from starlette.middleware import Middleware
from starlette.middleware.cors import CORSMiddleware
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("Middleware Example")

@mcp.tool()
def ping() -> str:
    """서버 상태를 확인한다"""
    return "pong"

async def health(request):
    return JSONResponse({"status": "ok"})

@contextlib.asynccontextmanager
async def lifespan(app):
    async with mcp.session_manager.run():
        yield

app = Starlette(
    routes=[
        Route("/health", health),
        Mount("/mcp", app=mcp.streamable_http_app()),
    ],
    middleware=[
        # CORS 미들웨어 — 모든 출처 허용
        Middleware(
            CORSMiddleware,
            allow_origins=["*"],
            allow_methods=["*"],
            allow_headers=["*"],
        ),
    ],
    lifespan=lifespan,
)

커스텀 미들웨어를 작성하여 요청 로깅, 인증, 속도 제한 등을 구현할 수도 있다. 

import time
import logging
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response, JSONResponse

logger = logging.getLogger("mcp-server")

class RequestLoggingMiddleware(BaseHTTPMiddleware):
    """모든 요청의 처리 시간을 로깅하는 미들웨어"""

    async def dispatch(self, request: Request, call_next):
        start = time.time()
        response = await call_next(request)
        elapsed = (time.time() - start) * 1000

        logger.info(
            f"{request.method} {request.url.path} "
            f"→ {response.status_code} ({elapsed:.1f}ms)"
        )
        # 응답 헤더에 처리 시간 추가
        response.headers["X-Process-Time"] = f"{elapsed:.1f}ms"
        return response

class APIKeyAuthMiddleware(BaseHTTPMiddleware):
    """API 키 인증 미들웨어 (MCP 경로에만 적용)"""

    VALID_API_KEYS = {"key-abc-123", "key-def-456"}

    async def dispatch(self, request: Request, call_next):
        # MCP 경로에만 인증 적용
        if request.url.path.startswith("/mcp"):
            api_key = request.headers.get("x-api-key", "")
            if api_key not in self.VALID_API_KEYS:
                return JSONResponse(
                    {"error": "유효하지 않은 API 키"},
                    status_code=401,
                )
        return await call_next(request)

# 미들웨어 적용
app = Starlette(
    routes=[...],
    middleware=[
        Middleware(CORSMiddleware, allow_origins=["*"], allow_methods=["*"]),
        Middleware(RequestLoggingMiddleware),
        Middleware(APIKeyAuthMiddleware),
    ],
    lifespan=lifespan,
)
WARNING

미들웨어는 선언 순서대로 바깥에서 안쪽으로 감싸지므로, 인증 미들웨어를 CORS 미들웨어보다 뒤에 선언해야 프리플라이트(OPTIONS) 요청이 인증 없이 통과한다. 순서를 잘못 설정하면 브라우저 클라이언트에서 CORS 오류가 발생할 수 있다.

13.6 다중 서버 마운트

하나의 Starlette 앱에 여러 MCP 서버를 경로별로 분리하여 마운트할 수 있다. 도메인별로 도구를 분리하면 관심사 분리가 명확해지고, 각 서버를 독립적으로 관리할 수 있다. 

import contextlib
from starlette.applications import Starlette
from starlette.routing import Mount
from mcp.server.fastmcp import FastMCP

# API 관련 MCP 서버
api_mcp = FastMCP("API", stateless_http=True, json_response=True)

@api_mcp.tool()
def get_user(user_id: int) -> str:
    """사용자 정보를 조회한다"""
    users = {1: "김철수", 2: "이영희", 3: "박지민"}
    name = users.get(user_id, "알 수 없음")
    return f"사용자 {user_id}: {name}"

@api_mcp.tool()
def list_products() -> str:
    """상품 목록을 조회한다"""
    return "상품: [노트북, 키보드, 마우스]"

# 채팅 관련 MCP 서버
chat_mcp = FastMCP("Chat", stateless_http=True, json_response=True)

@chat_mcp.tool()
def send_message(channel: str, message: str) -> str:
    """메시지를 전송한다"""
    return f"[{channel}] 메시지 전송됨: {message}"

@chat_mcp.tool()
def get_history(channel: str, limit: int = 10) -> str:
    """채팅 기록을 조회한다"""
    return f"[{channel}] 최근 {limit}개 메시지 조회됨"

# 두 MCP 서버의 세션 매니저를 모두 관리하는 lifespan
@contextlib.asynccontextmanager
async def lifespan(app):
    async with contextlib.AsyncExitStack() as stack:
        await stack.enter_async_context(api_mcp.session_manager.run())
        await stack.enter_async_context(chat_mcp.session_manager.run())
        yield

app = Starlette(
    routes=[
        Mount("/api-mcp", app=api_mcp.streamable_http_app()),
        Mount("/chat-mcp", app=chat_mcp.streamable_http_app()),
    ],
    lifespan=lifespan,
)

stateless_http=True를 설정하면 서버가 세션 상태를 유지하지 않으므로, 여러 인스턴스로 수평 확장(스케일 아웃)이 용이하다. 상태가 필요 없는 간단한 도구 서버에 적합하다.

13.7 실전 예제: REST + MCP 통합 서버

사용자 관리 REST API와 이를 활용하는 MCP 도구를 하나의 서버에서 제공하는 완성 예제를 구현한다. REST API로 CRUD 작업을 수행하고, MCP 도구를 통해 LLM이 같은 데이터에 접근할 수 있다. 

"""
REST + MCP 통합 서버: 사용자 관리 시스템

실행:
  uvicorn user_server:app --host 0.0.0.0 --port 8000 --reload
"""
import contextlib
import json
from dataclasses import dataclass, field, asdict
from starlette.applications import Starlette
from starlette.routing import Mount, Route
from starlette.requests import Request
from starlette.responses import JSONResponse
from starlette.middleware import Middleware
from starlette.middleware.cors import CORSMiddleware
from mcp.server.fastmcp import FastMCP, Context


# ── 데이터 모델 ──────────────────────────────────────────

@dataclass
class User:
    id: int
    name: str
    email: str
    role: str = "member"

    def to_dict(self) -> dict:
        return asdict(self)


# ── 인메모리 데이터 저장소 ──────────────────────────────────

@dataclass
class UserStore:
    """사용자 데이터를 관리하는 저장소 (REST와 MCP가 공유)"""
    users: dict[int, User] = field(default_factory=dict)
    next_id: int = 1

    def create(self, name: str, email: str, role: str = "member") -> User:
        user = User(id=self.next_id, name=name, email=email, role=role)
        self.users[self.next_id] = user
        self.next_id += 1
        return user

    def get(self, user_id: int) -> User | None:
        return self.users.get(user_id)

    def list_all(self) -> list[User]:
        return list(self.users.values())

    def update(self, user_id: int, **kwargs) -> User | None:
        user = self.users.get(user_id)
        if not user:
            return None
        for key, value in kwargs.items():
            if hasattr(user, key) and value is not None:
                setattr(user, key, value)
        return user

    def delete(self, user_id: int) -> bool:
        return self.users.pop(user_id, None) is not None

    def search(self, query: str) -> list[User]:
        query = query.lower()
        return [
            u for u in self.users.values()
            if query in u.name.lower() or query in u.email.lower()
        ]


# 전역 저장소 인스턴스 (REST API와 MCP 도구가 공유)
store = UserStore()

# 초기 데이터 삽입
store.create("김철수", "chulsoo@example.com", "admin")
store.create("이영희", "younghee@example.com", "member")
store.create("박지민", "jimin@example.com", "member")


# ── REST API 핸들러 ──────────────────────────────────────

async def rest_health(request: Request):
    """헬스 체크"""
    return JSONResponse({"status": "ok", "service": "user-management"})

async def rest_list_users(request: Request):
    """사용자 목록 조회"""
    users = store.list_all()
    return JSONResponse({"users": [u.to_dict() for u in users]})

async def rest_get_user(request: Request):
    """사용자 단건 조회"""
    user_id = int(request.path_params["user_id"])
    user = store.get(user_id)
    if not user:
        return JSONResponse({"error": "사용자를 찾을 수 없음"}, status_code=404)
    return JSONResponse({"user": user.to_dict()})

async def rest_create_user(request: Request):
    """사용자 생성"""
    body = await request.json()
    name = body.get("name")
    email = body.get("email")
    role = body.get("role", "member")

    if not name or not email:
        return JSONResponse(
            {"error": "name과 email은 필수 항목"},
            status_code=400,
        )

    user = store.create(name, email, role)
    return JSONResponse({"user": user.to_dict()}, status_code=201)

async def rest_update_user(request: Request):
    """사용자 수정"""
    user_id = int(request.path_params["user_id"])
    body = await request.json()

    user = store.update(
        user_id,
        name=body.get("name"),
        email=body.get("email"),
        role=body.get("role"),
    )
    if not user:
        return JSONResponse({"error": "사용자를 찾을 수 없음"}, status_code=404)
    return JSONResponse({"user": user.to_dict()})

async def rest_delete_user(request: Request):
    """사용자 삭제"""
    user_id = int(request.path_params["user_id"])
    if not store.delete(user_id):
        return JSONResponse({"error": "사용자를 찾을 수 없음"}, status_code=404)
    return JSONResponse({"message": f"사용자 {user_id} 삭제됨"})


# ── MCP 서버 설정 ─────────────────────────────────────────

mcp = FastMCP("User Management MCP", json_response=True)
mcp.settings.streamable_http_path = "/"

@mcp.tool()
async def search_users(ctx: Context, query: str) -> str:
    """사용자를 이름 또는 이메일로 검색한다"""
    await ctx.info(f"검색어: {query}")
    results = store.search(query)

    if not results:
        await ctx.info("검색 결과 없음")
        return "검색 결과가 없다."

    await ctx.info(f"{len(results)}명의 사용자를 찾음")
    lines = []
    for u in results:
        lines.append(f"• {u.name} ({u.email}) — 역할: {u.role}, ID: {u.id}")
    return "\n".join(lines)

@mcp.tool()
async def get_user_info(ctx: Context, user_id: int) -> str:
    """사용자 ID로 상세 정보를 조회한다"""
    user = store.get(user_id)
    if not user:
        await ctx.warning(f"사용자 {user_id}를 찾을 수 없음")
        return f"ID {user_id}에 해당하는 사용자가 없다."

    await ctx.info(f"사용자 조회 완료: {user.name}")
    return json.dumps(user.to_dict(), ensure_ascii=False, indent=2)

@mcp.tool()
async def create_new_user(
    ctx: Context, name: str, email: str, role: str = "member"
) -> str:
    """새 사용자를 생성한다"""
    await ctx.info(f"사용자 생성: {name} ({email})")
    user = store.create(name, email, role)
    await ctx.info({"event": "user_created", "user_id": user.id})
    return f"사용자 생성 완료: ID={user.id}, 이름={user.name}, 이메일={user.email}"

@mcp.tool()
async def get_user_statistics(ctx: Context) -> str:
    """사용자 통계를 반환한다"""
    users = store.list_all()
    total = len(users)
    role_counts: dict[str, int] = {}
    for u in users:
        role_counts[u.role] = role_counts.get(u.role, 0) + 1

    await ctx.info({"event": "stats_query", "total_users": total})

    lines = [
        f"총 사용자 수: {total}명",
        "",
        "역할별 분포:",
    ]
    for role, count in sorted(role_counts.items()):
        lines.append(f"  • {role}: {count}명")

    return "\n".join(lines)


# ── 앱 조합 ──────────────────────────────────────────────

@contextlib.asynccontextmanager
async def lifespan(app):
    async with mcp.session_manager.run():
        yield

app = Starlette(
    routes=[
        # REST API 라우트
        Route("/health", rest_health),
        Route("/api/users", rest_list_users, methods=["GET"]),
        Route("/api/users", rest_create_user, methods=["POST"]),
        Route("/api/users/{user_id:int}", rest_get_user, methods=["GET"]),
        Route("/api/users/{user_id:int}", rest_update_user, methods=["PUT"]),
        Route("/api/users/{user_id:int}", rest_delete_user, methods=["DELETE"]),

        # MCP 엔드포인트
        Mount("/mcp", app=mcp.streamable_http_app()),
    ],
    middleware=[
        Middleware(
            CORSMiddleware,
            allow_origins=["*"],
            allow_methods=["*"],
            allow_headers=["*"],
        ),
    ],
    lifespan=lifespan,
)

TIP

이 패턴의 핵심은 UserStore를 REST API와 MCP 도구가 공유한다는 점이다. 실제 프로젝트에서는 UserStore 대신 SQLAlchemy, MongoDB 등의 데이터베이스 계층을 사용하면 동일한 데이터에 대해 REST와 MCP 두 가지 인터페이스를 자연스럽게 제공할 수 있다.

INFO

운영 환경에서는 인증·인가 미들웨어를 추가하여, REST API 요청에는 JWT를, MCP 요청에는 API 키를 검증하는 방식으로 보안을 강화하는 것이 좋다. 13.5절의 미들웨어 예제를 참고한다.
