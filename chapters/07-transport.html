Chapter 7. Transport

MCP의 통신 방식인 Transport를 이해하고 선택한다

7.1 Transport란?

Transport는 MCP 서버와 클라이언트 사이의 통신 계층이다. 프로토콜 메시지(JSON-RPC)를 어떤 방식으로 주고받을지 정의하며, MCP의 핵심 기능과는 독립적으로 동작한다.

MCP Python SDK는 현재 두 가지 Transport를 지원한다.

  • stdio — 표준 입출력(stdin/stdout)을 통한 통신
  • Streamable HTTP — HTTP 기반의 스트리밍 통신

7.2 stdio Transport

stdio Transport는 표준 입출력(stdin/stdout)을 통해 메시지를 교환하는 방식이다. 클라이언트가 서버 프로세스를 직접 실행하고, 프로세스의 stdin/stdout 파이프를 통해 JSON-RPC 메시지를 주고받는다.

사용 방법

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("my-server")

# 도구 등록
@mcp.tool()
def hello(name: str) -> str:
    """인사 메시지를 반환한다"""
    return f"안녕하세요, {name}!"

# stdio로 실행 (기본값)
mcp.run()

# 또는 명시적으로 지정
mcp.run(transport="stdio")

Claude Desktop 설정

Claude Desktop에서 stdio 서버를 연결하려면 claude_desktop_config.json에 다음과 같이 설정한다.

{
  "mcpServers": {
    "my-server": {
      "command": "uv",
      "args": ["run", "/path/to/server.py"]
    }
  }
}

stdio의 특징

항목 설명
통신 방식 프로세스의 stdin/stdout 파이프
사용 시나리오 로컬 통합, Claude Desktop, IDE 플러그인
네트워크 불필요 (로컬 프로세스 간 통신)
설정 복잡도 매우 낮음
다중 클라이언트 불가 (1:1 연결)
장점 설정이 간단하고 보안 걱정이 적다
단점 원격 접근 불가, 클라이언트당 별도 프로세스 필요
TIP

stdio Transport를 사용할 때는 서버 코드에서 print() 문을 사용하지 않도록 주의한다. stdout이 MCP 통신 채널로 사용되므로, 디버그 출력은 반드시 stderr이나 로깅 모듈을 사용해야 한다.

7.3 Streamable HTTP Transport

Streamable HTTP Transport는 HTTP를 기반으로 메시지를 교환하는 방식이다. 서버가 HTTP 엔드포인트를 열고, 클라이언트가 HTTP 요청으로 연결한다. 원격 서버 배포, 다중 클라이언트 접속이 필요한 경우에 사용한다.

기본 사용법

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("my-server")

@mcp.tool()
def hello(name: str) -> str:
    """인사 메시지를 반환한다"""
    return f"안녕하세요, {name}!"

# Streamable HTTP로 실행
mcp.run(transport="streamable-http", host="0.0.0.0", port=8000)

위 코드를 실행하면 http://0.0.0.0:8000/mcp에 MCP 엔드포인트가 열린다.

설정 옵션

# 모든 옵션을 명시적으로 지정
mcp.run(
    transport="streamable-http",
    host="0.0.0.0",        # 바인딩할 호스트 (기본: "0.0.0.0")
    port=8000,             # 바인딩할 포트 (기본: 8000)
    path="/mcp",           # 엔드포인트 경로 (기본: "/mcp")
)
옵션 기본값 설명
host "0.0.0.0" 서버가 바인딩할 네트워크 인터페이스
port 8000 서버가 리스닝할 포트 번호
path "/mcp" MCP 엔드포인트 경로

Streamable HTTP의 특징

항목 설명
통신 방식 HTTP POST + Server-Sent Events(SSE)
사용 시나리오 원격 배포, 웹 서비스, 클라우드 환경
네트워크 필요
설정 복잡도 중간
다중 클라이언트 가능
장점 원격 접근 가능, 다중 클라이언트 지원, 표준 HTTP 인프라 활용
단점 네트워크 설정 필요, 보안(인증/HTTPS) 별도 구성 필요
WARNING

SSE(Server-Sent Events) Transport는 deprecated되었으며 Streamable HTTP로 대체되었다. 기존에 transport="sse"를 사용하던 코드는 transport="streamable-http"로 마이그레이션해야 한다.

어떤 Transport를 선택할지는 서버의 배포 환경과 요구사항에 따라 결정한다. 다음 비교표를 참고하여 적합한 Transport를 선택하자.

사용 시나리오 로컬 통합 원격 배포
설정 복잡도 낮음 중간
네트워크 불필요 필요
다중 클라이언트 불가 가능

배포 로컬 전용 Docker, 클라우드 등 유연

Transport 선택 의사결정 트리: 서버를 어디서 실행하는가? ├── 로컬 (같은 머신) │ └── stdio 선택 ✓ │ • Claude Desktop, IDE 플러그인 │ • 설정 간단, 보안 걱정 적음 │ └── 원격 (다른 머신 / 클라우드) └── Streamable HTTP 선택 ✓ • 웹 서비스, API 서버 • 다중 클라이언트 지원 필요 • HTTPS + 인증 설정 권장
TIP

개발 중에는 stdio를 사용하고, 프로덕션 배포 시에는 Streamable HTTP로 전환하는 전략이 일반적이다. MCP 서버 코드를 변경할 필요 없이 mcp.run()의 transport 파라미터만 바꾸면 된다.

7.5 Stateless HTTP 모드

Streamable HTTP Transport는 기본적으로 세션 상태를 유지한다. 클라이언트와 서버 간의 연결이 유지되는 동안 세션 정보가 보존된다. 그러나 특정 상황에서는 상태 비유지(Stateless) 모드가 필요하다.

Stateless 모드란?

Stateless 모드에서는 각 HTTP 요청이 독립적으로 처리된다. 서버가 클라이언트의 세션 상태를 유지하지 않으므로, 수평 확장(scale-out)이 용이하다. 

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("stateless-server")

@mcp.tool()
def calculate(expression: str) -> str:
    """수식을 계산한다 (상태 비유지)"""
    try:
        # 주의: 실제 프로덕션에서는 eval을 사용하지 않는다
        result = eval(expression, {"__builtins__": {}})
        return f"{expression} = {result}"
    except Exception as e:
        return f"계산 오류: {e}"

# Stateless HTTP 모드로 실행
mcp.run(
    transport="streamable-http",
    host="0.0.0.0",
    port=8000,
    stateless_http=True,
)

Stateless 모드의 사용 시나리오

  • 로드 밸런서 뒤에서 실행 — 여러 서버 인스턴스 중 어느 것이 요청을 처리해도 동일한 결과를 보장한다
  • 서버리스 환경 — AWS Lambda, Google Cloud Functions 등 상태를 유지할 수 없는 환경
  • 단순 도구 서버 — 이전 요청 컨텍스트가 불필요한 독립적인 도구만 제공하는 서버
Stateful vs Stateless 비교: Stateful (기본): 요청 1 ──▶ 세션 생성 ──▶ 상태 저장 요청 2 ──▶ 세션 조회 ──▶ 상태 활용 (같은 서버 인스턴스) 요청 3 ──▶ 세션 조회 ──▶ 상태 활용 Stateless: 요청 1 ──▶ 처리 ──▶ 응답 (서버 A) 요청 2 ──▶ 처리 ──▶ 응답 (서버 B) ← 다른 인스턴스도 가능 요청 3 ──▶ 처리 ──▶ 응답 (서버 C)
INFO

Stateless 모드에서는 세션 기반의 기능(예: 서버 측 상태 관리, 구독 등)이 제한된다. 서버의 도구가 모두 독립적으로 동작 가능한 경우에만 사용한다.
