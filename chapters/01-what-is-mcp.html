MCP란 무엇인가

Model Context Protocol의 개념과 아키텍처를 이해한다

MCP란?

MCP(Model Context Protocol)는 Anthropic이 만든 개방형 프로토콜이다. LLM 애플리케이션이 외부 데이터 소스 및 도구와 통신하기 위한 표준 인터페이스를 정의한다.

USB-C가 하나의 커넥터로 충전, 데이터 전송, 영상 출력을 모두 처리하듯이, MCP는 하나의 프로토콜로 LLM이 다양한 외부 서비스와 소통할 수 있게 해준다. 즉, AI를 위한 USB-C라고 할 수 있다.

INFO

MCP는 2024년 11월 Anthropic에 의해 오픈 소스로 공개되었으며, 현재 다양한 LLM 애플리케이션에서 채택되고 있다.

왜 MCP가 필요한가?

MCP가 없던 시절을 생각해보자. N개의 LLM 애플리케이션과 M개의 외부 도구가 있다면, 각 앱이 각 도구에 대한 연동 코드를 따로 작성해야 한다. 이는 N × M개의 통합 코드를 만들어야 한다는 뜻이다. 

  MCP 없이 (N × M 문제)                  MCP 사용 (N + M 해결)
  ─────────────────────                  ─────────────────────

  Claude Desktop ──┬── Slack             Claude Desktop ──┐
  Cursor       ──┼── GitHub                  Cursor ──┤
  VS Code      ──┼── PostgreSQL            VS Code ──┤
  Custom App   ──┴── Google Drive        Custom App ──┘
                                                      │
  각 앱이 모든 도구에 대해                         MCP 프로토콜
  개별 연동 필요 → 12개 연동                         │
                                                ┌────┴────┐
                                             Slack  GitHub ...
                                            서버    서버

                                          각 도구는 MCP 서버 1개만
                                          만들면 됨 → 4 + 4 = 8개 연동

MCP를 사용하면 각 도구는 MCP 서버를 하나만 구현하면 되고, 각 LLM 앱은 MCP 클라이언트를 하나만 구현하면 된다. 결과적으로 N + M개의 통합만 필요하다.

MCP 아키텍처

MCP 아키텍처는 3가지 핵심 구성요소로 이루어진다.

  ┌─────────────────────────────────────────────────┐
  │                    Host                         │
  │           (LLM 애플리케이션)                      │
  │                                                 │
  │   ┌────────────┐  ┌────────────┐                │
  │   │  Client A  │  │  Client B  │   ...          │
  │   └──────┬─────┘  └──────┬─────┘                │
  └──────────┼───────────────┼──────────────────────┘
             │               │
      MCP 프로토콜       MCP 프로토콜
        (1:1 연결)        (1:1 연결)
             │               │
  ┌──────────┴──────┐ ┌──────┴──────────┐
  │    Server A     │ │    Server B     │
  │  (파일 시스템)   │ │   (데이터베이스) │
  └─────────────────┘ └─────────────────┘

Host (호스트)

사용자가 직접 상호작용하는 LLM 애플리케이션이다. Claude Desktop, IDE(Cursor, VS Code), 또는 직접 만든 AI 앱이 해당된다. Host는 하나 이상의 Client를 내부에 관리한다.

Client (클라이언트)

Host 내부에서 동작하며, 하나의 Server와 1:1 연결을 유지한다. 프로토콜 협상, 메시지 라우팅, 기능(capability) 관리를 담당한다. 사용자가 직접 Client를 다루는 경우는 드물다.

Server (서버)

외부 도구나 데이터를 MCP 프로토콜로 노출하는 경량 프로그램이다. 파일 시스템, 데이터베이스, API 등 다양한 리소스를 LLM이 사용할 수 있도록 만들어준다. 이 튜토리얼에서 우리가 주로 만들게 되는 것이 바로 MCP Server이다.

TIP

대부분의 MCP 개발은 Server를 만드는 작업이다. Python SDK의 FastMCP 클래스를 사용하면 몇 줄의 코드로 서버를 구축할 수 있다.

3가지 프리미티브

MCP Server는 3가지 프리미티브(primitive)를 통해 기능을 제공한다.

프리미티브 설명 제어 주체 예시
Tools LLM이 호출하는 함수. 부수 효과(side effect)를 가질 수 있다. 모델(LLM)이 호출 계산, API 호출, 파일 쓰기
Resources LLM이 읽는 데이터. 읽기 전용이며 부수 효과가 없다. 애플리케이션이 제어 파일 내용, DB 레코드, 설정 값
Prompts 미리 정의된 프롬프트 템플릿. 재사용 가능한 상호작용 패턴이다. 사용자가 선택 코드 리뷰 프롬프트, 번역 프롬프트
WARNING

Tools는 외부 시스템에 변경을 가할 수 있으므로 (예: 파일 삭제, 이메일 전송), 반드시 사용자 확인 과정을 거치도록 설계해야 한다. Resources는 읽기 전용이므로 비교적 안전하다.

MCP 통신 흐름

사용자가 LLM에 요청을 보내면 MCP 통신이 어떻게 이루어지는지 살펴보자.

  사용자                Host (LLM 앱)             MCP Server
    │                      │                        │
    │  "서울 날씨 알려줘"   │                        │
    │ ────────────────────▶│                        │
    │                      │                        │
    │                      │  1. LLM이 도구 필요    │
    │                      │     여부 판단           │
    │                      │                        │
    │                      │  2. Client를 통해      │
    │                      │     도구 호출 요청      │
    │                      │ ──────────────────────▶│
    │                      │                        │
    │                      │                        │  3. 외부 API 호출
    │                      │                        │     (날씨 API)
    │                      │                        │
    │                      │  4. 결과 반환           │
    │                      │ ◀──────────────────────│
    │                      │                        │
    │                      │  5. LLM이 결과를       │
    │                      │     자연어로 정리       │
    │                      │                        │
    │  "서울의 현재 기온은   │                        │
    │   15°C이고..."       │                        │
    │ ◀────────────────────│                        │
    │                      │                        │

핵심 흐름을 정리하면 다음과 같다:

  1. 사용자 → Host: 자연어로 요청을 보낸다.
  2. Host 내부 LLM: 요청을 분석하고, 필요한 도구(Tool)가 있는지 판단한다.
  3. Client → Server: LLM이 도구를 사용하기로 결정하면, Client가 해당 Server에 JSON-RPC 요청을 보낸다.
  4. Server: 요청을 처리하고 (외부 API 호출, 파일 읽기 등) 결과를 반환한다.
  5. Host 내부 LLM: 반환된 결과를 바탕으로 사용자에게 자연어 응답을 생성한다.
INFO

MCP는 JSON-RPC 2.0을 기반으로 통신한다. 전송 계층은 stdio(표준 입출력) 또는 Streamable HTTP를 사용할 수 있으며, 로컬 서버는 주로 stdio를, 원격 서버는 Streamable HTTP를 사용한다.
