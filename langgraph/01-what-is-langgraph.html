LangGraph란 무엇인가

그래프 기반 AI 에이전트 프레임워크를 이해한다

LangGraph란?

LangGraph는 LangChain 팀이 개발한 Python 프레임워크로, AI 에이전트의 실행 흐름을 그래프(Graph) 구조로 모델링한다. 전통적인 프로그래밍에서 함수를 순차적으로 호출하는 방식과 달리, LangGraph는 각 작업 단위를 노드(Node)로 정의하고 노드 간의 연결을 엣지(Edge)로 표현하여 복잡한 AI 워크플로우를 직관적으로 구성할 수 있게 해준다.

LangGraph의 핵심 아이디어는 간단하다. AI 에이전트의 동작을 상태 머신(State Machine)으로 표현하는 것이다. 그래프의 각 노드가 상태를 받아 처리하고, 처리 결과에 따라 다음 노드로 이동하거나 조건에 따라 분기한다. 이 방식은 복잡한 에이전트 흐름을 시각적으로 이해하기 쉽게 만들어 주며, 디버깅과 유지보수를 크게 개선한다. 

graph TD
    START_NODE((START)) --> A["Node A"]
    A --> B["Node B"]
    B --> END_NODE((END))
    A -.->|"조건부 엣지"| C["Node C"]
    C --> END_NODE2((END))

LangGraph는 단독으로 사용할 수도 있고, LangChain과 함께 사용할 수도 있다. LangChain의 도구(Tools), 프롬프트(Prompts), LLM 인터페이스를 LangGraph의 노드 안에서 자유롭게 활용할 수 있기 때문에 기존 LangChain 사용자라면 자연스럽게 전환할 수 있다.

왜 LangGraph인가?

기존의 AI 에이전트 프레임워크들은 몇 가지 근본적인 한계를 가지고 있었다. 이러한 한계를 이해하면 LangGraph가 왜 필요한지 명확해진다.

기존 에이전트 프레임워크의 문제점

1. 단순 체이닝의 한계: 초기 LLM 애플리케이션은 프롬프트 → LLM → 파서 → 출력과 같은 단순한 직렬 체인으로 구성되었다. 하지만 실제 프로덕션 환경에서는 조건에 따라 다른 처리를 해야 하거나, 루프를 돌거나, 여러 경로를 병렬로 실행해야 하는 경우가 빈번하다. 단순 체이닝으로는 이런 복잡한 흐름을 표현하기 어렵다.

2. 에러 처리와 재시도의 어려움: LLM 호출은 본질적으로 불안정하다. API 타임아웃, 토큰 한도 초과, 잘못된 출력 형식 등 다양한 오류가 발생할 수 있다. 기존 프레임워크에서는 이런 에러를 체계적으로 처리하고 특정 단계만 재시도하는 것이 까다로웠다.

3. 상태 관리의 부재: 에이전트가 여러 단계를 거치면서 중간 결과를 기억하고 활용해야 하는 경우, 상태를 명시적으로 관리하는 메커니즘이 필요하다. 대부분의 프레임워크는 이를 개발자에게 전적으로 맡겼다.

4. 사람의 개입(Human-in-the-loop) 지원 부족: 중요한 결정을 내리기 전에 사람의 승인을 받거나, 에이전트의 판단을 사람이 수정해야 하는 경우가 많다. 이런 워크플로우를 자연스럽게 지원하는 프레임워크가 드물었다.

핵심 개념

LangGraph를 이해하기 위해 반드시 알아야 하는 핵심 개념들을 하나씩 살펴본다. 이 개념들은 이후 모든 챕터의 기초가 된다.

State (상태)

State는 그래프를 통해 전달되는 공유 데이터 구조이다. Python의 TypedDict를 사용하여 정의하며, 모든 노드가 이 State를 읽고 수정할 수 있다. State는 그래프 실행의 시작부터 끝까지 유지되며, 각 노드가 처리를 마칠 때마다 업데이트된다. 

from typing_extensions import TypedDict

class State(TypedDict):
    """그래프에서 사용할 상태 정의"""
    question: str      # 사용자 질문
    answer: str        # 생성된 답변
    context: list[str] # 참조 문서 목록

Node (노드)

Node는 실제 작업을 수행하는 단위이다. Python 함수로 정의하며, State를 입력으로 받아 업데이트할 부분만 딕셔너리로 반환한다. LLM 호출, 데이터베이스 조회, API 호출 등 어떤 작업이든 노드 안에서 수행할 수 있다. 

def search_node(state: State) -> dict:
    """질문에 관련된 문서를 검색하는 노드"""
    query = state["question"]
    # 검색 로직 수행
    documents = search_database(query)
    return {"context": documents}  # State의 context만 업데이트

Edge (엣지)

Edge는 노드 간의 연결을 정의한다. 크게 두 가지 종류가 있다:

  • 일반 엣지 (Normal Edge): A 노드 실행 후 항상 B 노드로 이동
  • 조건부 엣지 (Conditional Edge): 조건에 따라 다른 노드로 분기
# 일반 엣지: node_a 다음에 항상 node_b 실행
graph.add_edge("node_a", "node_b")

# 조건부 엣지: 조건 함수의 반환값에 따라 분기
def decide_next(state: State) -> str:
    if state["score"] > 0.8:
        return "good_path"
    return "retry_path"

graph.add_conditional_edges("evaluate", decide_next)

START와 END

START는 그래프의 진입점이다. 그래프가 실행되면 START에서 시작하여 연결된 첫 번째 노드로 이동한다. END는 그래프의 종료 지점이다. 노드가 END로 연결되면 그래프 실행이 완료되고 최종 State가 반환된다. 

from langgraph.graph import START, END, StateGraph

graph = StateGraph(State)
graph.add_edge(START, "first_node")   # 시작점 → 첫 번째 노드
graph.add_edge("last_node", END)      # 마지막 노드 → 종료

StateGraph와 compile()

StateGraph는 그래프를 정의하는 클래스이다. 여기에 노드와 엣지를 추가하여 그래프의 구조를 설계한다. 설계가 완료되면 compile() 메서드를 호출하여 실행 가능한 CompiledGraph 객체를 생성한다. 

# 1. 그래프 정의
graph = StateGraph(State)

# 2. 노드 추가
graph.add_node("search", search_node)
graph.add_node("generate", generate_node)

# 3. 엣지 추가
graph.add_edge(START, "search")
graph.add_edge("search", "generate")
graph.add_edge("generate", END)

# 4. 컴파일 → 실행 가능한 객체 생성
app = graph.compile()

invoke() 실행

invoke()는 컴파일된 그래프를 실행하는 메서드이다. 초기 State를 딕셔너리로 전달하면, 그래프가 START부터 END까지 순차적으로 실행되고 최종 State를 반환한다. 

# 초기 상태를 전달하여 그래프 실행
result = app.invoke({
    "question": "LangGraph란 무엇인가?",
    "answer": "",
    "context": []
})

# 최종 상태 확인
print(result["answer"])  # 생성된 답변 출력
TIP

invoke() 외에도 stream() 메서드를 사용하면 각 노드의 실행 결과를 실시간으로 스트리밍 받을 수 있다. 이는 사용자에게 진행 상황을 보여줄 때 유용하다.

LangGraph는 LangChain Agent를 대체하는 것이 아니라 보완하는 관계이다. 간단한 도구 사용 에이전트는 LangChain Agent로 충분하지만, 복잡한 워크플로우나 프로덕션 환경에서는 LangGraph가 더 적합하다. 실제로 LangChain 팀도 복잡한 에이전트 구축에는 LangGraph를 권장하고 있다.

그래프 실행 흐름 이해하기

가장 간단한 직렬 그래프의 실행 흐름을 단계별로 살펴본다. 아래 그래프는 START에서 시작하여 Node A와 Node B를 거쳐 END에 도달하는 구조이다. 

graph LR
    START_NODE((START)) -->|"초기 State\n{text: 시작}"| A["Node A"]
    A -->|"State 업데이트\n{text: 시작→A}"| B["Node B"]
    B -->|"State 업데이트\n{text: 시작→A→B}"| END_NODE((END))

실행 단계 상세

Step 1 — 초기화: invoke()에 전달된 초기 State가 그래프에 주입된다. 예를 들어 {"text": "시작"}이 초기 State이다.

Step 2 — START → Node A: START에서 첫 번째 노드인 Node A로 이동한다. Node A 함수가 현재 State를 인자로 받아 실행된다. Node A는 {"text": "시작 → A 처리됨"}을 반환하고, LangGraph가 이 반환값으로 State를 업데이트한다.

Step 3 — Node A → Node B: 업데이트된 State가 Node B로 전달된다. Node B는 받은 State를 처리하고 {"text": "시작 → A 처리됨 → B 처리됨"}을 반환한다. State가 다시 업데이트된다.

Step 4 — Node B → END: Node B의 처리가 완료되면 END로 이동하여 그래프 실행이 종료된다. 최종 State인 {"text": "시작 → A 처리됨 → B 처리됨"}이 invoke()의 반환값으로 돌아온다.

TIP

이 직렬 흐름은 가장 기본적인 패턴이다. 실제 에이전트에서는 조건부 엣지를 사용한 분기, 루프, 병렬 실행 등 더 복잡한 흐름을 구성하게 된다. 이후 챕터에서 이런 고급 패턴을 하나씩 배워 나간다.

정리

이번 챕터에서 배운 내용을 정리한다:

  • LangGraph는 AI 에이전트의 실행 흐름을 그래프로 모델링하는 프레임워크이다.
  • 핵심 구성 요소는 State(공유 데이터), Node(작업 단위), Edge(연결)이다.
  • StateGraph로 그래프를 정의하고, compile()로 컴파일한 뒤, invoke()로 실행한다.
  • LangGraph는 실행 지속성, 사람 개입, 메모리, 디버깅, 프로덕션 배포 등 5대 장점을 제공한다.
  • 기존 프레임워크 대비 세밀한 흐름 제어와 명시적 상태 관리가 LangGraph의 핵심 차별점이다.
