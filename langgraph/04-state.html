기본 과정

State 관리

그래프의 핵심인 State를 효과적으로 설계하고 관리한다

State란?

State는 LangGraph 그래프의 모든 노드가 공유하는 데이터 구조이다. 그래프가 실행되는 동안 State는 노드에서 노드로 전달되며, 각 노드는 State를 읽어 작업을 수행하고, 변경이 필요한 부분만 반환한다. LangGraph는 반환된 업데이트를 기존 State에 자동으로 병합(merge)한다. 

graph LR
    A["Node A\nState 읽기 / 일부 수정"] -->|"State 전달"| B["Node B\nState 읽기 / 일부 수정"]
    B -->|"State 전달"| C["Node C\nState 읽기 / 일부 수정"]
    A -.->|"{answer: ...}"| M1["State에 자동 병합"]
    B -.->|"{score: 0.9}"| M1
    C -.->|"{final: ...}"| M2["State에 자동 병합"]

State를 이해하는 것은 LangGraph를 마스터하는 핵심이다. State 설계가 잘못되면 노드 간 데이터 전달이 복잡해지고, 디버깅이 어려워진다. 반대로 State를 잘 설계하면 각 노드가 깔끔하게 분리되고 전체 흐름이 명확해진다.

TypedDict로 State 정의

LangGraph에서 State는 Python의 TypedDict를 사용하여 정의한다. TypedDict는 딕셔너리의 각 키에 타입을 지정할 수 있는 타입 힌트이다. 이를 통해 State에 어떤 데이터가 포함되는지 명시적으로 선언할 수 있다.

기본 State 정의

from typing_extensions import TypedDict


class State(TypedDict):
    """질의응답 에이전트의 상태"""
    question: str        # 사용자의 질문
    answer: str          # 생성된 답변
    context: list[str]   # 참조한 문서 목록

위 예제에서 State는 3개의 필드를 가진다. question은 문자열, answer도 문자열, context는 문자열 리스트이다. 이 타입 정보는 런타임에 강제되지는 않지만, IDE의 자동완성과 타입 검사기를 통해 개발 시 도움을 받을 수 있다.

더 복잡한 State 예시

from typing_extensions import TypedDict


class AgentState(TypedDict):
    """복잡한 에이전트의 상태 — 다양한 타입을 포함할 수 있다"""
    # 기본 타입
    query: str                    # 사용자 입력
    response: str                 # 최종 응답

    # 리스트 타입
    documents: list[str]          # 검색된 문서
    messages: list[dict]          # 대화 히스토리

    # 숫자 타입
    relevance_score: float        # 관련성 점수
    retry_count: int              # 재시도 횟수

    # 딕셔너리 타입
    metadata: dict                # 메타데이터

    # 불리언 타입
    is_relevant: bool             # 관련성 여부
    needs_retry: bool             # 재시도 필요 여부
TIP

State 필드의 이름은 그 역할이 명확하게 드러나도록 짓는 것이 좋다. data, info 같은 모호한 이름보다 search_results, relevance_score처럼 구체적인 이름을 사용한다. 주석을 함께 달아두면 다른 개발자가 코드를 이해하기 쉬워진다.

State 업데이트 규칙

LangGraph에서 State 업데이트의 기본 동작은 덮어쓰기(replace)이다. 노드가 반환하는 딕셔너리의 키에 해당하는 값만 업데이트되고, 반환하지 않은 필드는 기존 값이 그대로 유지된다.

기본 동작: 덮어쓰기

from langgraph.graph import START, END, StateGraph
from typing_extensions import TypedDict


class State(TypedDict):
    question: str
    answer: str
    score: float


def answer_node(state: State) -> dict:
    """answer와 score만 업데이트한다. question은 그대로 유지된다."""
    return {
        "answer": f"'{state['question']}'에 대한 답변이다.",
        "score": 0.95
    }
    # question은 반환하지 않았으므로 기존 값 유지


# 그래프 구성 및 실행
graph = StateGraph(State)
graph.add_node("answer", answer_node)
graph.add_edge(START, "answer")
graph.add_edge("answer", END)
app = graph.compile()

result = app.invoke({
    "question": "LangGraph란?",
    "answer": "",
    "score": 0.0
})

print(result)
# {
#   'question': 'LangGraph란?',     ← 변경되지 않음
#   'answer': "'LangGraph란?'에 대한 답변이다.",  ← 업데이트됨
#   'score': 0.95                    ← 업데이트됨
# }

여러 노드의 순차 업데이트

여러 노드가 같은 필드를 업데이트하면, 마지막 노드의 값이 최종 결과가 된다. 

def node_1(state: State) -> dict:
    """첫 번째 노드: answer를 설정"""
    return {"answer": "초기 답변"}


def node_2(state: State) -> dict:
    """두 번째 노드: answer를 덮어쓴다"""
    return {"answer": "수정된 답변"}


# node_1 → node_2 순서로 실행하면
# 최종 answer는 "수정된 답변"이 된다 (덮어쓰기)
WARNING

리스트 타입의 필드도 기본적으로 덮어쓰기가 적용된다. 즉, 기존 리스트에 항목을 추가하고 싶다면 기본 동작으로는 불가능하다. 이 문제를 해결하는 것이 바로 다음에 설명할 Reducer이다. 

# 덮어쓰기의 함정 — 리스트가 교체된다!

class BadState(TypedDict):
    items: list[str]


def add_item(state: BadState) -> dict:
    # 의도: 기존 리스트에 추가
    # 실제: 기존 리스트가 새 리스트로 교체됨!
    return {"items": ["새로운 항목"]}


# 실행 결과:
# 초기: {"items": ["기존 항목1", "기존 항목2"]}
# 결과: {"items": ["새로운 항목"]}  ← 기존 항목이 사라짐!

Reducer (Annotated)

Reducer는 State 필드가 업데이트될 때 어떻게 병합할지를 정의하는 함수이다. 기본 동작은 덮어쓰기이지만, Reducer를 사용하면 리스트에 append하거나, 숫자를 누적하거나, 커스텀 병합 로직을 적용할 수 있다.

Reducer는 Python의 Annotated 타입 힌트를 사용하여 State 필드에 지정한다.

operator.add로 리스트 이어붙이기

from typing import Annotated
from typing_extensions import TypedDict
import operator
from langgraph.graph import START, END, StateGraph


class State(TypedDict):
    # Annotated로 reducer 지정: operator.add는 리스트를 이어붙인다
    messages: Annotated[list[str], operator.add]
    count: int  # reducer 없음 → 기본 덮어쓰기


def node_a(state: State) -> dict:
    """메시지를 추가한다"""
    return {
        "messages": ["A에서 추가된 메시지"],  # 기존 리스트에 이어붙임
        "count": state["count"] + 1           # 덮어쓰기
    }


def node_b(state: State) -> dict:
    """메시지를 더 추가한다"""
    return {
        "messages": ["B에서 추가된 메시지 1", "B에서 추가된 메시지 2"],
        "count": state["count"] + 1
    }


# 그래프 구성
graph = StateGraph(State)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)
graph.add_edge(START, "node_a")
graph.add_edge("node_a", "node_b")
graph.add_edge("node_b", END)

app = graph.compile()

result = app.invoke({
    "messages": ["초기 메시지"],
    "count": 0
})

print("messages:", result["messages"])
# ['초기 메시지', 'A에서 추가된 메시지', 'B에서 추가된 메시지 1', 'B에서 추가된 메시지 2']

print("count:", result["count"])
# 2

Reducer의 동작 원리

Annotated[list[str], operator.add]의 의미를 분해하면:

  • list[str] — 이 필드의 타입은 문자열 리스트이다.
  • operator.add — 업데이트 시 current + update를 수행한다.

Python의 operator.add는 리스트에 대해 + 연산을 수행한다. 즉, [1, 2] + [3, 4]의 결과는 [1, 2, 3, 4]이다. 이를 통해 각 노드가 반환하는 리스트가 기존 리스트에 이어붙여진다.

Reducer 동작 과정 (operator.add): 초기 State: messages = ["초기 메시지"] Node A 반환: {"messages": ["A에서 추가"]} → reducer 적용: ["초기 메시지"] + ["A에서 추가"] → 결과: ["초기 메시지", "A에서 추가"] Node B 반환: {"messages": ["B-1", "B-2"]} → reducer 적용: ["초기 메시지", "A에서 추가"] + ["B-1", "B-2"] → 결과: ["초기 메시지", "A에서 추가", "B-1", "B-2"]
INFO

Reducer가 없는 필드(위 예제의 count)는 기본 덮어쓰기 동작을 한다. 따라서 count는 각 노드가 state["count"] + 1을 직접 계산하여 반환해야 한다. Reducer를 사용할지 말지는 필드의 특성에 따라 결정한다.

MessagesState

AI 에이전트에서 가장 흔한 패턴은 메시지 기반 대화이다. 사용자 메시지와 AI 응답이 리스트에 누적되는 구조이다. LangGraph는 이 패턴을 위해 MessagesState라는 미리 정의된 State를 제공한다. 

from langgraph.graph import MessagesState

# MessagesState는 내부적으로 다음과 같이 정의되어 있다:
# class MessagesState(TypedDict):
#     messages: Annotated[list[BaseMessage], add_messages]

MessagesState는 messages라는 단일 필드를 가지며, add_messages라는 특별한 Reducer를 사용한다.

add_messages Reducer의 특별한 기능

add_messages는 단순한 리스트 이어붙이기가 아니다. 메시지의 id를 기반으로 지능적인 업데이트를 수행한다:

  • 새 메시지 추가: 새로운 ID의 메시지는 리스트 끝에 추가된다.
  • 기존 메시지 업데이트: 같은 ID의 메시지가 이미 있으면 새 내용으로 교체한다.
  • 중복 방지: 같은 ID의 메시지가 중복 추가되지 않는다.
"""MessagesState 사용 예제"""

from langgraph.graph import START, END, StateGraph, MessagesState
from langchain_core.messages import HumanMessage, AIMessage


def chatbot(state: MessagesState) -> dict:
    """간단한 에코 챗봇 노드"""
    # 마지막 메시지 가져오기
    last_message = state["messages"][-1]

    # 응답 메시지 생성 (여기서는 LLM 대신 단순 에코)
    response = AIMessage(
        content=f"당신이 말한 내용: '{last_message.content}'"
    )

    # messages에 응답 추가 (add_messages reducer가 자동 처리)
    return {"messages": [response]}


# 그래프 구성
graph = StateGraph(MessagesState)
graph.add_node("chatbot", chatbot)
graph.add_edge(START, "chatbot")
graph.add_edge("chatbot", END)

app = graph.compile()

# 실행
result = app.invoke({
    "messages": [HumanMessage(content="안녕하세요!")]
})

for msg in result["messages"]:
    role = "사용자" if isinstance(msg, HumanMessage) else "AI"
    print(f"[{role}] {msg.content}")

# 출력:
# [사용자] 안녕하세요!
# [AI] 당신이 말한 내용: '안녕하세요!'

MessagesState 확장하기

MessagesState는 messages 필드만 제공하므로, 추가 필드가 필요하면 상속하여 확장할 수 있다. 

from langgraph.graph import MessagesState


class MyState(MessagesState):
    """MessagesState를 확장한 커스텀 State"""
    # messages 필드는 MessagesState에서 상속받는다
    user_name: str           # 사용자 이름
    conversation_id: str     # 대화 ID
    tool_results: list[dict] # 도구 실행 결과
TIP

메시지 기반 에이전트를 만들 때는 MessagesState를 상속하여 시작하는 것이 가장 효율적이다. messages 필드의 Reducer 설정을 직접 할 필요 없이 add_messages의 지능적인 업데이트 기능을 바로 사용할 수 있다.

커스텀 Reducer

operator.add만으로는 해결할 수 없는 복잡한 병합 로직이 필요할 때, 직접 Reducer 함수를 만들 수 있다. Reducer 함수는 두 개의 인자를 받는다: 현재 값(current)과 새로운 값(update). 그리고 병합된 결과를 반환한다.

중복 제거 Reducer

from typing import Annotated
from typing_extensions import TypedDict


def dedup_reducer(current: list[str], update: list[str]) -> list[str]:
    """중복을 제거하면서 리스트를 병합하는 Reducer"""
    # set을 사용하여 중복 제거 후 정렬하여 반환
    return sorted(set(current + update))


class State(TypedDict):
    tags: Annotated[list[str], dedup_reducer]

이 Reducer의 동작을 확인해 본다:

from langgraph.graph import START, END, StateGraph


def node_a(state: State) -> dict:
    return {"tags": ["python", "ai", "langgraph"]}


def node_b(state: State) -> dict:
    # "python"은 이미 있으므로 중복 제거됨
    return {"tags": ["python", "agent", "automation"]}


graph = StateGraph(State)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)
graph.add_edge(START, "node_a")
graph.add_edge("node_a", "node_b")
graph.add_edge("node_b", END)

app = graph.compile()

result = app.invoke({"tags": ["tutorial"]})
print(result["tags"])
# ['agent', 'ai', 'automation', 'langgraph', 'python', 'tutorial']
# → 중복 없이 정렬된 결과

최댓값 유지 Reducer

def max_reducer(current: float, update: float) -> float:
    """항상 더 큰 값을 유지하는 Reducer"""
    return max(current, update)


class ScoringState(TypedDict):
    query: str
    best_score: Annotated[float, max_reducer]  # 최고 점수만 유지

카운터 Reducer

def add_reducer(current: int, update: int) -> int:
    """값을 누적하는 Reducer"""
    return current + update


class CountState(TypedDict):
    text: str
    total_tokens: Annotated[int, add_reducer]  # 토큰 수 누적

# 각 노드에서 사용한 토큰 수만 반환하면 자동 누적
def llm_node(state: CountState) -> dict:
    # LLM 호출 후
    return {"total_tokens": 150}  # 이 호출에서 사용한 토큰 수

def another_llm_node(state: CountState) -> dict:
    return {"total_tokens": 200}  # 이 호출에서 사용한 토큰 수

# 실행 후 total_tokens = 0 + 150 + 200 = 350
WARNING

커스텀 Reducer 함수는 반드시 순수 함수(pure function)여야 한다. 외부 상태에 의존하거나 부수 효과(side effect)를 가져서는 안 된다. 또한 current나 update를 직접 수정(mutate)하면 안 된다. 항상 새로운 값을 생성하여 반환해야 한다.

실전 예제: RAG 파이프라인 State

지금까지 배운 State 관리 기법을 종합하여 RAG(Retrieval-Augmented Generation) 파이프라인의 State를 설계하고 구현한다. RAG는 질문에 관련된 문서를 검색하고, 검색 결과를 바탕으로 답변을 생성하는 패턴이다. 

graph LR
    START_NODE((START)) --> R["검색\n(retrieve)"] --> E["평가\n(evaluate)"] --> G["생성\n(generate)"] --> END_NODE((END))

"""실전 예제: RAG 파이프라인 State 관리"""

from typing import Annotated
from typing_extensions import TypedDict
import operator
from langgraph.graph import START, END, StateGraph


# ============================================================
# 커스텀 Reducer 정의
# ============================================================

def max_reducer(current: float, update: float) -> float:
    """최댓값을 유지하는 Reducer"""
    return max(current, update)


def unique_list_reducer(current: list[str], update: list[str]) -> list[str]:
    """중복 없이 리스트를 병합하는 Reducer"""
    seen = set(current)
    result = list(current)
    for item in update:
        if item not in seen:
            result.append(item)
            seen.add(item)
    return result


# ============================================================
# State 정의 — 다양한 Reducer 활용
# ============================================================

class RAGState(TypedDict):
    # 기본 필드 (덮어쓰기)
    question: str                                          # 사용자 질문
    answer: str                                            # 생성된 답변

    # operator.add Reducer — 리스트 이어붙이기
    documents: Annotated[list[str], operator.add]          # 검색된 문서
    scores: Annotated[list[float], operator.add]           # 관련성 점수

    # 커스텀 Reducer — 최댓값 유지
    best_score: Annotated[float, max_reducer]              # 최고 점수

    # 커스텀 Reducer — 중복 제거 리스트
    sources: Annotated[list[str], unique_list_reducer]     # 참조 출처


# ============================================================
# 노드 함수 정의
# ============================================================

# 시뮬레이션용 문서 데이터베이스
DOCUMENT_DB = {
    "langgraph": [
        "LangGraph는 LangChain 팀이 개발한 그래프 기반 에이전트 프레임워크이다.",
        "LangGraph는 StateGraph를 사용하여 노드와 엣지로 워크플로우를 정의한다.",
        "LangGraph는 체크포인트 기능으로 실행 지속성을 제공한다.",
    ],
    "agent": [
        "AI 에이전트는 LLM을 사용하여 자율적으로 작업을 수행하는 시스템이다.",
        "에이전트는 도구(Tool)를 호출하여 외부 시스템과 상호작용할 수 있다.",

def retrieve(state: RAGState) -> dict:
    """검색 노드: 질문과 관련된 문서를 검색한다"""
    question = state["question"].lower()
    found_docs = []

    # 간단한 키워드 매칭으로 문서 검색 (실제로는 벡터 검색 사용)
    for keyword, docs in DOCUMENT_DB.items():
        if keyword in question:
            found_docs.extend(docs)

    # 문서를 못 찾으면 기본 메시지
    if not found_docs:
        found_docs = ["관련 문서를 찾지 못했다."]

    print(f"[검색] {len(found_docs)}개 문서 발견")
    return {
        "documents": found_docs,  # operator.add로 이어붙임
        "sources": ["document_db"]  # unique_list_reducer로 중복 제거
    }


def evaluate(state: RAGState) -> dict:
    """평가 노드: 검색된 문서의 관련성을 평가한다"""
    documents = state["documents"]
    question = state["question"].lower()

    # 간단한 관련성 점수 계산 (실제로는 LLM 또는 임베딩 유사도 사용)
    doc_scores = []
    for doc in documents:
        # 질문의 키워드가 문서에 포함된 비율로 점수 계산
        keywords = question.split()
        matches = sum(1 for kw in keywords if kw in doc.lower())
        score = round(matches / max(len(keywords), 1), 2)
        doc_scores.append(score)

    best = max(doc_scores) if doc_scores else 0.0
    print(f"[평가] 점수: {doc_scores}, 최고: {best}")

    return {
        "scores": doc_scores,      # operator.add로 이어붙임
        "best_score": best          # max_reducer로 최댓값 유지
    }


def generate(state: RAGState) -> dict:
    """생성 노드: 검색된 문서를 바탕으로 답변을 생성한다"""
    documents = state["documents"]
    scores = state["scores"]
    question = state["question"]

    # 점수가 높은 문서만 선별 (0.3 이상)
    relevant_docs = [
        doc for doc, score in zip(documents, scores)
        if score >= 0.3
    ]

    if relevant_docs:
        # 관련 문서를 조합하여 답변 생성 (실제로는 LLM 사용)
        context = " ".join(relevant_docs)
        answer = f"질문: {question}\n\n"
        answer += f"답변: 검색된 {len(relevant_docs)}개의 관련 문서를 기반으로 답변한다.\n\n"
        for i, doc in enumerate(relevant_docs, 1):
            answer += f"  {i}. {doc}\n"
        answer += f"\n최고 관련성 점수: {state['best_score']}"
    else:
        answer = f"'{question}'에 대한 충분한 정보를 찾지 못했다."

    print(f"[생성] 답변 생성 완료 (관련 문서 {len(relevant_docs)}개 사용)")

    return {
        "answer": answer,          # 덮어쓰기
        "sources": ["llm_generate"] # unique_list_reducer로 중복 제거
    }


# ============================================================
# 그래프 구성 및 실행
# ============================================================

graph = StateGraph(RAGState)

graph.add_node("retrieve", retrieve)
graph.add_node("evaluate", evaluate)
graph.add_node("generate", generate)

graph.add_edge(START, "retrieve")
graph.add_edge("retrieve", "evaluate")
graph.add_edge("evaluate", "generate")
graph.add_edge("generate", END)

app = graph.compile()

# 그래프 구조 확인
app.get_graph().print_ascii()

# 실행
print("\n" + "=" * 60)
result = app.invoke({
    "question": "LangGraph란 무엇인가?",
    "answer": "",
    "documents": [],
    "scores": [],
    "best_score": 0.0,
    "sources": []
})

print("\n" + "=" * 60)
print("\n[최종 결과]")
print(f"답변:\n{result['answer']}")
print(f"\n문서 수: {len(result['documents'])}")
print(f"점수: {result['scores']}")

print(f"출처: {result['sources']}")

필드 Reducer 동작

TIP

State를 설계할 때는 각 필드의 업데이트 패턴을 먼저 생각하라. "이 필드는 덮어써야 하는가, 누적해야 하는가, 특별한 병합이 필요한가?"를 결정하면 적절한 Reducer를 선택할 수 있다. 대부분의 경우 덮어쓰기(기본)와 operator.add로 충분하며, 특수한 요구사항이 있을 때만 커스텀 Reducer를 만든다.

State 설계 가이드

좋은 State 설계를 위한 실용적인 가이드라인을 정리한다.

1. 필요한 필드만 정의한다

State에 불필요한 필드를 넣으면 코드가 복잡해지고 디버깅이 어려워진다. 실제로 노드 간에 전달이 필요한 데이터만 포함한다.

2. 필드 이름을 명확하게 짓는다

# 나쁜 예
class State(TypedDict):
    data: dict        # 무슨 데이터?
    result: str       # 어떤 결과?
    flag: bool        # 무엇을 나타내는 플래그?

# 좋은 예
class State(TypedDict):
    search_results: dict   # 검색 결과
    final_answer: str      # 최종 답변
    needs_retry: bool      # 재시도 필요 여부

3. Reducer 선택 기준

패턴 권장 Reducer 예시
최종 값만 필요 없음 (기본 덮어쓰기) answer, query, is_valid
리스트에 누적 operator.add messages, documents, logs
메시지 관리 add_messages 대화 히스토리 (MessagesState)
중복 제거 누적 커스텀 (set 기반) tags, categories, sources
최댓값/최솟값 커스텀 (max/min) best_score, min_latency
숫자 누적 커스텀 (덧셈) total_tokens, total_cost

4. 초기값을 잊지 않는다

invoke()에 전달하는 초기 State에 모든 필드의 값을 포함하는 것이 안전하다. 특히 Reducer가 있는 필드는 빈 리스트([])나 기본값(0, 0.0)을 명시해야 한다. 

# 초기 State — 모든 필드에 적절한 기본값을 설정한다
result = app.invoke({
    "question": "사용자 질문",
    "answer": "",            # 빈 문자열
    "documents": [],         # 빈 리스트
    "scores": [],            # 빈 리스트
    "best_score": 0.0,       # 기본 숫자
    "sources": []            # 빈 리스트
})

정리

이번 챕터에서 배운 내용을 정리한다:

  • State는 TypedDict로 정의하며, 그래프의 모든 노드가 공유하는 데이터 구조이다.
  • 기본 업데이트 동작은 덮어쓰기(replace)이다. 노드가 반환하지 않은 필드는 그대로 유지된다.
  • Reducer를 사용하면 리스트 이어붙이기, 최댓값 유지 등 커스텀 업데이트 로직을 정의할 수 있다.
  • Annotated[type, reducer_fn] 형식으로 Reducer를 지정한다.
  • MessagesState는 메시지 기반 에이전트를 위한 내장 State로, add_messages Reducer를 사용한다.
  • 커스텀 Reducer는 (current, update) -> merged 형태의 순수 함수로 만든다.
  • State 설계는 필드 이름의 명확성, 적절한 Reducer 선택, 초기값 설정이 핵심이다.

State 관리를 이해했으므로, 다음 챕터에서는 조건부 엣지를 사용하여 분기와 루프가 있는 복잡한 그래프를 구성하는 방법을 배운다.
