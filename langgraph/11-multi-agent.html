멀티 에이전트

여러 전문 에이전트를 조합하여 복잡한 작업을 처리한다

멀티 에이전트란?

지금까지 우리는 하나의 에이전트가 모든 작업을 처리하는 구조를 다뤘다. 도구를 호출하고, 조건에 따라 분기하고, 루프를 돌며 결과를 개선하는 것 모두 하나의 에이전트 안에서 이루어졌다. 이 방식은 단순한 작업에는 잘 동작하지만, 작업이 복잡해질수록 한계가 드러난다.

멀티 에이전트(Multi-Agent) 패턴은 이 한계를 극복하기 위한 설계 방식이다. 하나의 "만능" 에이전트 대신, 각각 특정 역할에 전문화된 여러 에이전트가 팀처럼 협력하여 복잡한 작업을 수행한다. 이것은 실세계의 조직 구조와 매우 유사하다. 회사에서 하나의 프로젝트를 기획자, 개발자, 디자이너, QA 엔지니어가 나눠서 처리하는 것과 같은 원리이다.

INFO

멀티 에이전트의 핵심 이점은 관심사의 분리(Separation of Concerns)이다. 각 에이전트가 자신의 전문 분야에만 집중하므로 프롬프트가 짧고 명확해지고, 개별 에이전트의 성능이 향상되며, 독립적으로 테스트하고 개선할 수 있다.

언제 멀티 에이전트가 필요한가?

모든 상황에서 멀티 에이전트가 필요한 것은 아니다. 다음 조건 중 하나 이상에 해당하면 멀티 에이전트 패턴을 고려해야 한다:

  • 역할이 명확히 구분되는 경우: 연구, 작성, 검증 등 서로 다른 전문성이 요구되는 작업
  • 단일 프롬프트가 너무 길어지는 경우: 하나의 시스템 프롬프트에 모든 역할을 설명하면 LLM 성능이 저하된다
  • 병렬 처리가 가능한 경우: 독립적인 하위 작업들을 동시에 실행하여 전체 처리 시간을 단축할 수 있다
  • 서로 다른 모델이 필요한 경우: 비용이 중요한 작업에는 저렴한 모델, 복잡한 추론에는 고성능 모델을 사용할 수 있다
  • 독립적인 개선이 필요한 경우: 한 에이전트의 프롬프트나 도구를 변경해도 다른 에이전트에 영향을 주지 않아야 한다

멀티 에이전트 아키텍처 종류

LangGraph에서 멀티 에이전트를 구현하는 주요 아키텍처 패턴은 다음과 같다:

패턴 설명 적합한 상황
Supervisor 중앙 관리자가 작업을 분배하고 결과를 종합 역할이 명확하고 순서가 유동적인 작업
서브그래프 각 에이전트가 독립적인 그래프로 존재하며 상위 그래프에서 호출 에이전트 내부 로직이 복잡한 경우
병렬 실행 (Send) 여러 에이전트를 동시에 실행하여 결과를 취합 독립적인 하위 작업을 동시에 처리할 때
순차 파이프라인 에이전트 A → 에이전트 B → 에이전트 C 순서로 처리 각 단계의 출력이 다음 단계의 입력인 경우

Supervisor 패턴

Supervisor 패턴은 멀티 에이전트 시스템에서 가장 널리 사용되는 아키텍처이다. 중앙의 Supervisor(관리자) 에이전트가 팀 리더 역할을 하며, 사용자의 요청을 분석하고, 적절한 전문 에이전트에게 작업을 위임하고, 그 결과를 종합하여 최종 응답을 생성한다. 

graph TD
    User["사용자"] -->|"요청"| Sup["Supervisor\n(팀 리더)"]
    Sup -->|"작업 위임"| Res["연구 에이전트"]
    Sup -->|"작업 위임"| Wri["작성 에이전트"]
    Sup -->|"작업 위임"| Rev["검증 에이전트"]
    Res -->|"결과 보고"| Sup
    Wri -->|"결과 보고"| Sup
    Rev -->|"결과 보고"| Sup
    Sup -->|"FINISH"| E(("END"))

Supervisor 패턴 직접 구현

Supervisor 패턴을 처음부터 직접 구현해 본다. 핵심은 Supervisor 노드가 현재 대화 상태를 분석하여 다음에 어떤 에이전트를 호출할지 결정하는 라우팅 로직이다. 

from langgraph.graph import StateGraph, MessagesState, START, END
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import HumanMessage, SystemMessage

model = ChatAnthropic(model="claude-sonnet-4-20250514")


# ── 전문 에이전트 노드들 정의 ─────────────────────────────────

def researcher(state: MessagesState) -> dict:
    """연구원: 주어진 주제에 대한 정보를 수집하고 정리한다"""
    response = model.invoke([
        SystemMessage(content="""당신은 전문 연구원이다.
주어진 주제에 대해 핵심 사실, 통계, 최신 동향을 조사하여 정리한다.
조사 결과를 구조화된 형태로 제공한다."""),
        *state["messages"]
    ])
    return {"messages": [response]}


def writer(state: MessagesState) -> dict:
    """작성자: 연구 결과를 바탕으로 글을 작성한다"""
    response = model.invoke([
        SystemMessage(content="""당신은 전문 작성자이다.
주어진 연구 자료를 바탕으로 명확하고 읽기 쉬운 글을 작성한다.
적절한 제목, 소제목, 문단 구조를 사용한다."""),
        *state["messages"]
    ])
    return {"messages": [response]}


def reviewer(state: MessagesState) -> dict:
    """검증자: 작성된 글의 정확성과 품질을 검토한다"""
    response = model.invoke([
        SystemMessage(content="""당신은 편집자이다. 글의 정확성, 논리적 흐름, 문법을 검토한다.
문제가 있으면 구체적인 수정 사항을 제시한다.
품질이 충분하면 'APPROVED'라고 명시한다."""),
        *state["messages"]
    ])
    return {"messages": [response]}


# ── Supervisor 노드 ──────────────────────────────────────────

def supervisor(state: MessagesState) -> dict:
    """Supervisor: 현재 상태를 분석하여 다음 단계를 결정한다"""
    response = model.invoke([
        SystemMessage(content="""당신은 콘텐츠 제작팀의 리더이다.
현재까지의 대화를 분석하여 다음에 누가 작업해야 할지 결정한다.

팀원:
- researcher: 주제에 대한 조사가 필요할 때
- writer: 조사 결과를 바탕으로 글을 작성할 때
- reviewer: 작성된 글을 검토할 때
- FINISH: 최종 결과물이 완성되었을 때

반드시 위 선택지 중 하나만 답변하라. 다른 설명은 불필요하다."""),
        *state["messages"]
    ])
    return {"messages": [response]}

이제 Supervisor의 결정에 따라 다음 노드를 선택하는 라우팅 함수와 전체 그래프를 조립한다. 

# ── 라우팅 함수 ──────────────────────────────────────────────

def route_supervisor(state: MessagesState) -> str:
    """Supervisor의 마지막 응답을 파싱하여 다음 노드를 결정한다"""
    last_message = state["messages"][-1].content.strip().lower()

    if "researcher" in last_message:
        return "researcher"
    elif "writer" in last_message:
        return "writer"
    elif "reviewer" in last_message:
        return "reviewer"
    elif "finish" in last_message:
        return END
    else:
        # 기본값: Supervisor에게 다시 판단을 요청
        return "supervisor"


# ── 그래프 조립 ──────────────────────────────────────────────

graph = StateGraph(MessagesState)

# 노드 추가
graph.add_node("supervisor", supervisor)
graph.add_node("researcher", researcher)
graph.add_node("writer", writer)
graph.add_node("reviewer", reviewer)

# 엣지 설정
graph.add_edge(START, "supervisor")  # 시작 → Supervisor

# Supervisor의 결정에 따라 조건부 라우팅
graph.add_conditional_edges(
    "supervisor",
    route_supervisor,
    {
        "researcher": "researcher",
        "writer": "writer",
        "reviewer": "reviewer",
        "supervisor": "supervisor",
        END: END,
    }
)

# 각 에이전트가 작업을 마치면 다시 Supervisor에게 보고
graph.add_edge("researcher", "supervisor")
graph.add_edge("writer", "supervisor")
graph.add_edge("reviewer", "supervisor")

# 컴파일
app = graph.compile()


# ── 실행 ─────────────────────────────────────────────────────

result = app.invoke({
    "messages": [
        HumanMessage(content="인공지능의 최신 동향에 대한 블로그 글을 작성해줘")
    ]
})

# 최종 메시지 출력
for msg in result["messages"]:
    print(f"[{msg.type}] {msg.content[:100]}...")
TIP

Supervisor의 라우팅 정확도를 높이려면 구조화된 출력(Structured Output)을 사용하는 것이 좋다. 문자열 파싱 대신 Pydantic 모델을 사용하면 라우팅 오류를 줄일 수 있다.

구조화된 출력으로 라우팅 개선

위 예제에서 Supervisor의 응답을 문자열로 파싱하는 방식은 불안정할 수 있다. with_structured_output을 사용하면 LLM이 정해진 형식으로만 응답하도록 강제할 수 있다. 

from pydantic import BaseModel
from typing import Literal


class SupervisorDecision(BaseModel):
    """Supervisor의 라우팅 결정"""
    next_agent: Literal["researcher", "writer", "reviewer", "FINISH"]
    reason: str  # 결정 이유 (디버깅에 유용)


def supervisor_structured(state: MessagesState) -> dict:
    """구조화된 출력을 사용하는 Supervisor"""
    structured_model = model.with_structured_output(SupervisorDecision)

    decision = structured_model.invoke([
        SystemMessage(content="""당신은 콘텐츠 제작팀의 리더이다.
현재까지의 대화를 분석하여 다음에 누가 작업해야 할지 결정한다.

팀원:
- researcher: 주제에 대한 조사가 필요할 때
- writer: 조사 결과를 바탕으로 글을 작성할 때
- reviewer: 작성된 글을 검토할 때
- FINISH: 최종 결과물이 완성되었을 때"""),
        *state["messages"]
    ])

    # 결정 이유를 메시지로 기록 (디버깅용)
    from langchain_core.messages import AIMessage
    return {
        "messages": [
            AIMessage(content=f"[Supervisor 결정] 다음: {decision.next_agent}, 이유: {decision.reason}")
        ]
    }


def route_supervisor_structured(state: MessagesState) -> str:
    """구조화된 Supervisor 응답에서 라우팅 결정을 추출한다"""
    last_message = state["messages"][-1].content

    if "researcher" in last_message:
        return "researcher"
    elif "writer" in last_message:
        return "writer"
    elif "reviewer" in last_message:
        return "reviewer"
    elif "FINISH" in last_message:
        return END
    return "supervisor"

langgraph-supervisor 라이브러리

위에서 Supervisor 패턴을 직접 구현하는 방법을 살펴보았다. 하지만 매번 이런 보일러플레이트를 작성하는 것은 번거롭다. langgraph-supervisor 라이브러리는 Supervisor 패턴을 몇 줄의 코드로 간편하게 구현할 수 있게 해주는 사전 구축된 솔루션이다. 

# 라이브러리 설치
pip install langgraph-supervisor

기본 사용법

from langgraph_supervisor import create_supervisor
from langgraph.prebuilt import create_react_agent
from langchain_anthropic import ChatAnthropic

model = ChatAnthropic(model="claude-sonnet-4-20250514")

# 전문 에이전트 생성
research_agent = create_react_agent(
    model=model,
    tools=[search_tool, wiki_tool],
    name="researcher",
    prompt="당신은 전문 연구원이다. 주어진 주제를 조사하고 핵심 정보를 정리한다."
)

writing_agent = create_react_agent(
    model=model,
    tools=[],
    name="writer",
    prompt="당신은 전문 작성자이다. 연구 자료를 바탕으로 읽기 쉬운 글을 작성한다."
)

review_agent = create_react_agent(
    model=model,
    tools=[],
    name="reviewer",
    prompt="당신은 편집자이다. 글의 정확성과 품질을 검토한다."
)

# Supervisor 생성 — 한 줄로 멀티 에이전트 시스템 완성
supervisor = create_supervisor(
    agents=[research_agent, writing_agent, review_agent],
    model=model,
    prompt=(
        "당신은 콘텐츠 제작팀의 리더이다. "
        "사용자의 요청을 분석하고 적절한 팀원에게 작업을 위임한다."
    ),
)

# 컴파일 및 실행
app = supervisor.compile()
result = app.invoke({
    "messages": [
        {"role": "user", "content": "양자 컴퓨팅의 현재와 미래에 대한 글을 작성해줘"}
    ]
})
INFO

create_supervisor는 내부적으로 Supervisor 노드, 라우팅 로직, 에이전트 노드 간의 엣지 연결을 자동으로 처리한다. 직접 구현 대비 코드량이 크게 줄어들며, Handoff(작업 인계) 메커니즘이 내장되어 있어 에이전트 간 전환이 매끄럽다.

create_supervisor 옵션

# 다양한 옵션을 사용하여 Supervisor를 커스터마이즈할 수 있다
supervisor = create_supervisor(
    agents=[research_agent, writing_agent, review_agent],
    model=model,
    prompt="팀 리더 프롬프트",
    # 출력 형식: 마지막 메시지만 또는 전체 메시지 히스토리
    output_mode="last_message",  # "full_history" | "last_message"
)

서브그래프 패턴

서브그래프(Subgraph) 패턴은 각 에이전트를 독립적인 그래프로 구현하고, 상위 그래프에서 이 서브그래프를 노드처럼 호출하는 방식이다. 이 패턴은 각 에이전트 내부에 복잡한 로직이 있을 때 특히 유용하다. 

graph LR
    S(("START")) --> RSG
    subgraph RSG["연구 서브그래프"]
        direction LR
        R1["검색"] --> R2["요약"] --> R3["정리"]
    end
    RSG --> WSG
    subgraph WSG["작성 서브그래프"]
        direction LR
        W1["초안"] --> W2["편집"] --> W3["교정"]
    end
    WSG --> E(("END"))

서브그래프 구현

먼저 각 에이전트를 독립적인 서브그래프로 만든다. 각 서브그래프는 자체 State를 가지며, 내부적으로 여러 노드와 엣지를 포함할 수 있다. 

from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langchain_anthropic import ChatAnthropic

model = ChatAnthropic(model="claude-sonnet-4-20250514")


# ── 연구 에이전트 서브그래프 ──────────────────────────────────

class ResearchState(TypedDict):
    """연구 에이전트의 내부 상태"""
    topic: str
    raw_data: str
    summary: str


def search_node(state: ResearchState) -> dict:
    """외부 소스에서 정보를 검색한다"""
    response = model.invoke(
        f"'{state['topic']}'에 대해 핵심 사실과 최신 동향을 5가지 이상 나열하라."
    )
    return {"raw_data": response.content}


def summarize_node(state: ResearchState) -> dict:
    """검색 결과를 요약한다"""
    response = model.invoke(
        f"다음 연구 자료를 핵심 포인트로 요약하라:\n\n{state['raw_data']}"
    )
    return {"summary": response.content}


# 연구 서브그래프 조립
research_graph = StateGraph(ResearchState)
research_graph.add_node("search", search_node)
research_graph.add_node("summarize", summarize_node)
research_graph.add_edge(START, "search")
research_graph.add_edge("search", "summarize")
research_graph.add_edge("summarize", END)

# 서브그래프 컴파일
research_agent = research_graph.compile()


# ── 작성 에이전트 서브그래프 ──────────────────────────────────

class WritingState(TypedDict):
    """작성 에이전트의 내부 상태"""
    research_summary: str
    draft: str
    final_article: str


def draft_node(state: WritingState) -> dict:
    """초안을 작성한다"""
    response = model.invoke(
        f"다음 연구 요약을 바탕으로 블로그 글 초안을 작성하라:\n\n{state['research_summary']}"
    )
    return {"draft": response.content}


def edit_node(state: WritingState) -> dict:
    """초안을 편집하고 완성한다"""
    response = model.invoke(
        f"다음 초안을 편집하여 완성도를 높여라. 문법, 논리, 가독성을 개선하라:\n\n{state['draft']}"
    )
    return {"final_article": response.content}


# 작성 서브그래프 조립
writing_graph = StateGraph(WritingState)
writing_graph.add_node("draft", draft_node)
writing_graph.add_node("edit", edit_node)
writing_graph.add_edge(START, "draft")
writing_graph.add_edge("draft", "edit")
writing_graph.add_edge("edit", END)

# 서브그래프 컴파일
writing_agent = writing_graph.compile()

메인 그래프에서 서브그래프 사용

컴파일된 서브그래프를 메인 그래프의 노드로 등록하여 사용한다. 서브그래프 간 상태 전달을 위해 메인 그래프의 State에서 서브그래프 State로 변환하는 래퍼 함수가 필요하다. 

# ── 메인 그래프 ──────────────────────────────────────────────

class MainState(TypedDict):
    """메인 그래프의 상태"""
    topic: str
    research_result: str
    final_output: str


def run_research(state: MainState) -> dict:
    """연구 서브그래프를 실행한다"""
    result = research_agent.invoke({
        "topic": state["topic"],
        "raw_data": "",
        "summary": "",
    })
    return {"research_result": result["summary"]}


def run_writing(state: MainState) -> dict:
    """작성 서브그래프를 실행한다"""
    result = writing_agent.invoke({
        "research_summary": state["research_result"],
        "draft": "",
        "final_article": "",
    })
    return {"final_output": result["final_article"]}


# 메인 그래프 조립
main_graph = StateGraph(MainState)
main_graph.add_node("research", run_research)
main_graph.add_node("write", run_writing)
main_graph.add_edge(START, "research")
main_graph.add_edge("research", "write")
main_graph.add_edge("write", END)

# 메인 그래프 컴파일 및 실행
app = main_graph.compile()

result = app.invoke({
    "topic": "2025년 AI 에이전트 트렌드",
    "research_result": "",
    "final_output": "",
})

print(result["final_output"])
TIP

서브그래프 패턴의 장점은 캡슐화에 있다. 연구 에이전트 내부에 검색 → 필터링 → 요약 → 팩트 체크 등 아무리 복잡한 로직이 있어도, 메인 그래프에서는 단순히 하나의 노드로만 보인다. 각 서브그래프를 독립적으로 개발하고 테스트할 수 있으므로 유지보수가 용이하다.

에이전트 간 상태 전달

멀티 에이전트 시스템에서 가장 까다로운 부분 중 하나는 에이전트 간에 데이터를 어떻게 주고받을 것인가이다. LangGraph에서는 여러 가지 방법으로 상태를 전달할 수 있다.

방법 1: 공유 State 키 사용

가장 간단한 방법은 메인 그래프의 State에 공유 필드를 두고, 각 에이전트가 이 필드를 읽고 쓰는 것이다. 

from typing import Annotated
from operator import add

class SharedState(TypedDict):
    """에이전트 간 공유되는 상태"""
    query: str
    # Annotated + add를 사용하면 리스트에 값이 누적된다
    findings: Annotated[list[str], add]
    final_report: str


def agent_a(state: SharedState) -> dict:
    """에이전트 A가 발견한 정보를 findings에 추가"""
    return {"findings": ["에이전트 A의 발견 사항"]}


def agent_b(state: SharedState) -> dict:
    """에이전트 B도 findings에 추가 — 기존 값과 합쳐진다"""
    return {"findings": ["에이전트 B의 발견 사항"]}


def synthesizer(state: SharedState) -> dict:
    """모든 에이전트의 findings를 종합하여 보고서 작성"""
    all_findings = "\n".join(state["findings"])
    return {"final_report": f"종합 보고서:\n{all_findings}"}

방법 2: MessagesState로 메시지 전달

MessagesState를 사용하면 모든 에이전트가 메시지 리스트를 통해 자연스럽게 소통할 수 있다. 이전 에이전트의 출력이 다음 에이전트의 컨텍스트로 자동으로 포함된다. 

from langgraph.graph import MessagesState

# MessagesState를 사용하면 messages 키로 자동으로 메시지가 누적된다
def agent_alpha(state: MessagesState) -> dict:
    """에이전트 Alpha의 응답이 메시지 리스트에 추가된다"""
    response = model.invoke(state["messages"])
    return {"messages": [response]}


def agent_beta(state: MessagesState) -> dict:
    """에이전트 Beta는 Alpha의 메시지를 포함한 전체 히스토리를 본다"""
    response = model.invoke([
        SystemMessage(content="이전 에이전트의 분석을 참고하여 보완하라."),
        *state["messages"]
    ])
    return {"messages": [response]}

방법 3: 서브그래프 State 변환

서브그래프가 메인 그래프와 다른 State 구조를 가질 때, 래퍼 함수에서 명시적으로 변환해야 한다. 앞선 서브그래프 예제에서 run_research와 run_writing 함수가 바로 이 변환 역할을 수행했다.

WARNING

서브그래프 간 State 변환 시 데이터가 누락되지 않도록 주의해야 한다. 특히 리스트 타입의 State 필드는 Annotated 리듀서 설정에 따라 덮어쓰기(replace) 또는 추가(append)가 달라지므로 의도한 동작을 확인해야 한다.

Send로 병렬 실행

Send는 LangGraph에서 제공하는 강력한 기능으로, 하나의 노드에서 여러 노드 인스턴스를 동시에 실행할 수 있게 해준다. 이것은 Map-Reduce 패턴과 유사하다. 여러 하위 작업을 병렬로 처리한 후 결과를 하나로 합치는 구조이다. 

graph TD
    dispatch["dispatch_agents"] -->|"Send()"| R1["R1"]
    dispatch -->|"Send()"| R2["R2"]
    dispatch -->|"Send()"| R3["R3"]
    R1 --> agg["aggregate\n(결과 취합)"]
    R2 --> agg
    R3 --> agg

from typing import Annotated
from operator import add
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send
from langchain_anthropic import ChatAnthropic

model = ChatAnthropic(model="claude-sonnet-4-20250514")


# ── State 정의 ───────────────────────────────────────────────

class ResearchRequest(TypedDict):
    """개별 연구 요청 (Send로 전달되는 상태)"""
    topic: str


class OverallState(TypedDict):
    """전체 워크플로우 상태"""
    topics: list[str]
    # Annotated + add를 사용하여 병렬 결과를 자동으로 누적
    results: Annotated[list[str], add]
    final_summary: str


# ── 노드 함수 ────────────────────────────────────────────────

def research_topic(state: ResearchRequest) -> dict:
    """개별 주제를 연구한다 — 병렬로 여러 인스턴스가 실행된다"""
    response = model.invoke(
        f"'{state['topic']}'에 대해 3문장으로 핵심 요약을 작성하라."
    )
    return {"results": [f"[{state['topic']}] {response.content}"]}


def aggregate_results(state: OverallState) -> dict:
    """모든 연구 결과를 종합한다"""
    combined = "\n\n".join(state["results"])
    response = model.invoke(
        f"다음 연구 결과들을 종합하여 하나의 보고서로 작성하라:\n\n{combined}"
    )
    return {"final_summary": response.content}


# ── 디스패치 함수 (Send 사용) ─────────────────────────────────

def dispatch_researchers(state: OverallState) -> list[Send]:
    """각 주제에 대해 병렬로 연구 에이전트를 실행한다"""
    return [
        Send("research", {"topic": topic})
        for topic in state["topics"]
    ]


# ── 그래프 조립 ──────────────────────────────────────────────

graph = StateGraph(OverallState)
graph.add_node("research", research_topic)
graph.add_node("aggregate", aggregate_results)

# START에서 조건부 엣지로 Send를 사용하여 병렬 디스패치
graph.add_conditional_edges(START, dispatch_researchers, ["research"])
graph.add_edge("research", "aggregate")
graph.add_edge("aggregate", END)

app = graph.compile()

# 3개 주제를 병렬로 연구
result = app.invoke({
    "topics": ["양자 컴퓨팅", "핵융합 에너지", "우주 탐사"],
    "results": [],
    "final_summary": "",
})

print(result["final_summary"])
INFO

Send는 동적으로 노드를 생성하여 실행한다. 위 예제에서 topics 리스트에 3개 항목이 있으면 3개의 research 노드가, 10개 항목이 있으면 10개의 research 노드가 병렬로 실행된다. 각 Send로 실행된 노드의 결과는 Annotated[list, add] 리듀서를 통해 자동으로 합쳐진다.

실전 예제: 콘텐츠 제작 팀

지금까지 배운 개념들을 모두 합쳐서, Supervisor가 연구원, 작성자, 편집자 에이전트를 조율하여 블로그 포스트를 작성하는 완성된 멀티 에이전트 시스템을 만들어 본다. 이 시스템은 다음 흐름으로 동작한다:

  1. 사용자가 주제를 입력한다
  2. Supervisor가 연구원에게 조사를 지시한다
  3. 연구 결과를 작성자에게 전달하여 글을 작성한다
  4. 작성된 글을 편집자가 검토한다
  5. 품질이 미달이면 작성자에게 재작성을 지시한다
  6. 품질이 충분하면 최종 결과물을 반환한다
graph TD
    User["사용자 요청"] --> Sup["Supervisor"]
    Sup --> Res["Researcher\n(연구원)"]
    Sup --> Wri["Writer\n(작성자)"]
    Sup --> Rev["Reviewer\n(편집자)"]
    Res --> Sup
    Wri --> Sup
    Rev --> QC{"품질 판단\nPASS / FAIL"}
    QC -->|"FAIL"| Sup
    QC -->|"PASS"| E(("END"))

"""
콘텐츠 제작 멀티 에이전트 시스템
Supervisor가 연구원, 작성자, 편집자를 조율하여 블로그 포스트를 생성한다.
"""
from typing import Annotated, Literal
from typing_extensions import TypedDict
from operator import add

from pydantic import BaseModel
from langgraph.graph import StateGraph, START, END
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import HumanMessage, SystemMessage, AIMessage

model = ChatAnthropic(model="claude-sonnet-4-20250514")


# ── 상태 정의 ────────────────────────────────────────────────

class ContentState(TypedDict):
    """콘텐츠 제작 파이프라인의 전체 상태"""
    topic: str                                # 사용자가 입력한 주제
    research_data: str                        # 연구원이 수집한 자료
    draft: str                                # 작성자가 쓴 초안
    review_feedback: str                      # 편집자의 피드백
    final_article: str                        # 최종 완성 글
    revision_count: int                       # 수정 횟수 (무한 루프 방지)
    current_agent: str                        # 현재 활성 에이전트


# ── Supervisor의 라우팅 결정 스키마 ───────────────────────────

class RoutingDecision(BaseModel):
    next: Literal["researcher", "writer", "reviewer", "FINISH"]
    instruction: str  # 다음 에이전트에게 전달할 지시사항


# ── 에이전트 노드들 ──────────────────────────────────────────

def supervisor_node(state: ContentState) -> dict:
    """Supervisor: 현재 상태를 분석하여 다음 에이전트와 지시사항을 결정"""
    structured_model = model.with_structured_output(RoutingDecision)

    context_parts = [f"주제: {state['topic']}"]
    if state.get("research_data"):
        context_parts.append(f"연구 자료: 수집 완료 ({len(state['research_data'])}자)")
    if state.get("draft"):
        context_parts.append(f"초안: 작성 완료 ({len(state['draft'])}자)")
    if state.get("review_feedback"):
        context_parts.append(f"편집 피드백: {state['review_feedback'][:200]}")
    context_parts.append(f"수정 횟수: {state.get('revision_count', 0)}")

    context = "\n".join(context_parts)

    decision = structured_model.invoke([
        SystemMessage(content=f"""당신은 콘텐츠 제작팀의 리더이다.
현재 프로젝트 상태를 보고 다음 단계를 결정하라.

현재 상태:
{context}

규칙:
1. 연구 자료가 없으면 researcher를 선택
2. 연구 자료는 있지만 초안이 없으면 writer를 선택
3. 초안이 있지만 검토되지 않았으면 reviewer를 선택
4. 피드백이 있고 수정이 필요하면 writer를 선택 (재작성)
5. 검토를 통과했거나 수정 횟수가 3회 이상이면 FINISH를 선택"""),
        HumanMessage(content="다음 단계를 결정하라.")
    ])

    return {
        "current_agent": decision.next,
    }


def researcher_node(state: ContentState) -> dict:
    """연구원: 주제에 대한 심층 조사를 수행한다"""
    response = model.invoke([
        SystemMessage(content="""당신은 전문 연구원이다.
주어진 주제에 대해 다음을 포함하는 상세한 연구 보고서를 작성하라:
1. 주제의 배경과 정의
2. 현재 상태와 최신 동향
3. 핵심 통계와 데이터
4. 주요 인물/기업/기관
5. 미래 전망과 시사점

가능한 한 구체적이고 정확한 정보를 제공하라."""),
        HumanMessage(content=f"다음 주제를 조사하라: {state['topic']}")
    ])
    return {"research_data": response.content}


def writer_node(state: ContentState) -> dict:
    """작성자: 연구 자료를 바탕으로 블로그 글을 작성한다"""
    prompt_parts = [
        f"연구 자료:\n{state['research_data']}",
    ]

    # 수정 요청인 경우 이전 초안과 피드백을 포함
    if state.get("review_feedback") and state.get("draft"):
        prompt_parts.append(f"\n이전 초안:\n{state['draft']}")
        prompt_parts.append(f"\n편집자 피드백:\n{state['review_feedback']}")
        prompt_parts.append("\n위 피드백을 반영하여 글을 수정하라.")
    else:
        prompt_parts.append("\n위 자료를 바탕으로 블로그 글을 새로 작성하라.")

    response = model.invoke([
        SystemMessage(content="""당신은 전문 블로그 작성자이다.
다음 원칙에 따라 글을 작성하라:
- 매력적인 제목과 소제목 사용
- 명확하고 읽기 쉬운 문체
- 적절한 문단 나누기
- 핵심 포인트 강조
- 독자의 관심을 끄는 도입부와 인사이트 있는 결론"""),
        HumanMessage(content="\n".join(prompt_parts))
    ])

    return {
        "draft": response.content,
        "revision_count": state.get("revision_count", 0) + (1 if state.get("review_feedback") else 0),
    }


def reviewer_node(state: ContentState) -> dict:
    """편집자: 작성된 글을 검토하고 피드백을 제공한다"""
    response = model.invoke([
        SystemMessage(content="""당신은 전문 편집자이다.
다음 항목을 기준으로 글을 검토하라:

1. 정확성: 사실 관계가 올바른가?
2. 논리성: 글의 흐름이 자연스러운가?
3. 가독성: 읽기 쉽고 이해하기 쉬운가?
4. 완성도: 도입/본문/결론이 잘 구성되어 있는가?
5. 문법: 맞춤법이나 문법 오류가 없는가?

검토 결과를 'APPROVED' 또는 'REVISION_NEEDED'로 시작하고,
구체적인 피드백을 제공하라."""),
        HumanMessage(content=f"다음 블로그 글을 검토하라:\n\n{state['draft']}")
    ])

    return {"review_feedback": response.content}


# ── 라우팅 함수 ──────────────────────────────────────────────

def route_from_supervisor(state: ContentState) -> str:
    """Supervisor의 결정에 따라 다음 노드를 선택"""
    agent = state.get("current_agent", "researcher")
    if agent == "FINISH":
        return "finalize"
    return agent


def route_from_reviewer(state: ContentState) -> str:
    """편집자의 피드백에 따라 다음 경로를 선택"""
    feedback = state.get("review_feedback", "")
    revision_count = state.get("revision_count", 0)

    # 수정 횟수가 3회 이상이면 강제 완료
    if revision_count >= 3:
        return "finalize"

    if feedback.startswith("APPROVED"):
        return "finalize"
    else:
        return "supervisor"


# ── 최종 결과 노드 ───────────────────────────────────────────

def finalize_node(state: ContentState) -> dict:
    """최종 결과물을 확정한다"""
    return {"final_article": state.get("draft", "")}


# ── 그래프 조립 ──────────────────────────────────────────────

graph = StateGraph(ContentState)

# 노드 등록
graph.add_node("supervisor", supervisor_node)
graph.add_node("researcher", researcher_node)
graph.add_node("writer", writer_node)
graph.add_node("reviewer", reviewer_node)
graph.add_node("finalize", finalize_node)

# 엣지 연결
graph.add_edge(START, "supervisor")

graph.add_conditional_edges(
    "supervisor",
    route_from_supervisor,
    {
        "researcher": "researcher",
        "writer": "writer",
        "reviewer": "reviewer",
        "finalize": "finalize",
    }
)

# 각 에이전트 작업 완료 후의 경로
graph.add_edge("researcher", "supervisor")   # 연구 완료 → Supervisor에게 보고
graph.add_edge("writer", "supervisor")       # 작성 완료 → Supervisor에게 보고

# 편집자는 결과에 따라 분기
graph.add_conditional_edges(
    "reviewer",
    route_from_reviewer,
    {
        "supervisor": "supervisor",   # 수정 필요 → Supervisor에게 재지시
        "finalize": "finalize",       # 승인 → 완료
    }
)

graph.add_edge("finalize", END)

# 컴파일
app = graph.compile()


# ── 실행 ─────────────────────────────────────────────────────

result = app.invoke({
    "topic": "2025년 AI 에이전트 기술의 현황과 전망",
    "research_data": "",
    "draft": "",
    "review_feedback": "",
    "final_article": "",
    "revision_count": 0,
    "current_agent": "",
})

print("=" * 60)
print("최종 블로그 포스트")
print("=" * 60)
print(result["final_article"])
print(f"\n총 수정 횟수: {result['revision_count']}")
TIP

실전에서는 revision_count와 같은 안전장치를 반드시 포함해야 한다. LLM의 판단에 따라 에이전트가 무한히 수정을 반복할 수 있기 때문이다. 최대 반복 횟수를 설정하거나, 그래프 컴파일 시 recursion_limit을 지정하여 무한 루프를 방지한다.

정리

이번 챕터에서 배운 내용을 정리한다:

  • 멀티 에이전트는 전문화된 여러 에이전트가 협력하여 복잡한 작업을 처리하는 패턴이다.
  • Supervisor 패턴은 중앙 관리자가 작업을 분배하고 결과를 종합하는 가장 일반적인 구조이다.
  • langgraph-supervisor 라이브러리를 사용하면 Supervisor 패턴을 빠르게 구현할 수 있다.
  • 서브그래프 패턴은 각 에이전트를 독립적인 그래프로 캡슐화하여 유지보수를 용이하게 한다.
  • Send를 사용하면 여러 에이전트를 병렬로 실행하여 처리 시간을 단축할 수 있다.
  • 에이전트 간 상태 전달은 공유 State, MessagesState, 또는 래퍼 함수를 통해 구현한다.
  • 무한 루프 방지를 위해 최대 반복 횟수 등의 안전장치를 반드시 포함해야 한다.

다음 챕터에서는 MCP(Model Context Protocol) 서버를 LangGraph 에이전트에 연동하여 표준화된 도구 생태계를 활용하는 방법을 배운다.
