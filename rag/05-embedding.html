임베딩 모델

텍스트를 벡터로 변환하여 의미적 검색의 기반을 만든다

임베딩이란?

임베딩(Embedding)은 텍스트를 고차원 벡터(숫자 배열)로 변환하는 과정이다. 사람이 읽는 자연어를 컴퓨터가 수학적으로 처리할 수 있는 형태로 바꾸는 것이다. 핵심 원리는 단순하다: 의미적으로 유사한 텍스트는 벡터 공간에서 가까이 위치한다.

지도에서 서울과 부산의 좌표가 물리적 위치를 나타내는 것처럼, 임베딩은 텍스트의 의미적 좌표를 생성한다. "고양이"와 "강아지"의 임베딩은 가까이 위치하고, "고양이"와 "미적분"의 임베딩은 멀리 떨어진다. 

graph LR
    A["텍스트\n'서울의 인구는?'"] --> B["임베딩 모델\n(Transformer)"]
    B --> C["벡터\n[0.023, -0.112, 0.891, ..., 0.034]\n(1536차원)"]
    style A fill:#e8f4f8,stroke:#333
    style B fill:#fff3cd,stroke:#333
    style C fill:#d4edda,stroke:#333

임베딩의 강력함은 벡터 연산으로 의미 관계를 포착할 수 있다는 데 있다. 가장 유명한 예시가 Word2Vec에서 발견된 관계이다:

TIP

벡터("왕") - 벡터("남자") + 벡터("여자") ≈ 벡터("여왕")

이는 임베딩이 단순한 키워드 매칭이 아니라 의미적 관계를 수학적으로 인코딩한다는 것을 보여준다. RAG에서 이 특성이 왜 중요한가? 사용자가 "한국의 수도"라고 질문했을 때, 문서에 "서울"이라는 단어만 있어도 관련 문서를 찾아낼 수 있기 때문이다.

임베딩 모델의 작동 원리

현대의 임베딩 모델은 거의 대부분 Transformer 기반 인코더(BERT 계열)로 구축된다. 입력 텍스트를 토큰으로 분리한 뒤, 여러 층의 Self-Attention을 거쳐 각 토큰의 문맥적 의미를 파악한다.

문장 임베딩 vs 토큰 임베딩

Transformer 인코더는 기본적으로 각 토큰에 대한 벡터를 출력한다. 하지만 RAG에서는 문장이나 문단 전체를 하나의 벡터로 표현해야 한다. 이때 Pooling 전략이 사용된다.

Pooling 전략 방법 특징
CLS 토큰 [CLS] 토큰의 출력 벡터를 문장 임베딩으로 사용 BERT 원본 방식. 단순하지만 정보 손실 가능
Mean Pooling 모든 토큰의 출력 벡터를 평균 가장 일반적. 문장 전체의 의미를 고르게 반영
Max Pooling 각 차원에서 최댓값 선택 핵심 특징을 강조하지만 노이즈에 민감

차원(Dimension)의 의미

임베딩 벡터의 차원은 모델이 텍스트의 의미를 표현하는 데 사용하는 숫자의 개수이다. 768차원이면 768개의 숫자로 텍스트의 의미를 표현하는 것이다.

차원 대표 모델 벡터당 용량 특성
384d all-MiniLM-L6-v2 1.5 KB 경량. 빠른 프로토타이핑에 적합
768d BERT-base, BGE-base 3 KB 범용적. 비용 대비 성능 균형
1024d BGE-large, E5-large 4 KB 고성능. 복잡한 의미 포착
1536d text-embedding-3-small 6 KB OpenAI 기본. API 기반
3072d text-embedding-3-large 12 KB 최고 정밀도. 비용 높음
WARNING

차원이 높을수록 의미 표현력이 증가하지만, 저장 비용과 검색 속도에 직접적 영향을 준다. 100만 개 문서를 3072d로 저장하면 약 11.4 GB의 벡터 저장소가 필요하다. 반드시 비용과 성능을 함께 고려해야 한다.

임베딩 모델 선택 기준

임베딩 모델 선택은 RAG 시스템의 검색 품질을 결정짓는 가장 중요한 의사결정 중 하나이다. 아래는 개발자가 실제로 따져야 할 핵심 기준이다.

MTEB 벤치마크 읽는 법

MTEB(Massive Text Embedding Benchmark)는 임베딩 모델을 평가하는 표준 벤치마크이다. 8가지 카테고리(Classification, Clustering, PairClassification, Reranking, Retrieval, STS, Summarization, BitextMining)로 모델 성능을 측정한다.

TIP

RAG 개발자에게 가장 중요한 카테고리는 Retrieval이다. 이 카테고리의 nDCG@10 점수가 높은 모델이 RAG에서 좋은 검색 결과를 제공할 가능성이 높다. 한국어 RAG를 구축한다면 Retrieval (kor) 또는 다국어(multilingual) 벤치마크 결과를 확인해야 한다. MTEB Leaderboard에서 최신 순위를 확인할 수 있다.

비용 vs 성능 트레이드오프

API 기반 모델(OpenAI, Cohere)은 구축이 간편하지만 사용량에 비례하여 비용이 발생한다. 오픈소스 모델(BGE, E5)은 초기 설정이 복잡하지만, 자체 호스팅하면 대량 처리 시 비용을 크게 절감할 수 있다.

모델 호스팅 방식

방식 장점 단점 적합한 경우
API 기반 즉시 사용 가능, 인프라 관리 불필요 비용 발생, 데이터 외부 전송, 레이턴시 프로토타입, 소규모 서비스
자체 호스팅 비용 절감, 데이터 통제, 커스터마이징 GPU 인프라 필요, 운영 부담 대규모 서비스, 민감 데이터

차원 수와 저장소 비용 계산

벡터 저장소 용량은 다음 공식으로 추정할 수 있다:

PYTHON 
# 저장소 용량 계산
dimensions = 1536       # 벡터 차원
num_documents = 100_000 # 문서(청크) 수
bytes_per_float = 4     # float32 = 4 bytes

total_bytes = dimensions * num_documents * bytes_per_float
total_gb = total_bytes / (1024 ** 3)
print(f"필요 저장소: {total_gb:.2f} GB")  # 약 0.57 GB
기준 핵심 질문 영향
검색 품질 MTEB Retrieval 점수가 충분한가? RAG 답변 정확도
한국어 지원 한국어 벤치마크 결과가 있는가? 한국어 검색 정밀도
비용 토큰당 가격 × 예상 사용량은? 월간 운영 비용
레이턴시 임베딩 생성 시간이 허용 범위인가? 사용자 응답 속도
최대 토큰 청크 크기에 맞는 컨텍스트 윈도우인가? 청크 전략과 연동
호스팅 API 호출 vs 자체 서버 운영? 인프라 아키텍처

주요 임베딩 모델 비교

2024~2025년 기준으로 RAG에서 가장 많이 사용되는 임베딩 모델을 비교한다. 각 모델의 특성을 이해하면 프로젝트에 맞는 최적의 선택을 할 수 있다.

OpenAI text-embedding-3 시리즈

OpenAI의 3세대 임베딩 모델이다. 가장 큰 특징은 Matryoshka Representation Learning을 지원하여 차원을 유연하게 축소할 수 있다는 점이다. 예를 들어 3072차원 모델을 1024차원이나 256차원으로 줄여 사용할 수 있다. 차원 축소 시에도 성능 저하가 최소화되도록 학습되어 있어, 저장 비용을 절감하면서도 검색 품질을 유지할 수 있다.

OpenAI text-embedding-ada-002

이전 세대 모델이지만 여전히 많은 프로덕션 시스템에서 사용 중이다. 1536차원 고정이며, 토큰당 비용이 3세대보다 높다. 신규 프로젝트에서는 text-embedding-3-small로 마이그레이션하는 것이 권장된다.

BGE (BAAI General Embedding)

베이징 인공지능 아카데미(BAAI)가 개발한 오픈소스 임베딩 모델이다. MTEB 벤치마크에서 최상위권 성능을 기록했으며, 한국어를 포함한 다국어를 지원한다. Apache 2.0 라이선스로 자유롭게 상업적 사용이 가능하다. 자체 호스팅이 가능하여 데이터 보안이 중요한 환경에 적합하다.

E5 (Microsoft)

Microsoft Research에서 개발한 임베딩 모델이다. E5-Instruct 시리즈는 task-specific instruction을 입력에 추가하여 검색, 분류 등 특정 작업에 맞춘 임베딩을 생성할 수 있다. 예를 들어 검색 작업에서는 쿼리 앞에 "query: "를, 문서에는 "passage: "를 붙여 역할을 명시한다.

Cohere embed-v3

Cohere의 3세대 임베딩 모델이다. 100개 이상의 언어를 지원하는 강력한 다국어 모델이며, Matryoshka 방식의 차원 축소를 지원한다. input_type 파라미터로 검색 문서(search_document)와 검색 쿼리(search_query)를 구분하여 비대칭 검색 성능을 높인다.

Jina Embeddings v3

Jina AI가 개발한 임베딩 모델로, 8192 토큰의 긴 컨텍스트 윈도우가 특징이다. 긴 문서를 분할하지 않고도 임베딩할 수 있어 청크 전략에 유연성을 제공한다. 다국어 지원이 뛰어나며, Late Chunking 기법을 활용하면 문서 전체의 문맥을 유지한 임베딩이 가능하다.

한국어 RAG에서는 BGE-M3와 Cohere embed-v3가 특히 좋은 성능을 보인다. BGE-M3는 Dense, Sparse, Multi-vector 세 가지 검색 방식을 동시에 지원하여 하이브리드 검색에도 유리하다. API 편의성이 중요하다면 OpenAI text-embedding-3-small이 비용 대비 우수한 선택이다.

LangChain에서 임베딩 사용

LangChain은 다양한 임베딩 모델을 통일된 인터페이스로 사용할 수 있게 해준다. embed_query()와 embed_documents() 두 가지 메서드만 알면 된다.

OpenAI 임베딩

PYTHON 
from langchain_openai import OpenAIEmbeddings

# 임베딩 모델 초기화
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 단일 텍스트 임베딩
vector = embeddings.embed_query("서울의 인구는?")
print(f"차원: {len(vector)}")   # 1536
print(f"처음 5개 값: {vector[:5]}")

차원 축소 (Matryoshka)

text-embedding-3 시리즈는 dimensions 파라미터로 벡터 차원을 줄일 수 있다. 차원을 줄이면 저장 비용이 감소하고 검색 속도가 빨라지지만, 검색 정밀도가 약간 떨어질 수 있다.

PYTHON 
from langchain_openai import OpenAIEmbeddings

# 3072차원 → 1024차원으로 축소
embeddings = OpenAIEmbeddings(
    model="text-embedding-3-large",
    dimensions=1024,
)

vector = embeddings.embed_query("서울의 인구는?")
print(f"차원: {len(vector)}")  # 1024 (원래 3072에서 축소)

문서 임베딩 vs 쿼리 임베딩

LangChain의 임베딩 인터페이스는 embed_documents()와 embed_query()를 구분한다. 일부 모델(E5, Cohere 등)은 문서와 쿼리에 서로 다른 prefix를 붙여 비대칭 검색 성능을 높인다. OpenAI 모델은 내부적으로 동일하게 처리하지만, 코드 레벨에서 구분하는 습관을 들이는 것이 좋다.

PYTHON 
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 여러 문서를 한 번에 임베딩 (배치 처리)
documents = [
    "서울특별시의 인구는 약 950만 명이다.",
    "부산광역시의 인구는 약 330만 명이다.",
    "대한민국의 수도는 서울이다.",
]
doc_vectors = embeddings.embed_documents(documents)
print(f"문서 수: {len(doc_vectors)}")       # 3
print(f"각 벡터 차원: {len(doc_vectors[0])}")  # 1536

# 쿼리 임베딩 (단일)
query_vector = embeddings.embed_query("한국에서 가장 큰 도시는?")
print(f"쿼리 벡터 차원: {len(query_vector)}")  # 1536

HuggingFace 임베딩 (자체 호스팅)

PYTHON 
from langchain_huggingface import HuggingFaceEmbeddings

# BGE-M3 모델 로드 (로컬 GPU 사용)
embeddings = HuggingFaceEmbeddings(
    model_name="BAAI/bge-m3",
    model_kwargs={"device": "cuda"},  # GPU 사용
    encode_kwargs={"normalize_embeddings": True},  # 정규화
)

vector = embeddings.embed_query("서울의 인구는?")
print(f"차원: {len(vector)}")  # 1024
TIP

normalize_embeddings=True로 설정하면 벡터가 단위 벡터(길이 1)로 정규화된다. 이렇게 하면 코사인 유사도와 내적(Dot Product)의 결과가 동일해지므로, 더 빠른 내적 연산으로 코사인 유사도를 대체할 수 있다.

유사도 측정

임베딩 벡터를 생성한 후, 두 벡터가 얼마나 유사한지를 수치로 계산하는 방법이 필요하다. 이것이 바로 거리 메트릭(Distance Metric)이다. RAG의 검색 단계에서 "쿼리와 가장 가까운 문서 K개를 찾아라"라는 것은 결국 이 거리를 기준으로 정렬하는 것이다.

코사인 유사도 (Cosine Similarity)

두 벡터의 방향이 얼마나 유사한지를 측정한다. 벡터의 크기(길이)는 무시하고 방향만 비교하므로, 문서 길이에 영향을 받지 않는다. 결과 범위는 -1(정반대) ~ 0(무관) ~ 1(동일 방향)이며, 가장 널리 사용되는 메트릭이다.

내적 (Dot Product)

두 벡터의 내적은 방향과 크기를 모두 고려한다. 벡터가 정규화(단위 벡터)되어 있으면 코사인 유사도와 동일한 결과를 낸다. 정규화된 벡터에서는 내적이 연산 비용이 낮아 벡터 DB에서 주로 사용된다.

유클리드 거리 (L2 Distance)

두 벡터 사이의 절대적 거리를 측정한다. 값이 작을수록 유사하다. 정규화되지 않은 벡터에서 절대적 크기가 의미를 가질 때 사용한다. 대부분의 RAG 시스템에서는 코사인 유사도가 더 적합하다.

PYTHON 
import numpy as np
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 세 개의 텍스트 임베딩
texts = [
    "고양이가 소파에서 낮잠을 자고 있다",
    "강아지가 침대에서 잠을 자고 있다",
    "주식 시장이 급등했다",
]
vectors = embeddings.embed_documents(texts)
query = embeddings.embed_query("반려동물이 자고 있는 모습")

# NumPy 배열로 변환
vectors = np.array(vectors)
query = np.array(query)

# 코사인 유사도 계산
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

for i, text in enumerate(texts):
    sim = cosine_similarity(query, vectors[i])
    print(f"[{sim:.4f}] {text}")

# 예상 출력:
# [0.8234] 고양이가 소파에서 낮잠을 자고 있다
# [0.8012] 강아지가 침대에서 잠을 자고 있다
# [0.1523] 주식 시장이 급등했다
메트릭 수학적 정의 범위 유사 = ? 사용 시점
코사인 유사도 cos(θ) = (A · B) / (||A|| × ||B||) -1 ~ 1 1에 가까움 가장 일반적. 문서 길이 무관
내적 A · B = Σ aibi -∞ ~ +∞ 큰 값 정규화된 벡터에서 코사인 대체
유클리드 거리 ||A - B||2 = √Σ(ai - bi)² 0 ~ +∞ 0에 가까움 절대적 거리가 의미 있을 때
INFO

대부분의 임베딩 모델은 정규화된 벡터를 출력하도록 학습된다. 이 경우 코사인 유사도, 내적, 유클리드 거리의 순위(ranking)가 모두 동일하다. 벡터 스토어 설정 시 어떤 메트릭을 선택하든 검색 결과는 같지만, 내적이 연산 비용이 가장 낮으므로 대규모 시스템에서는 내적을 선택하는 것이 유리하다.

임베딩 비용 계산

프로덕션 RAG 시스템에서 임베딩 비용은 크게 두 가지로 나뉜다: 문서를 처음 임베딩하는 1회성 비용과, 사용자 쿼리를 매번 임베딩하는 지속 비용이다.

문서 임베딩 비용 (1회성)

PYTHON 
# 10,000 페이지 PDF 문서의 임베딩 비용 추정
pages = 10_000
tokens_per_page = 500       # 평균 500 토큰/페이지 (한국어 기준)
chunks_per_page = 2         # 페이지당 평균 2개 청크
overhead = 1.1              # 청크 오버랩으로 인한 10% 추가

total_tokens = pages * tokens_per_page * chunks_per_page * overhead
print(f"총 토큰 수: {total_tokens:,.0f}")  # 11,000,000

# OpenAI text-embedding-3-small: $0.02 / 1M tokens
cost_small = (total_tokens / 1_000_000) * 0.02
print(f"3-small 비용: ${cost_small:.2f}")  # $0.22

# OpenAI text-embedding-3-large: $0.13 / 1M tokens
cost_large = (total_tokens / 1_000_000) * 0.13
print(f"3-large 비용: ${cost_large:.2f}")  # $1.43

쿼리 임베딩 비용 (지속)

PYTHON 
# 일일 쿼리 비용 추정
daily_queries = 10_000
avg_query_tokens = 30       # 평균 쿼리 길이

daily_tokens = daily_queries * avg_query_tokens
monthly_tokens = daily_tokens * 30

# text-embedding-3-small 기준
monthly_cost = (monthly_tokens / 1_000_000) * 0.02
print(f"월간 쿼리 토큰: {monthly_tokens:,.0f}")    # 9,000,000
print(f"월간 쿼리 비용: ${monthly_cost:.2f}")       # $0.18
TIP

API 기반 임베딩의 비용은 대부분 매우 저렴하다. 10,000 페이지 문서 + 일일 10,000 쿼리를 운영해도 text-embedding-3-small 기준으로 월 $0.40 수준이다. 임베딩보다는 LLM 생성 비용이 RAG 시스템 운영비의 대부분을 차지한다. 다만, 수억 건 이상의 문서를 다루는 대규모 시스템에서는 자체 호스팅이 비용 효율적이다.

다국어 임베딩과 한국어

한국어 RAG 시스템을 구축할 때는 임베딩 모델의 한국어 처리 능력을 반드시 검증해야 한다. 영어에서 높은 성능을 보이는 모델이 한국어에서는 그렇지 않을 수 있다.

한국어 토크나이징 특성

한국어는 영어와 토크나이징 방식이 크게 다르다. 영어는 공백으로 단어가 자연스럽게 분리되지만, 한국어는 교착어로서 조사, 어미가 결합되어 있다. 대부분의 임베딩 모델은 WordPiece 또는 BPE 기반 토크나이저를 사용하는데, 한국어 학습 데이터가 부족한 모델에서는 한국어 텍스트가 지나치게 많은 토큰으로 분리된다.

PYTHON 
import tiktoken

# OpenAI 모델의 토크나이저
enc = tiktoken.encoding_for_model("text-embedding-3-small")

# 같은 의미의 영어와 한국어 비교
en_text = "The population of Seoul is about 9.5 million."
ko_text = "서울특별시의 인구는 약 950만 명이다."

en_tokens = enc.encode(en_text)
ko_tokens = enc.encode(ko_text)

print(f"영어: {len(en_tokens)} 토큰")  # 약 10 토큰
print(f"한국어: {len(ko_tokens)} 토큰")  # 약 18 토큰

# 한국어는 같은 의미를 전달하는 데 더 많은 토큰을 소비한다.
# 이는 비용 증가와 컨텍스트 윈도우 압박으로 이어진다.

다국어 모델 vs 한국어 특화 모델

유형 대표 모델 장점 단점
다국어 모델 BGE-M3, Cohere embed-v3, Jina v3 영어+한국어 혼합 검색 가능, 유지보수 단순 한국어 특화 모델 대비 정밀도 약간 낮을 수 있음

한국어 검색 품질 테스트

PYTHON 
from langchain_openai import OpenAIEmbeddings
import numpy as np

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 한국어 의미 검색 테스트
documents = [
    "대한민국의 수도는 서울특별시이다.",
    "김치는 한국의 대표적인 발효 음식이다.",
    "한강은 서울을 가로지르는 주요 하천이다.",
    "비트코인 가격이 사상 최고치를 기록했다.",
    "제주도는 한국에서 가장 큰 섬이다.",
]
doc_vectors = np.array(embeddings.embed_documents(documents))

# 다양한 표현으로 같은 의미의 질문
queries = [
    "한국의 수도가 어디야?",          # 직접적 질문
    "서울은 어느 나라 수도인가?",       # 역방향 질문
    "대한민국 행정 중심지",             # 다른 표현
]

for query in queries:
    query_vec = np.array(embeddings.embed_query(query))
    similarities = np.dot(doc_vectors, query_vec)
    top_idx = np.argmax(similarities)
    print(f"Q: {query}")
    print(f"A: [{similarities[top_idx]:.4f}] {documents[top_idx]}")
    print()
WARNING

한국어 RAG를 구축할 때는 반드시 실제 도메인 데이터로 검색 품질을 테스트해야 한다. 벤치마크 점수와 실제 서비스에서의 성능은 다를 수 있다. 특히 전문 용어(법률, 의료, 금융 등)가 많은 도메인에서는 일반 벤치마크가 현실을 반영하지 못할 수 있다.
