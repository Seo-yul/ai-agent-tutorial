RAG 평가

RAG 시스템의 품질을 정량적으로 측정하고 체계적으로 개선하는 방법을 다룬다

왜 평가가 필수인가?

"작동하는 것 같다"와 "잘 작동한다"는 전혀 다르다. RAG 시스템을 구축한 뒤 몇 가지 질문을 던져보고 "괜찮네"라고 판단하는 것은 평가가 아니다. 그것은 희망 사항이다.

평가 없이 최적화를 시도하면 방향을 완전히 잃는다. 청크 크기를 512에서 1024로 바꿨다고 가정하자. 답변이 좀 달라진 것 같은데, 좋아진 건가? 나빠진 건가? 측정하지 않으면 알 수 없다. 임베딩 모델을 변경하거나, 프롬프트를 수정하거나, retriever 전략을 바꿀 때도 마찬가지다.

RAG 시스템에서 실패가 발생하는 지점은 크게 두 곳이다: 검색 단계와 생성 단계. 검색이 실패하면 관련 없는 문서가 LLM에 전달되고, 생성이 실패하면 올바른 문서를 받고도 엉뚱한 답변을 만들어낸다. 각각을 별도로 평가해야 문제의 원인을 정확히 특정할 수 있다.

    R -->|"✅ 검색 성공"| G["생성 단계"]

WARNING

검색이 실패했는데 생성 단계만 개선하려고 하면 아무 효과가 없다. 반대로 생성이 문제인데 검색 전략을 바꾸면 시간만 낭비한다. 먼저 어디서 실패하는지 진단한 뒤에 개선해야 한다.

RAG 평가의 3가지 축

RAG 시스템을 평가하는 데 가장 널리 사용되는 프레임워크는 RAG Triad이다. 세 가지 축이 서로 다른 관점에서 시스템의 품질을 측정한다.

Context Relevance (컨텍스트 관련성)

검색된 문서가 질문과 관련이 있는가를 측정한다. 사용자가 "Python의 GIL이 뭐야?"라고 물었는데 Java 동시성 문서가 검색되면, context relevance가 낮은 것이다. 이 메트릭은 검색 단계의 품질을 직접적으로 반영한다.

Faithfulness (충실도)

답변이 검색된 문서에 기반하는가를 측정한다. LLM이 context에 없는 정보를 지어내면(환각), faithfulness가 낮아진다. 검색된 문서에 "Python은 1991년에 출시되었다"고 적혀 있는데 답변에서 "1995년에 출시되었다"고 하면 실패다. 이 메트릭은 생성 단계의 환각 여부를 감지한다.

Answer Relevance (답변 관련성)

답변이 질문에 적절히 대답하는가를 측정한다. 질문은 "GIL이 뭐야?"인데 답변이 "Python은 Guido van Rossum이 만들었다"이면, 사실이지만 관련성이 낮다. 검색도 잘 되고 환각도 없지만, 답변이 질문의 핵심을 빗나가는 경우를 감지한다.

평가 축 측정 대상 낮을 때의 의미 개선 방법
Context Relevance 질문 ↔ 검색된 문서 관련 없는 문서가 검색됨 청킹 전략 변경, 임베딩 모델 교체, 검색 방식 개선
Faithfulness 답변 ↔ 검색된 문서 LLM이 문서에 없는 내용을 생성 (환각) 프롬프트 강화, 답변 길이 제한, temperature 낮춤
Answer Relevance 질문 ↔ 답변 답변이 질문의 핵심을 벗어남 프롬프트 개선, LLM 모델 변경
TIP

세 가지 축이 모두 높아야 좋은 RAG 시스템이다. 하나라도 낮으면 전체 답변 품질이 저하된다. 평가할 때는 항상 세 축을 동시에 측정하여 어디가 병목인지 파악해야 한다.

RAGAS 프레임워크

RAGAS(RAG Assessment)는 RAG 시스템을 자동으로 평가하기 위해 설계된 전용 프레임워크다. 핵심 아이디어는 LLM을 활용하여 평가를 자동화하는 것이다. 사람이 수백 개의 답변을 일일이 검토하는 대신, LLM이 각 답변의 품질을 0~1 점수로 매긴다.

설치

BASH 
pip install ragas

핵심 메트릭

RAGAS는 네 가지 핵심 메트릭을 제공한다. 각각이 RAG Triad의 한 축 또는 세부 관점을 측정한다.

메트릭 점수 범위 계산 방법 해석
faithfulness 0 ~ 1 답변에서 주장(claim)을 추출한 뒤, 각 주장이 context에 근거하는지 LLM이 판단. 근거 있는 주장 수 / 전체 주장 수 1에 가까울수록 환각이 적다
answer_relevancy 0 ~ 1 답변에서 역으로 질문을 생성하고, 원래 질문과의 유사도를 계산 1에 가까울수록 답변이 질문에 적합하다
context_precision 0 ~ 1 관련 있는 context가 상위 순위에 위치하는지 평가. 관련 문서가 앞에 올수록 높은 점수 1에 가까울수록 검색 순위가 정확하다
context_recall 0 ~ 1 ground truth 답변의 각 문장이 검색된 context에서 찾을 수 있는지 확인 1에 가까울수록 필요한 정보가 모두 검색되었다

전체 평가 코드

INFO

RAGAS는 내부적으로 LLM(기본값: OpenAI)을 호출하여 평가를 수행한다. 따라서 평가 자체에도 API 비용이 발생한다. 평가 데이터셋이 클수록 비용이 증가하므로, 처음에는 20~50개 수준의 소규모 데이터셋으로 시작하는 것이 좋다.

평가 데이터셋 만들기

평가에서 가장 어려운 부분은 평가 데이터셋을 만드는 것이다. 특히 ground truth(정답 답변)가 필요하다는 점이 부담이 된다. context_recall을 계산하려면 "이 질문의 정답은 이것이다"라는 기준이 있어야 하기 때문이다.

수동 생성

도메인 전문가가 직접 질문-답변 쌍을 작성하는 방식이다. 가장 정확하지만 시간과 비용이 많이 든다. 최소 20~30개의 질문-답변 쌍을 만들되, 다양한 난이도와 유형을 포함해야 한다.

  • 단순 사실 질문: "Python은 언제 출시되었나?" — 문서에 직접 답이 있는 질문
  • 추론 질문: "GIL이 제거되면 어떤 영향이 있을까?" — 여러 문서의 정보를 조합해야 하는 질문
  • 비교 질문: "FastAPI와 Django의 성능 차이는?" — 여러 문서를 참조해야 하는 질문
  • 부정 질문: "문서에 없는 주제에 대한 질문" — 답변 불가를 올바르게 처리하는지 확인

LLM 기반 자동 생성

RAGAS는 문서로부터 평가용 질문-답변 쌍을 자동으로 생성하는 기능을 제공한다. 수동 생성보다 빠르고 저렴하지만, 생성된 데이터의 품질을 반드시 수동으로 검증해야 한다.

PYTHON 
from ragas.testset.generator import TestsetGenerator
from ragas.testset.evolutions import simple, reasoning, multi_context
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.document_loaders import DirectoryLoader

# 문서 로드
loader = DirectoryLoader("./docs", glob="**/*.md")
documents = loader.load()

# 테스트셋 생성기 초기화
generator_llm = ChatOpenAI(model="gpt-4o-mini")
critic_llm = ChatOpenAI(model="gpt-4o")
embeddings = OpenAIEmbeddings()

generator = TestsetGenerator.from_langchain(
    generator_llm=generator_llm,
    critic_llm=critic_llm,
    embeddings=embeddings,
)

# 테스트셋 생성 — 질문 유형 분포 지정
testset = generator.generate_with_langchain_docs(
    documents,
    test_size=20,
    distributions={
        simple: 0.5,       # 단순 사실 질문 50%
        reasoning: 0.25,   # 추론 질문 25%
        multi_context: 0.25,  # 다중 문서 질문 25%
    },
)

# DataFrame으로 변환하여 검토
test_df = testset.to_pandas()
print(test_df[["question", "ground_truth"]].head())
WARNING

자동 생성된 테스트셋을 그대로 사용하면 안 된다. LLM이 생성한 질문 중에는 모호하거나, ground truth가 부정확한 경우가 있다. 반드시 수동으로 리뷰하고 필터링한 뒤 사용해야 한다.

검색 단계 단독 평가

생성 단계(LLM 호출)까지 포함하면 평가가 복잡하고 비용도 높다. 검색 단계만 따로 평가하면 문제 원인을 빠르게 좁힐 수 있다. 검색 평가에는 전통적인 정보 검색(IR) 메트릭을 사용한다.

주요 검색 메트릭

  • Hit Rate (적중률): Top-K 결과에 정답 문서가 하나라도 포함되었으면 1, 아니면 0. 가장 단순하지만 직관적이다.
  • MRR (Mean Reciprocal Rank): 정답 문서가 몇 번째 순위에 나타나는지를 측정한다. 1위에 나오면 1.0, 2위면 0.5, 3위면 0.33이다. 전체 쿼리의 평균을 구한다.
  • nDCG (Normalized Discounted Cumulative Gain): 여러 정답 문서가 있을 때, 이들의 순위를 종합적으로 고려한다. 관련 문서가 상위에 몰려 있을수록 높은 점수를 받는다.
PYTHON 
import numpy as np
from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings


def evaluate_retrieval(
    vectorstore: Chroma,
    eval_queries: list[dict],
    k: int = 5,
) -> dict:
    """
    검색 성능을 평가한다.

    Parameters:
        vectorstore: 벡터 스토어
        eval_queries: [{"query": "...", "relevant_doc_ids": ["id1", "id2"]}]
        k: Top-K 검색 수
    """
    hit_rates = []
    mrrs = []

    for item in eval_queries:
        query = item["query"]
        relevant_ids = set(item["relevant_doc_ids"])

        # 검색 실행
        results = vectorstore.similarity_search(query, k=k)
        result_ids = [doc.metadata.get("doc_id") for doc in results]

        # Hit Rate: Top-K에 정답 문서가 하나라도 있는가?
        hit = 1.0 if relevant_ids & set(result_ids) else 0.0
        hit_rates.append(hit)

        # MRR: 정답 문서의 최초 등장 순위의 역수
        rr = 0.0
        for rank, doc_id in enumerate(result_ids, start=1):
            if doc_id in relevant_ids:
                rr = 1.0 / rank
                break
        mrrs.append(rr)

    return {
        "hit_rate": np.mean(hit_rates),
        "mrr": np.mean(mrrs),
        "total_queries": len(eval_queries),
    }


# 사용 예시
eval_queries = [
    {
        "query": "Python의 GIL이란?",
        "relevant_doc_ids": ["doc_python_gil_01", "doc_python_gil_02"],
    },
    {
        "query": "FastAPI의 비동기 처리 방식",
        "relevant_doc_ids": ["doc_fastapi_async_01"],
    },
]

results = evaluate_retrieval(vectorstore, eval_queries, k=5)
print(f"Hit Rate: {results['hit_rate']:.3f}")
print(f"MRR: {results['mrr']:.3f}")
TIP

검색 평가는 LLM 호출이 필요 없어서 빠르고 저렴하다. 청킹 전략이나 임베딩 모델을 변경할 때, 먼저 검색 평가로 효과를 확인한 뒤에 전체 RAG 평가(RAGAS)를 실행하는 것이 효율적이다.

평가 자동화

평가는 한 번 하고 끝나는 것이 아니다. 문서가 추가되거나, 청킹 전략이 변경되거나, 모델이 교체될 때마다 평가를 다시 실행해야 한다. 이를 수동으로 하면 빠지는 경우가 반드시 생긴다. 평가를 CI/CD 파이프라인에 통합하여 자동화해야 한다.

PYTHON 
"""
evaluate_rag.py — RAG 평가 자동화 스크립트
CI/CD 파이프라인에서 실행하거나 주기적으로 실행한다.
"""
import json
import sys
from datetime import datetime
from pathlib import Path

from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)
from datasets import Dataset

# 평가 기준 임계값
THRESHOLDS = {
    "faithfulness": 0.8,
    "answer_relevancy": 0.75,
    "context_precision": 0.7,
    "context_recall": 0.7,
}


def load_eval_dataset(path: str) -> Dataset:
    """저장된 평가 데이터셋을 로드한다."""
    with open(path) as f:
        data = json.load(f)
    return Dataset.from_dict(data)


def run_evaluation(dataset: Dataset) -> dict:
    """RAGAS 평가를 실행한다."""
    results = evaluate(
        dataset,
        metrics=[
            faithfulness,
            answer_relevancy,
            context_precision,
            context_recall,
        ],
    )
    return dict(results)


def check_thresholds(results: dict) -> list[str]:
    """평가 결과가 임계값을 충족하는지 확인한다."""
    failures = []
    for metric, threshold in THRESHOLDS.items():
        score = results.get(metric, 0)
        if score < threshold:
            failures.append(
                f"{metric}: {score:.3f} (임계값: {threshold})"
            )
    return failures


def save_report(results: dict, failures: list[str]):
    """평가 결과를 파일로 저장한다."""
    report = {
        "timestamp": datetime.now().isoformat(),
        "results": results,
        "thresholds": THRESHOLDS,
        "failures": failures,
        "status": "FAIL" if failures else "PASS",
    }
    report_dir = Path("reports")
    report_dir.mkdir(exist_ok=True)
    report_path = report_dir / f"eval_{datetime.now():%Y%m%d_%H%M%S}.json"
    with open(report_path, "w") as f:
        json.dump(report, f, indent=2, ensure_ascii=False)
    print(f"리포트 저장: {report_path}")
    return report


if __name__ == "__main__":
    dataset = load_eval_dataset("eval_dataset.json")
    results = run_evaluation(dataset)

    print("\n=== RAG 평가 결과 ===")
    for metric, score in results.items():
        status = "✓" if score >= THRESHOLDS.get(metric, 0) else "✗"
        print(f"  {status} {metric}: {score:.3f}")

    failures = check_thresholds(results)
    report = save_report(results, failures)

    if failures:
        print(f"\n❌ 평가 실패: {len(failures)}개 메트릭이 임계값 미달")
        for f in failures:
            print(f"  - {f}")
        sys.exit(1)  # CI/CD에서 실패로 처리
    else:
        print("\n✅ 모든 메트릭이 임계값을 충족")
INFO

위 스크립트를 GitHub Actions나 GitLab CI에 통합하면, 코드 변경 시 자동으로 평가가 실행되고 품질 저하가 감지되면 배포가 차단된다. 평가 리포트를 저장하면 시간에 따른 품질 추이도 확인할 수 있다.

평가 결과 해석과 개선 전략

평가 점수가 나왔으면 그다음은 해석과 개선이다. 각 메트릭이 낮을 때 어디를 고쳐야 하는지 정리하면 다음과 같다.

메트릭이 낮을 때 원인 해결 방법
faithfulness 낮음 LLM이 context에 없는 내용을 지어냄 ① 프롬프트에 "context에 근거해서만 답변하라" 강화
② temperature를 0으로 낮춤
③ 답변 길이를 제한하여 환각 기회 감소
④ 더 강력한 LLM 모델 사용
context_precision 낮음 관련 없는 문서가 상위에 랭크됨 ① 청킹 전략 변경 (크기, 오버랩 조정)
② 더 좋은 임베딩 모델로 교체
③ 하이브리드 검색(벡터 + BM25) 적용
④ Re-ranker 추가
context_recall 낮음 필요한 문서가 검색되지 않음 ① Top-K 값을 늘림 (k=4 → k=8)
② 청크 크기를 줄여 정보 밀도 향상
③ 쿼리 변환(query transformation) 적용
④ 멀티쿼리 검색 적용
answer_relevancy 낮음 답변이 질문 핵심을 벗어남 ① 프롬프트에 "질문에 직접 답변하라" 지시 추가
② 불필요한 context 필터링 (컨텍스트 압축)
③ LLM 모델 업그레이드
TIP

개선을 할 때는 한 번에 하나의 변수만 변경하고 다시 평가해야 한다. 동시에 여러 가지를 바꾸면 어떤 변경이 효과가 있었는지 알 수 없다. 과학 실험과 같은 접근 방식이 필요하다: 가설 → 변경 → 측정 → 분석.

다음 단계

평가 체계가 갖춰지면 RAG 시스템의 품질을 정량적으로 관리할 수 있게 된다. "잘 되는 것 같다"가 아니라 "faithfulness 0.92, context_recall 0.85"처럼 숫자로 말할 수 있다.

이제 평가를 통해 검증된 RAG 시스템을 프로덕션 환경에 배포할 차례다. 다음 챕터에서는 Kubernetes에 RAG 시스템을 배포하는 방법을 다룬다. pgvector, Qdrant 등 벡터 스토어의 K8s 배포부터 FastAPI 서버, 인덱싱 파이프라인, 모니터링까지 프로덕션에 필요한 모든 것을 구성한다.
