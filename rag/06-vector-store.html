벡터 스토어

임베딩 벡터를 저장하고 유사도 검색을 수행하는 데이터베이스를 구축한다

벡터 스토어란?

벡터 스토어(Vector Store)는 임베딩 벡터를 저장하고, 주어진 쿼리 벡터와 가장 유사한 벡터를 빠르게 검색하는 데이터베이스이다. 일반적인 관계형 데이터베이스가 WHERE 조건으로 정확히 일치하는 데이터를 찾는 것과 달리, 벡터 스토어는 의미적 유사도를 기준으로 가장 가까운 데이터를 반환한다.

RAG 파이프라인에서 벡터 스토어는 "두뇌의 장기 기억"에 해당한다. 이전 챕터에서 생성한 임베딩 벡터가 이곳에 저장되고, 사용자 질문이 들어올 때마다 관련 문서를 찾아 LLM에 전달하는 역할을 한다.

"벡터 데이터베이스(Vector Database)"와 "벡터 스토어(Vector Store)"는 흔히 혼용된다. 엄밀히 말하면, 벡터 데이터베이스는 Qdrant, Pinecone처럼 벡터 검색 전용으로 설계된 시스템이고, 벡터 스토어는 pgvector처럼 기존 DB에 벡터 기능을 추가한 것도 포함하는 넓은 의미이다. 이 챕터에서는 모두 "벡터 스토어"로 통칭한다.

벡터 인덱싱 알고리즘

벡터 스토어의 핵심은 수백만 개의 벡터 중에서 유사한 것을 빠르게 찾는 인덱싱 알고리즘이다. 모든 벡터와 하나하나 비교(Brute-force)하면 정확하지만, 데이터가 많아지면 실시간 검색이 불가능해진다. 이를 해결하기 위해 다양한 근사 최근접 이웃(ANN: Approximate Nearest Neighbor) 알고리즘이 존재한다.

Flat (Brute-force)

가장 단순한 방식으로, 쿼리 벡터를 저장된 모든 벡터와 비교한다. 100% 정확하지만 시간 복잡도가 O(n)이므로 데이터 규모가 커지면 매우 느려진다. 수만 건 이하의 소규모 데이터나, 다른 알고리즘의 정확도를 검증하는 기준선(baseline)으로 사용한다.

IVF (Inverted File Index)

벡터를 K개의 클러스터로 나누고, 쿼리와 가장 가까운 클러스터 몇 개만 검색한다. 전체 데이터를 탐색하지 않으므로 속도가 빠르지만, 클러스터 경계에 있는 벡터를 놓칠 수 있다.

파라미터 의미 트레이드오프
nlist 클러스터 수 많을수록 검색 빠르지만 빌드 시간 증가
nprobe 탐색할 클러스터 수 많을수록 정확하지만 느림

HNSW (Hierarchical Navigable Small World)

현재 가장 널리 사용되는 ANN 알고리즘이다. 벡터를 다층 그래프 구조로 연결하고, 상위 층에서 대략적 위치를 찾은 뒤 하위 층에서 정밀 탐색을 수행한다. Skip List와 유사한 원리로, 계층적 탐색 덕분에 매우 빠른 검색 속도를 제공한다.

파라미터 의미 트레이드오프
M 각 노드의 최대 연결 수 클수록 정확하지만 메모리 사용 증가
ef_construction 인덱스 빌드 시 탐색 범위 클수록 인덱스 품질 향상, 빌드 시간 증가
ef_search 검색 시 탐색 범위 클수록 정확하지만 검색 느림

PQ (Product Quantization)

벡터를 여러 서브벡터로 분할하고 각각을 양자화(codebook으로 근사)하여 메모리를 절약한다. 원본 1536차원 float32 벡터(6 KB)를 수십 바이트로 압축할 수 있다. 단독으로 사용하기보다 IVF+PQ, HNSW+PQ처럼 다른 알고리즘과 결합하여 메모리를 절감하면서 속도를 유지한다.

알고리즘 종합 비교

알고리즘 검색 속도 정확도 (Recall) 메모리 적합한 규모
Flat 느림 (O(n)) 100% 벡터 크기 그대로 ~10만 건
IVF 빠름 95~99% 벡터 + 클러스터 정보 10만~1000만 건
HNSW 매우 빠름 95~99% 벡터 + 그래프 (높음) 10만~1억 건
PQ 빠름 90~95% 매우 낮음 (압축) 메모리 제한 환경
TIP

대부분의 RAG 프로젝트에서는 HNSW가 최선의 선택이다. pgvector와 Qdrant 모두 HNSW를 기본 인덱스로 지원한다. 인덱스 파라미터를 튜닝할 때는 ef_construction을 먼저 충분히 높게(128~512) 설정하고, 검색 시 ef_search를 조절하여 속도와 정확도의 균형을 맞추는 것이 일반적이다.

pgvector — PostgreSQL 확장

pgvector는 PostgreSQL에 벡터 데이터 타입과 유사도 검색 기능을 추가하는 확장(extension)이다. 이미 PostgreSQL 인프라를 운영하고 있다면, 별도의 벡터 데이터베이스를 추가하지 않고 기존 DB에서 벡터 검색까지 처리할 수 있다는 것이 최대 장점이다.

장점과 단점

장점 단점
기존 PostgreSQL 인프라 활용 수백만 벡터 이상에서 전용 DB 대비 성능 열위
SQL로 벡터 검색 + 메타데이터 필터링 동시 수행 HNSW 인덱스 빌드가 메모리를 많이 사용
ACID 트랜잭션으로 데이터 일관성 보장 분산 검색 기능이 제한적
관계형 데이터와 벡터 데이터 조인 가능 벡터 전용 고급 기능(양자화 등) 제한적

기본 설정

SQL 
-- pgvector 확장 활성화
CREATE EXTENSION vector;

-- 문서 테이블 생성 (1536차원 벡터 컬럼 포함)
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT NOT NULL,
    metadata JSONB DEFAULT '{}',
    embedding vector(1536)
);

-- HNSW 인덱스 생성 (코사인 유사도 기준)
CREATE INDEX ON documents
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 200);

-- 유사도 검색 쿼리
SELECT id, content, metadata,
       1 - (embedding <=> '[0.023, -0.112, ...]'::vector) AS similarity
FROM documents
ORDER BY embedding <=> '[0.023, -0.112, ...]'::vector
LIMIT 5;
INFO

pgvector에서 <=>는 코사인 거리, <->는 유클리드 거리(L2), <#>는 내적의 음수를 나타내는 연산자이다. 인덱스 생성 시 사용하는 vector_cosine_ops, vector_l2_ops, vector_ip_ops와 쿼리의 연산자를 일치시켜야 인덱스가 올바르게 동작한다.

HNSW 파라미터 가이드

파라미터 기본값 권장 범위 설명
m 16 12~48 노드당 최대 연결 수. 클수록 정확하지만 메모리 증가
ef_construction 64 128~512 인덱스 빌드 품질. 높을수록 좋지만 빌드 시간 증가
SQL 
-- 검색 시 ef_search 파라미터 조절 (세션 단위)
SET hnsw.ef_search = 100;  -- 기본값 40, 높일수록 정확하지만 느림

-- 인덱스 빌드 진행률 확인
SET client_min_messages TO NOTICE;
CREATE INDEX CONCURRENTLY ON documents
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 256);

LangChain 연동

PYTHON 
from langchain_openai import OpenAIEmbeddings
from langchain_postgres import PGVector

# 임베딩 모델 초기화
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# pgvector 벡터 스토어 초기화
CONNECTION_STRING = "postgresql+psycopg://user:password@localhost:5432/ragdb"

store = PGVector(
    connection=CONNECTION_STRING,
    collection_name="my_documents",
    embeddings=embeddings,
)

# 문서 추가
from langchain_core.documents import Document

docs = [
    Document(
        page_content="서울특별시의 인구는 약 950만 명이다.",
        metadata={"source": "statistics.pdf", "year": 2024},
    ),
    Document(
        page_content="부산광역시의 인구는 약 330만 명이다.",
        metadata={"source": "statistics.pdf", "year": 2024},
    ),
    Document(
        page_content="제주도의 연간 관광객 수는 약 1500만 명이다.",
        metadata={"source": "tourism.pdf", "year": 2024},
    ),
]
store.add_documents(docs)

# 유사도 검색
results = store.similarity_search("한국에서 인구가 많은 도시는?", k=2)
for doc in results:
    print(f"[{doc.metadata['source']}] {doc.page_content}")

# 점수 포함 검색
results_with_scores = store.similarity_search_with_score(
    "한국에서 인구가 많은 도시는?", k=2
)
for doc, score in results_with_scores:
    print(f"[{score:.4f}] {doc.page_content}")

Qdrant — 전용 벡터 데이터베이스

Qdrant는 처음부터 벡터 검색을 위해 설계된 전용 데이터베이스이다. Rust로 작성되어 빠르고 메모리 효율적이며, REST API와 gRPC를 모두 지원한다. Named Vectors(하나의 포인트에 여러 벡터 저장), Sparse Vectors, 풍부한 필터링 등 벡터 검색에 특화된 고급 기능을 제공한다.

장점과 단점

장점 단점
뛰어난 검색 성능과 대규모 확장성 별도 인프라 운영 필요
풍부한 Payload 필터링 관계형 데이터 조인 불가
Named Vectors, Sparse Vectors 지원 PostgreSQL 대비 생태계 작음
양자화(Scalar, Product, Binary) 내장 학습 곡선 (벡터 DB 특화 개념)
Rust 기반으로 안정적이고 메모리 효율적 ACID 트랜잭션 미지원

Collection 설정

PYTHON 
from qdrant_client import QdrantClient
from qdrant_client.models import (
    Distance, VectorParams, OptimizersConfigDiff,
    HnswConfigDiff, ScalarQuantizationConfig, ScalarType,
)

client = QdrantClient(url="http://localhost:6333")

# Collection 생성 (상세 설정)
client.create_collection(
    collection_name="documents",
    vectors_config=VectorParams(
        size=1536,                # 벡터 차원
        distance=Distance.COSINE, # 거리 메트릭
        on_disk=False,            # True이면 디스크에 저장 (메모리 절약)
    ),
    hnsw_config=HnswConfigDiff(
        m=16,                     # 노드당 연결 수
        ef_construct=256,         # 빌드 품질
    ),
    optimizers_config=OptimizersConfigDiff(
        indexing_threshold=20000, # 이 수 이상의 벡터가 있을 때 인덱스 빌드
    ),
    # 양자화 설정 (메모리 절약)
    quantization_config=ScalarQuantizationConfig(
        type=ScalarType.INT8,     # float32 → int8로 압축 (4배 절약)
        always_ram=True,          # 양자화된 벡터는 항상 RAM에 유지
    ),
)

TIP

Qdrant는 In-memory 모드도 지원한다. 프로토타입이나 테스트 환경에서는 QdrantClient(":memory:")로 서버 없이 사용할 수 있다. 데이터는 프로세스 종료 시 사라지므로 프로덕션에서는 사용하지 않는다.

pgvector vs Qdrant — 아키텍처 결정 가이드

두 벡터 스토어 중 어떤 것을 선택할지는 프로젝트의 규모, 기존 인프라, 요구사항에 따라 달라진다. 아래에서 주요 기준별로 비교한다.

종합 비교

기준 pgvector Qdrant
검색 성능 좋음 (100만 건 이하) 매우 좋음 (수억 건 가능)
운영 복잡도 낮음 (기존 PG 인프라 활용) 중간 (별도 서비스 운영)
필터링 SQL WHERE 절 (강력함) Payload 필터 (풍부함)
트랜잭션 ACID 지원 미지원
관계형 조인 지원 (SQL) 미지원
양자화 제한적 Scalar / Product / Binary
분산 배포 읽기 복제본 수준 네이티브 분산 모드
비용 기존 PG 인프라 비용만 별도 서버/컨테이너 비용
생태계 PostgreSQL 전체 생태계 벡터 검색 특화 도구

"정답"은 없다. pgvector와 Qdrant를 동시에 사용하는 하이브리드 아키텍처도 가능하다. 예를 들어, 메타데이터가 풍부한 구조화된 데이터는 pgvector에, 대규모 비정형 문서는 Qdrant에 저장하는 방식이다. 처음 시작한다면 pgvector로 프로토타입을 만들고, 규모가 커지면 Qdrant로 마이그레이션하는 전략도 유효하다.

메타데이터 필터링

실제 RAG 시스템에서는 단순한 유사도 검색만으로는 부족하다. "2024년 보고서에서만 검색", "인사부서 문서만 검색", "PDF 파일에서만 검색"처럼 메타데이터 조건과 벡터 검색을 결합해야 하는 경우가 대부분이다.

pgvector에서의 필터링

pgvector의 최대 강점이 바로 이 부분이다. SQL의 WHERE 절을 벡터 검색과 자연스럽게 결합할 수 있다.

SQL 
-- 2024년 보고서에서만 벡터 검색
SELECT id, content,
       1 - (embedding <=> query_vector) AS similarity
FROM documents
WHERE metadata->>'year' = '2024'
  AND metadata->>'source' LIKE '%.pdf'
ORDER BY embedding <=> query_vector
LIMIT 5;
PYTHON 
# LangChain에서 pgvector 메타데이터 필터링
results = store.similarity_search(
    "한국의 인구 통계",
    k=5,
    filter={"year": 2024, "source": "statistics.pdf"},
)

Qdrant에서의 필터링

Qdrant는 Payload Filter라는 전용 필터링 시스템을 제공한다. 다양한 조건 연산자(match, range, geo 등)를 조합할 수 있다.

PYTHON 
from qdrant_client.models import Filter, FieldCondition, MatchValue, Range

# Qdrant 네이티브 필터링
results = client.query_points(
    collection_name="documents",
    query=query_vector,
    query_filter=Filter(
        must=[
            FieldCondition(
                key="metadata.year",
                match=MatchValue(value=2024),
            ),
            FieldCondition(
                key="metadata.category",
                match=MatchValue(value="인구"),
            ),
        ]
    ),
    limit=5,
)

# LangChain에서 Qdrant 메타데이터 필터링
from langchain_qdrant import QdrantVectorStore

results = store.similarity_search(
    "한국의 인구 통계",
    k=5,
    filter={"year": 2024, "category": "인구"},
)
WARNING

메타데이터 필터링은 검색 대상을 좁히므로 검색 속도를 높이지만, 필터 조건이 너무 제한적이면 관련 문서를 놓칠 수 있다. 예를 들어 year=2024 필터를 걸면 2023년에 작성되었지만 여전히 유효한 정보를 찾지 못한다. 필터 조건을 설계할 때는 너무 좁지도, 너무 넓지도 않은 범위를 찾는 것이 중요하다.

Kubernetes에서 벡터 스토어 운영

프로덕션 환경에서 벡터 스토어를 운영할 때 Kubernetes는 유력한 선택지이다. pgvector와 Qdrant 모두 Kubernetes에서의 운영을 공식적으로 지원한다. 이 섹션에서는 핵심 고려사항만 간략히 다루고, 상세 배포 가이드는 후반부 챕터에서 다룬다.

pgvector on Kubernetes

PostgreSQL을 Kubernetes에서 운영하는 가장 성숙한 방법은 CloudNativePG Operator이다. pgvector 확장이 포함된 이미지를 사용하면 된다.

BASH 
# CloudNativePG Operator 설치
kubectl apply --server-side -f \
  https://raw.githubusercontent.com/cloudnative-pg/cloudnative-pg/release-1.25/releases/cnpg-1.25.0.yaml

# pgvector 포함 PostgreSQL 클러스터 배포
cat <<EOF | kubectl apply -f -
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: ragdb
spec:
  instances: 3
  imageName: ghcr.io/cloudnative-pg/postgresql:16-bookworm
  postgresql:
    shared_preload_libraries:
      - vector
  storage:
    size: 50Gi
    storageClass: standard
  resources:
    requests:
      memory: "4Gi"
      cpu: "2"
    limits:
      memory: "8Gi"
      cpu: "4"
EOF

Qdrant on Kubernetes

Qdrant는 공식 Helm Chart를 제공하며, 분산 모드로 수평 확장이 가능하다.

BASH 
# Qdrant Helm Chart 설치
helm repo add qdrant https://qdrant.github.io/qdrant-helm
helm repo update

# 분산 모드로 Qdrant 배포
helm install qdrant qdrant/qdrant \
  --set replicaCount=3 \
  --set persistence.size=50Gi \
  --set resources.requests.memory=4Gi \
  --set resources.requests.cpu=2 \
  --set resources.limits.memory=8Gi \
  --set config.storage.on_disk_payload=true

Kubernetes에서 벡터 스토어를 운영할 때 가장 중요한 것은 PVC(Persistent Volume Claim) 설정이다. 벡터 데이터와 인덱스는 메모리에 올려야 하므로, Pod의 메모리 limit을 충분히 설정해야 한다. HNSW 인덱스는 벡터 데이터 크기의 1.5~2배 메모리를 사용한다는 점을 기억하자. 자세한 Kubernetes 배포 전략은 이후 챕터에서 다룬다.
