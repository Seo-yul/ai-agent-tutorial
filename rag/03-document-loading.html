문서 로딩과 전처리

비정형 문서를 LLM이 처리할 수 있는 텍스트로 변환하는 RAG 파이프라인의 첫 단계

문서 로딩이란?

RAG 파이프라인의 첫 번째 단계는 문서 로딩(Document Loading)이다. 현실 세계의 데이터는 PDF, HTML, Word, CSV, Markdown 등 다양한 형식으로 존재한다. 이런 비정형 문서들을 LLM이 이해할 수 있는 순수 텍스트로 변환하는 과정이 바로 문서 로딩이다.

이 단계가 왜 중요한가? 데이터 과학에서 자주 인용되는 격언이 있다:

WARNING

"Garbage In, Garbage Out" — 입력이 쓰레기이면 출력도 쓰레기이다. 문서 로딩 단계에서 텍스트 추출이 부정확하면, 이후의 청킹, 임베딩, 검색, 생성까지 모든 단계의 품질이 연쇄적으로 하락한다. RAG 시스템의 최종 답변 품질은 이 첫 단계에서 이미 상당 부분 결정된다.

문서 로딩의 목표는 명확하다:

  • 원본 문서에서 텍스트 콘텐츠를 정확하게 추출한다
  • 문서의 메타데이터(파일명, 페이지번호, 작성일 등)를 함께 보존한다
  • 추출된 텍스트에서 노이즈(머리글, 바닥글, 페이지번호 등)를 제거한다
  • LLM이 이해하기 쉬운 깨끗한 텍스트로 정제한다
graph LR
    PDF["PDF 파일"] --> LOADER["Document Loader"]
    HTML["HTML 파일"] --> LOADER
    DOCX["Word 파일"] --> LOADER
    CSV["CSV 파일"] --> LOADER
    LOADER --> DOC["Document 객체\n(page_content + metadata)"]
    DOC --> PREPROCESS["전처리"]
    PREPROCESS --> CLEAN["정제된 텍스트"]

LangChain Document 객체

LangChain에서 모든 문서 로더는 Document 객체를 반환한다. 이 객체는 RAG 파이프라인 전체에서 텍스트 데이터를 표현하는 핵심 자료구조이다. 구조는 매우 단순하다 — 딱 두 가지 속성만 존재한다.

PYTHON 
from langchain_core.documents import Document

# Document 객체의 기본 구조
doc = Document(
    page_content="이것은 문서의 실제 텍스트 내용이다.",
    metadata={
        "source": "report.pdf",
        "page": 0,
        "author": "홍길동",
        "created_date": "2024-01-15"
    }
)

print(doc.page_content)
# "이것은 문서의 실제 텍스트 내용이다."

print(doc.metadata)
# {'source': 'report.pdf', 'page': 0, 'author': '홍길동', 'created_date': '2024-01-15'}
속성 타입 설명
page_content str 문서의 실제 텍스트 내용. 이후 청킹, 임베딩의 대상이 된다.
metadata dict 문서에 대한 부가 정보. 출처 추적, 필터링, 정렬에 사용된다.

metadata가 왜 중요한가?

초보자가 흔히 간과하는 부분이 metadata이다. "텍스트만 추출하면 되지, 부가 정보가 왜 필요한가?"라고 생각할 수 있다. 하지만 metadata는 실전에서 다음과 같은 핵심 역할을 한다:

  • 출처 추적(Source Attribution): 사용자에게 "이 답변은 report.pdf의 23페이지에서 가져왔다"라고 알려줄 수 있다. 출처 없는 RAG는 신뢰를 얻기 어렵다.
  • 필터링 검색: "2024년 이후 문서에서만 검색해줘"와 같은 조건부 검색이 가능해진다.
  • 중복 제거: 같은 source에서 온 중복 청크를 제거할 수 있다.
  • 디버깅: 검색 결과가 이상할 때 어떤 문서에서 왔는지 추적할 수 있다.
TIP

metadata에 최소한 source(파일 경로)는 반드시 포함해야 한다. 가능하다면 page(페이지 번호), title(문서 제목), created_date(작성일)까지 포함하는 것이 이상적이다. 나중에 "이 정보의 출처가 어디인가?"라는 질문에 답할 수 있어야 RAG 시스템의 신뢰성이 확보된다.

PDF 로더 비교

실무에서 RAG를 구축할 때 가장 많이 다루는 문서 형식은 단연 PDF이다. 기업 보고서, 논문, 매뉴얼, 계약서, 정부 문서 — 거의 모든 중요 문서가 PDF로 유통된다. LangChain은 다양한 PDF 로더를 제공하며, 각각 장단점이 뚜렷하다.

PyPDFLoader — 가장 기본적인 로더

PyPDFLoader는 pypdf 라이브러리를 기반으로 한 가장 단순한 PDF 로더이다. 페이지 단위로 텍스트를 추출하며, 설치가 간편하고 의존성이 가볍다. 단순한 텍스트 위주의 PDF에서 잘 동작한다.

PYTHON 
from langchain_community.document_loaders import PyPDFLoader

# PDF 로딩
loader = PyPDFLoader("data/report.pdf")
docs = loader.load()

# 결과 확인
print(f"총 페이지 수: {len(docs)}")
print(f"첫 페이지 내용 (앞 200자):\n{docs[0].page_content[:200]}")
print(f"메타데이터: {docs[0].metadata}")
# {'source': 'data/report.pdf', 'page': 0}

PyPDFLoader의 특징은 페이지별로 하나의 Document 객체를 생성한다는 점이다. 10페이지짜리 PDF를 로딩하면 10개의 Document 객체가 반환된다. metadata에는 source(파일 경로)와 page(0부터 시작하는 페이지 번호)가 자동으로 포함된다.

PyMuPDFLoader — 더 빠르고 정확한 추출

PyMuPDFLoader는 PyMuPDF(fitz) 라이브러리를 사용한다. C 기반 엔진을 사용하기 때문에 pypdf보다 훨씬 빠르다. 텍스트 추출 정확도도 더 높고, metadata도 더 풍부하게 제공한다.

BASH 
# 설치
pip install pymupdf
PYTHON 
from langchain_community.document_loaders import PyMuPDFLoader

loader = PyMuPDFLoader("data/report.pdf")
docs = loader.load()

print(f"총 페이지 수: {len(docs)}")
print(f"메타데이터: {docs[0].metadata}")
# metadata가 더 풍부하다:
# {
#   'source': 'data/report.pdf',
#   'file_path': 'data/report.pdf',
#   'page': 0,
#   'total_pages': 42,
#   'format': 'PDF 1.7',
#   'title': '2024년 연간 보고서',
#   'author': '홍길동',
#   'subject': '',
#   'keywords': '',
#   'creator': 'Microsoft Word',
#   'producer': 'macOS Quartz PDFContext',
#   'creationDate': "D:20240115103000+09'00'",
#   'modDate': "D:20240120154500+09'00'",
# }
TIP

PyMuPDFLoader는 total_pages, author, title, creationDate 등 PDF 문서의 메타 정보를 자동으로 추출한다. 별도의 메타데이터 보강 작업 없이도 풍부한 정보를 얻을 수 있어서 실무에서 선호도가 높다.

UnstructuredPDFLoader — 가장 강력한 로더

UnstructuredPDFLoader는 unstructured 라이브러리를 기반으로 하며, 단순 텍스트 추출을 넘어서 문서의 구조를 이해한다. 테이블, 이미지, 헤더/푸터를 인식하고, Hi-Res 모드에서는 OCR까지 지원한다. 가장 강력하지만 의존성이 무겁고 설치가 복잡하다.

BASH 
# 기본 설치
pip install unstructured
pip install unstructured[pdf]

# Hi-Res 모드 (OCR 포함) — 추가 의존성 필요
pip install unstructured[all-docs]
# Tesseract OCR 엔진도 별도 설치 필요
# macOS: brew install tesseract
# Ubuntu: sudo apt-get install tesseract-ocr
PYTHON 
from langchain_community.document_loaders import UnstructuredPDFLoader

# 기본 모드 — fast 모드 (텍스트만 추출)
loader = UnstructuredPDFLoader("data/report.pdf", mode="single")
docs = loader.load()

# elements 모드 — 문서 구조를 요소별로 분리
loader = UnstructuredPDFLoader("data/report.pdf", mode="elements")
docs = loader.load()

# 각 요소의 타입을 확인할 수 있다
for doc in docs[:5]:
    print(f"타입: {doc.metadata.get('category', 'unknown'):15} | "
          f"내용: {doc.page_content[:50]}")
# 타입: Title           | 내용: 2024년 연간 사업 보고서
# 타입: NarrativeText   | 내용: 본 보고서는 2024년도 사업 성과를 정리한...
# 타입: Table           | 내용: | 항목 | 실적 | 목표 | 달성률 |...
# 타입: Header          | 내용: 제1장 사업 개요
# 타입: ListItem        | 내용: 총 매출 1,200억원 달성
PYTHON 
# Hi-Res 모드 — OCR + 테이블 인식 + 이미지 캡션
loader = UnstructuredPDFLoader(
    "data/scanned_document.pdf",
    mode="elements",
    strategy="hi_res",           # 고해상도 분석
    languages=["kor", "eng"],    # OCR 언어 설정
)
docs = loader.load()

# 스캔된 PDF에서도 텍스트 추출 가능
for doc in docs:
    if doc.metadata.get("category") == "Table":
        print("테이블 발견:", doc.page_content[:100])

PDF 파싱의 현실적 한계

PDF는 원래 "화면에 보이는 그대로 인쇄"하기 위해 설계된 포맷이다. 텍스트의 논리적 구조(단락, 제목, 표)를 저장하도록 설계된 것이 아니다. 그래서 PDF에서 텍스트를 추출하는 것은 본질적으로 어려운 문제이다. 실무에서 개발자가 실제로 겪는 주요 문제들과 대응 방법을 정리한다.

문제 1: 2단 컬럼 레이아웃

학술 논문이나 신문 기사처럼 2단(또는 다단) 컬럼으로 구성된 PDF에서는 텍스트 추출 순서가 뒤섞인다. 왼쪽 컬럼의 첫 줄 다음에 오른쪽 컬럼의 첫 줄이 오는 식이다.

WARNING

대응 방법: UnstructuredPDFLoader의 Hi-Res 모드를 사용하면 레이아웃 분석을 통해 컬럼 순서를 올바르게 추출할 수 있다. 또는 PyMuPDF의 page.get_text("blocks")로 텍스트 블록의 좌표를 직접 분석하는 방법도 있다.

문제 2: 표(Table) 구조 손실

PDF의 표는 시각적으로 행과 열이 정렬되어 보이지만, 내부적으로는 각 셀의 텍스트가 독립적인 텍스트 조각으로 저장되어 있다. 단순 텍스트 추출 시 행/열 구조가 완전히 무너진다.

PYTHON 
# PyPDFLoader로 표를 추출하면 이렇게 된다:
# "항목 실적 목표 달성률 매출 1200억 1000억 120% 영업이익 300억 250억 120%"
# → 행/열 구조가 완전히 사라진다

# 대응 방법 1: UnstructuredPDFLoader의 elements 모드
from langchain_community.document_loaders import UnstructuredPDFLoader

loader = UnstructuredPDFLoader("data/report.pdf", mode="elements")
docs = loader.load()

# 테이블 요소만 필터링
tables = [d for d in docs if d.metadata.get("category") == "Table"]
for table in tables:
    print(table.page_content)

# 대응 방법 2: camelot 또는 tabula로 표만 별도 추출
# pip install camelot-py[cv]
import camelot

tables = camelot.read_pdf("data/report.pdf", pages="all")
for table in tables:
    print(table.df.to_markdown())

문제 3: 스캔된 PDF (이미지 PDF)

종이 문서를 스캔하여 만든 PDF는 텍스트 레이어가 없다. 각 페이지가 하나의 이미지이므로, 일반 PDF 로더로는 텍스트를 전혀 추출할 수 없다. OCR(Optical Character Recognition)이 필수적이다.

PYTHON 
# 스캔된 PDF → OCR로 텍스트 추출
from langchain_community.document_loaders import UnstructuredPDFLoader

loader = UnstructuredPDFLoader(
    "data/scanned.pdf",
    strategy="hi_res",
    languages=["kor", "eng"],  # 한국어 + 영어 OCR
)
docs = loader.load()

# 주의: OCR 결과는 100% 정확하지 않다
# 특히 한국어의 경우 인식률이 영어보다 낮을 수 있다
# 반드시 결과를 검수해야 한다

문제 4: 머리글/바닥글/페이지번호

대부분의 PDF 문서에는 머리글(header), 바닥글(footer), 페이지 번호가 있다. 이것들은 문서의 내용이 아니라 탐색을 위한 요소인데, 텍스트 추출 시 본문과 섞여서 노이즈가 된다. 특히 페이지 번호("- 23 -")나 반복되는 머리글("2024년 연간 보고서")이 모든 페이지에서 추출되면 검색 품질을 심각하게 떨어뜨린다.

PYTHON 
import re

def remove_headers_footers(text: str, patterns: list[str] = None) -> str:
    """머리글, 바닥글, 페이지 번호를 제거한다."""
    # 일반적인 페이지 번호 패턴
    text = re.sub(r'\n\s*-?\s*\d+\s*-?\s*\n', '\n', text)        # "- 23 -" 형식
    text = re.sub(r'\n\s*Page\s+\d+\s*(of\s+\d+)?\s*\n', '\n', text)  # "Page 23 of 100"
    text = re.sub(r'\n\s*\d+\s*/\s*\d+\s*\n', '\n', text)         # "23/100" 형식

    # 사용자 정의 패턴 제거
    if patterns:
        for pattern in patterns:
            text = text.replace(pattern, '')

    return text.strip()

# 사용 예시
for doc in docs:
    doc.page_content = remove_headers_footers(
        doc.page_content,
        patterns=["2024년 연간 보고서", "Confidential"]  # 반복되는 머리글/바닥글
    )

문제 5: 수식, 차트, 이미지 내 텍스트

수학 수식(LaTeX 렌더링), 차트의 축 레이블, 이미지에 포함된 텍스트는 일반 PDF 로더로 추출이 불가능하거나 깨져서 나온다.

  • 수식: nougat(Meta의 학술 문서 OCR) 또는 mathpix API 사용
  • 차트/이미지 내 텍스트: 멀티모달 LLM(GPT-4V, Claude Vision)에 이미지를 전달하여 설명 텍스트 생성
  • 실용적 판단: 모든 정보를 완벽하게 추출하려 하기보다, RAG의 목적에 맞게 필요한 정보만 선별적으로 추출하는 것이 현실적이다
INFO

PDF 파싱은 "완벽한 해결책"이 존재하지 않는 영역이다. 문서의 특성에 맞는 로더를 선택하고, 전처리로 보완하며, 최종적으로는 사람이 결과를 확인하는 과정이 반드시 필요하다. 완벽한 자동화를 기대하기보다 "80%를 자동화하고 20%를 수동으로 보완한다"는 마인드셋이 현실적이다.

전처리 파이프라인

문서 로딩이 끝나면, 추출된 텍스트를 정제하는 전처리(Preprocessing) 단계가 필요하다. 로더가 추출한 원본 텍스트에는 불필요한 공백, 깨진 문자, 반복되는 노이즈가 포함되어 있다. 이 단계에서 텍스트를 깨끗하게 다듬으면 이후 청킹과 임베딩의 품질이 크게 향상된다.

전처리의 핵심 작업

graph TD
    RAW["원본 텍스트\n(로더 출력)"] --> WS["1. 공백/줄바꿈 정리"]
    WS --> HF["2. 머리글/바닥글 제거"]
    HF --> CHAR["3. 특수문자 정규화"]
    CHAR --> META["4. 메타데이터 보강"]
    META --> CLEAN["정제된 Document"]
PYTHON 
import re
from langchain_core.documents import Document


def preprocess_document(doc: Document) -> Document:
    """Document 객체의 텍스트를 정제한다."""
    text = doc.page_content

    # 1. 불필요한 공백/줄바꿈 정리
    text = re.sub(r'\n{3,}', '\n\n', text)     # 3개 이상 연속 줄바꿈 → 2개로
    text = re.sub(r'[ \t]+', ' ', text)         # 연속 공백/탭 → 공백 1개로
    text = re.sub(r' *\n *', '\n', text)        # 줄바꿈 전후 공백 제거

    # 2. 머리글/바닥글/페이지번호 제거
    text = re.sub(r'\n\s*-?\s*\d{1,3}\s*-?\s*\n', '\n', text)
    text = re.sub(r'\n\s*Page\s+\d+.*?\n', '\n', text, flags=re.IGNORECASE)

    # 3. 특수문자 정규화
    text = text.replace('\u00a0', ' ')          # non-breaking space → 일반 공백
    text = text.replace('\u200b', '')            # zero-width space 제거
    text = text.replace('\ufeff', '')            # BOM 제거
    text = re.sub(r'[\x00-\x08\x0b\x0c\x0e-\x1f]', '', text)  # 제어 문자 제거

    # 4. 하이픈으로 끊긴 단어 복원 (영문 PDF에서 흔함)
    text = re.sub(r'(\w)-\n(\w)', r'\1\2', text)

    text = text.strip()

    # 메타데이터 보강
    metadata = doc.metadata.copy()
    metadata['char_count'] = len(text)
    metadata['word_count'] = len(text.split())

    return Document(page_content=text, metadata=metadata)


# 전처리 적용

메타데이터 보강

로더가 자동으로 제공하는 metadata 외에도, 프로젝트에 필요한 추가 정보를 metadata에 넣는 것이 좋다.

PYTHON 
import os
from datetime import datetime


def enrich_metadata(doc: Document, extra: dict = None) -> Document:
    """Document의 메타데이터를 보강한다."""
    metadata = doc.metadata.copy()

    # 파일 정보 추가
    source = metadata.get("source", "")
    if source and os.path.exists(source):
        stat = os.stat(source)
        metadata["file_size_kb"] = round(stat.st_size / 1024, 1)
        metadata["file_modified"] = datetime.fromtimestamp(
            stat.st_mtime
        ).isoformat()

    # 파일명에서 카테고리 추론 (프로젝트별 규칙)
    filename = os.path.basename(source)
    metadata["filename"] = filename

    # 텍스트 통계
    metadata["char_count"] = len(doc.page_content)
    metadata["is_empty"] = len(doc.page_content.strip()) == 0

    # 사용자 정의 메타데이터 병합
    if extra:
        metadata.update(extra)

    return Document(page_content=doc.page_content, metadata=metadata)


# 사용 예시
enriched_docs = [
    enrich_metadata(doc, extra={
        "project": "rag-tutorial",
        "loaded_at": datetime.now().isoformat(),
        "loader": "PyMuPDFLoader",
    })
    for doc in cleaned_docs
]

디렉토리 일괄 로딩

실무에서는 PDF 파일 하나가 아니라, 디렉토리에 있는 수십~수백 개의 PDF를 한번에 로딩해야 하는 경우가 대부분이다. DirectoryLoader는 이 작업을 간단하게 처리해준다.

PYTHON 
from langchain_community.document_loaders import DirectoryLoader, PyPDFLoader

# 디렉토리 내 모든 PDF를 한번에 로딩
loader = DirectoryLoader(
    "data/documents/",         # 검색할 디렉토리
    glob="**/*.pdf",           # 하위 디렉토리 포함, PDF만
    loader_cls=PyPDFLoader,    # 사용할 로더 클래스
    show_progress=True,        # 진행률 표시
    use_multithreading=True,   # 멀티스레딩으로 속도 향상
)
docs = loader.load()
print(f"총 {len(docs)}개의 Document 로딩 완료")
PYTHON 
# 다양한 형식의 파일을 한번에 로딩하는 패턴
from langchain_community.document_loaders import (
    DirectoryLoader,
    PyMuPDFLoader,
    TextLoader,
    UnstructuredMarkdownLoader,
)

def load_all_documents(base_dir: str) -> list[Document]:
    """디렉토리에서 다양한 형식의 문서를 일괄 로딩한다."""
    all_docs = []

    # PDF 로딩
    pdf_loader = DirectoryLoader(
        base_dir, glob="**/*.pdf",
        loader_cls=PyMuPDFLoader,
        show_progress=True,
    )
    all_docs.extend(pdf_loader.load())

    # 텍스트 파일 로딩
    txt_loader = DirectoryLoader(
        base_dir, glob="**/*.txt",
        loader_cls=TextLoader,
        loader_kwargs={"encoding": "utf-8"},
    )
    all_docs.extend(txt_loader.load())

    # Markdown 로딩
    md_loader = DirectoryLoader(
        base_dir, glob="**/*.md",
        loader_cls=UnstructuredMarkdownLoader,
    )
    all_docs.extend(md_loader.load())

    print(f"총 {len(all_docs)}개 Document 로딩 완료")
    print(f"  - PDF: {sum(1 for d in all_docs if d.metadata.get('source', '').endswith('.pdf'))}개")
    print(f"  - TXT: {sum(1 for d in all_docs if d.metadata.get('source', '').endswith('.txt'))}개")
    print(f"  - MD:  {sum(1 for d in all_docs if d.metadata.get('source', '').endswith('.md'))}개")

    return all_docs


docs = load_all_documents("data/knowledge_base/")
TIP

DirectoryLoader에 use_multithreading=True를 설정하면 여러 파일을 동시에 로딩하여 속도가 크게 향상된다. 파일이 수백 개 이상일 때는 반드시 활성화하는 것을 권장한다.

실전 팁

팁 1: 로딩 결과를 반드시 눈으로 확인할 것

문서 로딩 후 가장 중요한 작업은 결과 검증이다. 코드가 에러 없이 실행되었다고 해서 텍스트가 올바르게 추출되었다는 보장은 없다. 반드시 몇 개의 Document를 샘플링하여 내용을 확인해야 한다.

PYTHON 
import random

def inspect_documents(docs: list[Document], sample_size: int = 3):
    """로딩된 문서의 품질을 빠르게 확인한다."""
    print(f"=== 문서 로딩 결과 요약 ===")
    print(f"총 Document 수: {len(docs)}")

    # 빈 문서 검출
    empty_docs = [d for d in docs if len(d.page_content.strip()) == 0]
    print(f"빈 Document 수: {len(empty_docs)}")
    if empty_docs:
        print(f"  빈 문서 출처: {[d.metadata.get('source', '?') for d in empty_docs[:5]]}")

    # 텍스트 길이 통계
    lengths = [len(d.page_content) for d in docs if d.page_content.strip()]
    if lengths:
        print(f"평균 텍스트 길이: {sum(lengths) / len(lengths):.0f}자")
        print(f"최소 길이: {min(lengths)}자")
        print(f"최대 길이: {max(lengths)}자")

    # 랜덤 샘플 출력
    print(f"\n=== 랜덤 샘플 ({sample_size}개) ===")
    samples = random.sample(docs, min(sample_size, len(docs)))
    for i, doc in enumerate(samples):
        print(f"\n--- 샘플 {i+1} ---")
        print(f"출처: {doc.metadata.get('source', '알 수 없음')}")
        print(f"페이지: {doc.metadata.get('page', '알 수 없음')}")
        print(f"길이: {len(doc.page_content)}자")
        print(f"내용 미리보기:\n{doc.page_content[:300]}...")

inspect_documents(docs)

팁 2: 대용량 PDF는 lazy_load() 사용

수백 페이지짜리 PDF나 대량의 파일을 로딩할 때는 메모리 문제가 발생할 수 있다. load()는 모든 Document를 한번에 메모리에 올리지만, lazy_load()는 제너레이터로 하나씩 반환하여 메모리를 절약한다.

PYTHON 
from langchain_community.document_loaders import PyPDFLoader

loader = PyPDFLoader("data/huge_report_1000pages.pdf")

# 비효율적 — 모든 페이지를 한번에 메모리에 로딩
# docs = loader.load()

# 효율적 — 제너레이터로 하나씩 처리
for doc in loader.lazy_load():
    # 각 페이지를 하나씩 처리

팁 3: source 메타데이터는 필수이다

RAG에서 답변의 신뢰성은 출처 제공에 달려 있다. 사용자가 "이 정보의 근거가 무엇인가?"라고 물었을 때, "report.pdf의 23페이지에 근거한다"라고 답할 수 있어야 한다. 이를 위해 모든 Document의 metadata에 source 정보가 반드시 포함되어야 한다.

PYTHON 
# RAG 답변에서 출처를 제공하는 예시
def format_answer_with_sources(answer: str, source_docs: list[Document]) -> str:
    """답변에 출처 정보를 첨부한다."""
    sources = set()
    for doc in source_docs:
        source = doc.metadata.get("source", "알 수 없음")
        page = doc.metadata.get("page", "")
        if page != "":
            sources.add(f"{source} (p.{page + 1})")  # 0-indexed → 1-indexed
        else:
            sources.add(source)

    source_text = "\n".join(f"  - {s}" for s in sorted(sources))
    return f"{answer}\n\n[출처]\n{source_text}"
WARNING

PyPDFLoader의 page 번호는 0부터 시작한다. 사용자에게 출처를 표시할 때는 page + 1로 변환해야 한다. 이 작은 차이를 놓치면 사용자가 출처를 확인할 때 혼란이 생긴다.

다음 단계

이번 장에서는 RAG 파이프라인의 첫 단계인 문서 로딩과 전처리를 다뤘다. 비정형 문서를 LLM이 처리할 수 있는 깨끗한 텍스트로 변환하는 과정, 다양한 PDF 로더의 특성, 그리고 전처리를 통한 텍스트 정제 방법을 살펴보았다.

핵심을 정리하면 다음과 같다:

  • Document(page_content, metadata)는 LangChain에서 텍스트 데이터를 표현하는 핵심 객체이다
  • PDF 로더는 PyMuPDFLoader를 기본으로 선택하고, 복잡한 문서는 UnstructuredPDFLoader를 사용한다
  • PDF 파싱은 본질적으로 불완전하다 — 전처리로 보완하고, 결과를 반드시 검증해야 한다
  • metadata에 source 정보를 반드시 포함하여 답변의 출처를 추적할 수 있게 한다

다음 장에서는 로딩된 문서를 적절한 크기로 분할하는 청킹(Chunking) 전략을 다룬다. 청킹은 RAG의 검색 품질을 결정하는 가장 중요한 요소 중 하나이며, chunk_size, overlap, 분할 전략의 선택이 시스템 전체 성능에 직접적인 영향을 미친다.
