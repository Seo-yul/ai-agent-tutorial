검색 품질 최적화

MMR, 하이브리드 검색, Re-ranking, Contextual Compression으로 RAG를 프로덕션 수준으로 끌어올린다

왜 기본 검색으로는 부족한가?

앞선 챕터에서 구축한 Naive RAG 파이프라인은 동작은 하지만 프로덕션에 투입하기에는 부족하다. 사용자 질문을 임베딩하고 벡터 DB에서 Top-K 유사 문서를 가져오는 단순한 구조는 실제 환경에서 다양한 실패 모드를 드러낸다. 이 장에서는 그 실패 유형을 분석하고, 각각에 대한 검증된 해결 전략을 깊이 있게 다룬다.

Naive RAG의 검색 실패 유형

기본 벡터 검색이 실패하는 대표적인 패턴은 세 가지이다.

1. 어휘 불일치 (Lexical Mismatch)

사용자가 "CEO"로 질문했는데 문서에는 "대표이사"로 기록되어 있다. 임베딩 모델이 두 표현의 의미적 유사성을 잡아낼 수도 있지만, 도메인 특화 용어나 약어에서는 실패하는 경우가 빈번하다. "K8s"와 "Kubernetes", "RAG"와 "검색 증강 생성" 같은 경우이다.

2. 중복 문서 독점

비슷한 내용의 청크가 Top-K를 독점하는 문제이다. 예를 들어 회사 소개 문서가 여러 버전 존재하면, "회사 설립 연도"에 대한 질문에 거의 동일한 내용의 청크 4개가 반환된다. 다양한 관점의 정보를 놓치게 되고, 컨텍스트 윈도우가 낭비된다.

3. 맥락 부족 (Relevance Dilution)

질문과 관련은 있지만 핵심이 아닌 문서가 검색되는 현상이다. "2024년 매출"을 물었는데 "2024년 사업 전략"이나 "2023년 매출"이 상위에 올라온다. 벡터 유사도만으로는 미세한 관련성 차이를 구분하기 어렵다.

MMR (Maximal Marginal Relevance) — 다양성 확보

MMR(Maximal Marginal Relevance)은 검색 결과의 관련성(Relevance)과 다양성(Diversity) 사이의 균형을 잡는 알고리즘이다. 1998년 Carbonell과 Goldstein이 발표한 논문에서 처음 제안되었으며, 정보 검색 분야에서 오랫동안 사용되어 온 검증된 기법이다.

원리

일반 벡터 검색은 쿼리와의 유사도만 기준으로 상위 K개를 선택한다. MMR은 여기에 한 가지 조건을 추가한다: "이미 선택된 문서와 너무 유사한 문서에는 패널티를 부여한다."

구체적으로, MMR의 점수 공식은 다음과 같다:

MMR = λ × Sim(query, doc) - (1 - λ) × max(Sim(doc, selected_doc))

여기서 λ(lambda) 파라미터가 관련성과 다양성의 비율을 조절한다:

  • λ = 1.0: 관련성만 고려. 일반 벡터 검색과 동일하다.
  • λ = 0.0: 다양성만 고려. 이미 선택된 문서와 최대한 다른 문서를 선택한다.
  • λ = 0.5~0.7: 실무에서 가장 많이 사용하는 범위. 관련성을 유지하면서 적절한 다양성을 확보한다.

LangChain에서 MMR 사용

PYTHON 
# 벡터 스토어에서 MMR 검색기 생성
retriever = store.as_retriever(
    search_type="mmr",
    search_kwargs={
        "k": 4,              # 최종 반환할 문서 수
        "fetch_k": 20,       # MMR 계산을 위해 먼저 가져올 후보 문서 수
        "lambda_mult": 0.7,  # 관련성 vs 다양성 비율 (0~1)
    }
)

# 검색 실행
results = retriever.invoke("CEO의 경영 방침은 무엇인가?")

for doc in results:
    print(f"[{doc.metadata.get('source', '')}] {doc.page_content[:80]}...")

fetch_k vs k의 차이

MMR 검색에서 fetch_k와 k는 서로 다른 역할을 한다. 이 차이를 이해하는 것이 MMR 튜닝의 핵심이다.

  1. 1단계: 벡터 유사도 기준으로 fetch_k개의 후보 문서를 가져온다. (예: 20개)
  2. 2단계: 20개 후보에 대해 MMR 알고리즘을 적용하여, 관련성과 다양성을 모두 고려한 k개를 선별한다. (예: 4개)

fetch_k가 너무 작으면 MMR이 선택할 수 있는 후보 풀이 좁아져 다양성 확보가 어렵다. 일반적으로 fetch_k는 k의 3~5배로 설정하는 것이 좋다.

TIP

MMR은 문서 중복이 많을 때 특히 효과적이다. 예를 들어 같은 내용이 여러 버전으로 존재하는 사내 문서, 뉴스 기사 아카이브, 유사한 제품 설명이 반복되는 카탈로그 등에서 진가를 발휘한다. 반면 문서가 이미 충분히 다양하다면 일반 유사도 검색으로도 충분하다.

하이브리드 검색 (Sparse + Dense) — 어휘와 의미를 모두 잡는다

벡터 검색(Dense Retrieval)은 의미적 유사성을 잡는 데 탁월하지만, 정확한 키워드 매칭에서는 의외로 약하다. 반대로 키워드 기반 검색(Sparse Retrieval)은 정확한 용어를 찾는 데 강하지만, 동의어나 유사 표현을 놓친다. 하이브리드 검색은 이 두 가지를 결합하여 양쪽의 장점을 모두 취하는 전략이다.

Dense 검색 vs Sparse 검색

Dense 검색 (벡터 검색)은 텍스트를 고밀도 벡터(dense vector)로 변환하여 코사인 유사도 등으로 비교한다. "자동차"로 검색하면 "차량", "vehicle", "승용차"도 함께 매칭된다. 의미적 이해가 강점이지만, 모델이 학습하지 못한 전문 용어나 고유명사에서는 실패할 수 있다.

Sparse 검색 (BM25)은 TF-IDF를 개선한 BM25 알고리즘을 사용하여 문서와 쿼리 사이의 키워드 중복도를 계산한다. 정확한 용어 매칭에 강하고, 도메인 특화 약어나 코드명 검색에 유리하다. 반면 동의어나 유사 표현은 전혀 잡지 못한다.

적합한 상황 일반 자연어 질문 코드, 약어, 고유명사 검색 범용 (대부분의 상황)

EnsembleRetriever로 결합

LangChain의 EnsembleRetriever는 여러 검색기의 결과를 결합하는 가장 간편한 방법이다. 내부적으로 Reciprocal Rank Fusion (RRF) 알고리즘을 사용하여 각 검색기의 순위를 통합한다.

PYTHON 
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever

# 1. BM25 검색기 (Sparse)
bm25_retriever = BM25Retriever.from_documents(docs, k=4)

# 2. 벡터 검색기 (Dense)
vector_retriever = store.as_retriever(search_kwargs={"k": 4})

# 3. 하이브리드 검색기 — 두 검색기를 결합
ensemble = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.4, 0.6],  # BM25 40%, 벡터 60%
)

# 하이브리드 검색 실행
results = ensemble.invoke("Kubernetes 클러스터 설정 방법")
for doc in results:
    print(f"[Hybrid] {doc.page_content[:80]}...")

Reciprocal Rank Fusion (RRF) 알고리즘

RRF는 여러 검색 결과 리스트를 하나로 통합하는 알고리즘이다. 핵심 아이디어는 단순하다: 각 문서의 순위(rank)를 기반으로 점수를 계산하고, 여러 검색기의 점수를 합산한다.

각 문서의 RRF 점수는 다음과 같이 계산된다:

RRF_score(d) = Σ 1 / (k + rank(d))

여기서 k는 상수(보통 60)이고, rank(d)는 해당 검색기에서의 순위이다. 순위가 높을수록(숫자가 작을수록) 점수가 높고, 여러 검색기에서 동시에 상위에 등장하는 문서가 최종 상위에 오게 된다.

weight 튜닝 가이드

weights 파라미터는 각 검색기의 기여도를 조절한다. 최적의 비율은 데이터 특성에 따라 다르다:

  • 전문 용어/약어가 많은 기술 문서: BM25 가중치를 높인다 → [0.5, 0.5] 또는 [0.6, 0.4]
  • 일반 자연어 질문이 대부분: 벡터 가중치를 높인다 → [0.3, 0.7]
  • 고유명사/코드명이 중요한 도메인: BM25를 더 강하게 → [0.6, 0.4]
  • 기본 출발점: [0.4, 0.6]으로 시작하여 평가 결과를 보며 조정한다

Qdrant의 내장 하이브리드 검색

Qdrant 벡터 DB는 sparse vector를 네이티브로 지원한다. 별도의 BM25 인덱스를 관리할 필요 없이, 하나의 컬렉션에서 dense + sparse 검색을 동시에 수행할 수 있다. 대규모 프로덕션 환경에서는 이 방식이 별도 BM25 인덱스를 운영하는 것보다 관리 부담이 적다.

PYTHON 
from qdrant_client import QdrantClient, models

client = QdrantClient(url="http://localhost:6333")

# sparse vector를 포함한 컬렉션 생성
client.create_collection(
    collection_name="hybrid_docs",
    vectors_config={
        "dense": models.VectorParams(
            size=1536,
            distance=models.Distance.COSINE,
        )
    },
    sparse_vectors_config={
        "sparse": models.SparseVectorParams()
    },
)

# 하이브리드 검색 — dense + sparse를 동시에 사용
results = client.query_points(
    collection_name="hybrid_docs",
    prefetch=[
        models.Prefetch(
            query=dense_vector,       # dense 임베딩 벡터
            using="dense",
            limit=20,
        ),
        models.Prefetch(
            query=models.SparseVector(
                indices=sparse_indices,   # BM25 토큰 인덱스
                values=sparse_values,     # BM25 가중치
            ),
            using="sparse",
            limit=20,
        ),
    ],
    query=models.FusionQuery(fusion=models.Fusion.RRF),  # RRF 결합
    limit=4,
)
WARNING

BM25는 문서 전체를 메모리에 올려야 하므로, 대규모 데이터셋에서는 메모리 사용량에 주의해야 한다. 수백만 건 이상의 문서를 다루는 경우, Elasticsearch나 Qdrant 같은 네이티브 하이브리드 검색을 지원하는 엔진을 사용하는 것이 바람직하다.

Re-ranking (재순위화) — 검색 후 결과를 다시 정렬

벡터 검색에 사용되는 임베딩 모델은 Bi-encoder 구조이다. 쿼리와 문서를 각각 독립적으로 인코딩한 뒤 유사도를 비교한다. 빠르지만, 쿼리와 문서의 세밀한 상호작용을 포착하지 못하는 한계가 있다.

Re-ranking은 이 한계를 Cross-encoder로 보완한다. Cross-encoder는 쿼리와 문서를 함께 하나의 입력으로 넣어 관련성 점수를 직접 산출한다. 훨씬 정확하지만 모든 쿼리-문서 쌍에 대해 추론을 수행해야 하므로 느리다.

Two-stage Retrieval Pipeline

실무에서는 두 단계를 조합하여 속도와 정확도를 모두 잡는다:

  1. 1단계 (Bi-encoder): 전체 문서에서 빠르게 후보 N개를 검색한다. (예: 20~50개)
  2. 2단계 (Cross-encoder): 후보 N개를 정밀하게 재순위화하여 최종 K개를 선별한다. (예: 4개)

Cross-encoder의 원리

Bi-encoder와 Cross-encoder의 핵심 차이를 이해해야 Re-ranking이 왜 효과적인지 알 수 있다.

  • Bi-encoder: 쿼리와 문서를 독립적으로 인코딩한다. Embed(query)와 Embed(doc)를 각각 계산한 뒤 코사인 유사도를 비교한다. 문서 벡터를 미리 계산해둘 수 있어 빠르다. 하지만 쿼리의 특정 단어가 문서의 어떤 부분과 대응하는지 세밀하게 파악하지 못한다.
  • Cross-encoder: [CLS] query [SEP] document [SEP] 형태로 쿼리와 문서를 하나의 입력으로 결합하여 Transformer에 넣는다. Self-attention을 통해 쿼리와 문서의 모든 토큰이 상호작용한다. 훨씬 정확하지만, 모든 쌍에 대해 추론해야 하므로 느리다.

LangChain + Cohere Reranker

Cohere의 Rerank API는 가장 쉽게 사용할 수 있는 상용 Re-ranking 서비스이다. LangChain의 ContextualCompressionRetriever와 결합하여 간단히 적용할 수 있다.

BASH 
# Cohere 패키지 설치
pip install langchain-cohere
PYTHON 
from langchain.retrievers import ContextualCompressionRetriever
from langchain_cohere import CohereRerank

# Cohere Reranker 설정
reranker = CohereRerank(
    model="rerank-v3.5",   # 최신 rerank 모델
    top_n=4,               # 재순위화 후 반환할 문서 수
)

# 기본 벡터 검색기 (1단계 — 후보 추출)
vector_retriever = store.as_retriever(
    search_kwargs={"k": 20}  # 후보를 넉넉히 가져온다
)

# Re-ranking 검색기 (2단계 — 재순위화)
retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=vector_retriever,
)

# 검색 실행 — 내부적으로 20개 후보 → Rerank → 4개 반환
results = retriever.invoke("2024년 3분기 매출 실적은?")

for doc in results:
    print(f"[Score: {doc.metadata.get('relevance_score', 'N/A'):.3f}] "
          f"{doc.page_content[:80]}...")

오픈소스 대안: Cross-encoder 직접 사용

Cohere API 비용이 부담되거나 외부 API 호출이 제한된 환경이라면, Hugging Face의 오픈소스 Cross-encoder 모델을 직접 사용할 수 있다.

PYTHON 
from langchain.retrievers.document_compressors import CrossEncoderReranker
from langchain_community.cross_encoders import HuggingFaceCrossEncoder

# 오픈소스 Cross-encoder 모델 로딩
cross_encoder = HuggingFaceCrossEncoder(
    model_name="cross-encoder/ms-marco-MiniLM-L-6-v2"
)

# Reranker 래퍼
reranker = CrossEncoderReranker(
    model=cross_encoder,
    top_n=4,
)

# ContextualCompressionRetriever와 동일하게 사용
retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=vector_retriever,
)

results = retriever.invoke("Kubernetes pod 재시작 정책")
TIP

Re-ranking의 핵심 트레이드오프는 정확도 vs 지연시간/비용이다. 1단계에서 20개 후보를 가져와 Re-ranking하면, Cross-encoder 추론이 20회 필요하다. 후보 수를 늘릴수록 정확도는 올라가지만 지연시간도 증가한다. 실무에서는 20~50개 후보가 적절한 범위이다. Cohere Rerank API는 호출당 과금되므로, 트래픽이 많은 서비스에서는 비용 계산을 반드시 해야 한다.

Contextual Compression — 검색된 문서에서 핵심만 추출

검색된 문서(청크)가 항상 질문에 대한 답변으로 가득 차 있는 것은 아니다. 800토큰짜리 청크에서 실제로 관련된 부분은 100토큰에 불과한 경우가 흔하다. 나머지 700토큰은 LLM의 컨텍스트 윈도우를 낭비하고, 오히려 답변 품질을 저하시킬 수 있다.

Contextual Compression은 검색된 문서에서 질문에 관련된 부분만 추출하여 LLM에 전달하는 기법이다. 문서 전체가 아니라 핵심만 남기므로, 컨텍스트 효율성이 크게 향상된다.

LLMChainExtractor — LLM으로 관련 부분 추출

가장 정확한 방법이다. LLM을 사용하여 각 문서에서 질문과 관련된 부분만 추출한다. 품질은 높지만, 모든 문서에 대해 LLM 호출이 필요하므로 비용과 지연시간이 증가한다.

PYTHON 
from langchain.retrievers.document_compressors import LLMChainExtractor
from langchain.retrievers import ContextualCompressionRetriever
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# LLM 기반 추출기
compressor = LLMChainExtractor.from_llm(llm)

# 압축 검색기
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=vector_retriever,
)

# 검색 실행 — 각 문서에서 관련 부분만 추출되어 반환된다
results = compression_retriever.invoke("2024년 매출 목표는?")

for doc in results:
    # page_content가 원본보다 짧아져 있다
    print(f"[압축됨] {doc.page_content}")
    print(f"  원본 출처: {doc.metadata.get('source', '')}")
    print()

EmbeddingsFilter — 임베딩 유사도로 문장 단위 필터링

LLM 호출 없이 임베딩 유사도만으로 문장을 필터링하는 방법이다. 각 문장의 임베딩을 쿼리 임베딩과 비교하여, 유사도가 임계값 이상인 문장만 남긴다. LLM 호출이 없으므로 저렴하고 빠르다.

PYTHON 
from langchain.retrievers.document_compressors import EmbeddingsFilter
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 임베딩 기반 필터 — 유사도 0.75 이상인 문장만 남긴다
embeddings_filter = EmbeddingsFilter(
    embeddings=embeddings,
    similarity_threshold=0.75,
)

compression_retriever = ContextualCompressionRetriever(
    base_compressor=embeddings_filter,
    base_retriever=vector_retriever,
)

results = compression_retriever.invoke("서비스 장애 대응 절차는?")

for doc in results:
    print(f"[필터링됨] {doc.page_content[:100]}...")
INFO

LLMChainExtractor는 정확하지만 비용이 높고, EmbeddingsFilter는 저렴하지만 정밀도가 상대적으로 낮다. 실무에서는 청크 크기가 크고(1000토큰 이상) 비용에 여유가 있을 때 LLMChainExtractor를, 비용을 아끼면서 빠른 응답이 필요할 때 EmbeddingsFilter를 선택한다. Re-ranking과 Compression을 결합하여 파이프라인으로 구성하는 것도 가능하다.

메타데이터 필터링 — 구조화된 조건과 벡터 검색 결합

벡터 유사도 검색만으로는 "최근 1년 문서에서만 검색해줘"나 "기술 문서 카테고리에서만 찾아줘" 같은 구조화된 조건을 처리할 수 없다. 메타데이터 필터링은 벡터 검색에 구조화된 조건을 결합하여 검색 범위를 정밀하게 제어하는 기법이다.

시간 기반 필터

"최근 1년 문서만", "2024년 이후 작성된 보고서만" 같은 조건이다. 벡터 스토어의 metadata 필터를 직접 사용한다.

PYTHON 
# Chroma에서 메타데이터 필터링
retriever = store.as_retriever(
    search_kwargs={
        "k": 4,
        "filter": {
            "year": {"$gte": 2024},              # 2024년 이후
            "category": {"$eq": "tech_report"},   # 기술 보고서만
        }
    }
)

results = retriever.invoke("최신 아키텍처 변경 사항")

Self-Query Retriever — LLM이 필터 조건을 자동 추출

사용자가 명시적으로 필터 조건을 지정하지 않아도, LLM이 자연어 질문에서 필터 조건을 자동으로 추출하는 강력한 기능이다. "2024년 3분기 재무 보고서에서 매출 추이를 알려줘"라는 질문에서 LLM이 year=2024, quarter=3, category="financial"를 자동으로 파싱한다.

PYTHON 
from langchain.retrievers.self_query.base import SelfQueryRetriever
from langchain.chains.query_constructor.schema import AttributeInfo
from langchain_openai import ChatOpenAI

# 메타데이터 스키마 정의 — LLM이 이 정보를 보고 필터를 생성한다
metadata_field_info = [
    AttributeInfo(
        name="year",
        description="문서가 작성된 연도",
        type="integer",
    ),
    AttributeInfo(
        name="category",
        description="문서 카테고리. 'financial', 'tech_report', 'hr', 'legal' 중 하나",
        type="string",
    ),
    AttributeInfo(
        name="department",
        description="문서를 작성한 부서명",
        type="string",
    ),
]

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# Self-Query Retriever 생성
self_query_retriever = SelfQueryRetriever.from_llm(
    llm=llm,
    vectorstore=store,
    document_contents="회사의 내부 문서 (재무 보고서, 기술 문서, 인사 정책 등)",
    metadata_field_info=metadata_field_info,
    verbose=True,  # 생성된 필터를 확인할 수 있다
)

# 자연어 질문 → LLM이 필터 조건 자동 추출
results = self_query_retriever.invoke(
    "2024년 기술팀에서 작성한 아키텍처 관련 문서"
)
# 내부적으로 LLM이 다음 필터를 생성한다:
# filter: year == 2024 AND department == "기술팀"
# query: "아키텍처 관련 문서"
TIP

Self-Query Retriever는 매우 편리하지만, LLM이 필터를 잘못 생성할 수 있다는 점을 인지해야 한다. verbose=True로 설정하여 생성된 필터를 로깅하고, 잘못된 필터가 빈번하게 생성되면 메타데이터 스키마의 설명(description)을 더 상세하게 작성하는 것이 효과적이다.

여러 전략의 조합

실무에서는 단일 전략만 사용하는 경우가 드물다. 가장 효과적인 조합 패턴은 다음과 같다:

PYTHON 
from langchain.retrievers import EnsembleRetriever, ContextualCompressionRetriever
from langchain_community.retrievers import BM25Retriever
from langchain_cohere import CohereRerank

# 1단계: 하이브리드 검색 (BM25 + Vector)
bm25_retriever = BM25Retriever.from_documents(docs, k=10)
vector_retriever = store.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 10, "fetch_k": 30, "lambda_mult": 0.7}
)
ensemble = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.4, 0.6],
)

# 2단계: Re-ranking
reranker = CohereRerank(model="rerank-v3.5", top_n=4)
final_retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=ensemble,
)

# 실행: 하이브리드 검색 → MMR로 다양성 확보 → Re-ranking으로 정밀 순위화
results = final_retriever.invoke("서비스 아키텍처 변경 이력")
WARNING

여러 전략을 겹겹이 쌓을수록 지연시간과 비용이 증가한다. 반드시 간단한 것부터 시작하여 효과를 측정하고, 필요한 만큼만 추가해야 한다. 기본 벡터 검색 → 하이브리드 검색 → Re-ranking 순서로 하나씩 추가하면서 각 단계에서 실제로 품질이 개선되는지 평가(11장 RAGAS)하는 것이 올바른 접근법이다.

다음 단계

이 장에서는 Naive RAG의 검색 한계를 분석하고, MMR, 하이브리드 검색, Re-ranking, Contextual Compression, 메타데이터 필터링까지 검색 품질을 프로덕션 수준으로 끌어올리는 핵심 기법들을 다뤘다.

이 기법들은 검색 단계(Retrieval)의 최적화에 집중한다. 다음 장에서는 한 단계 더 나아가, 검색 전(Pre-retrieval)에 쿼리 자체를 변환하거나, 검색 결과를 LLM이 스스로 평가하고 보정하는 고급 RAG 패턴을 살펴본다. Multi-Query, HyDE, Corrective RAG, Self-RAG, Agentic RAG 등 최신 연구와 실무에서 검증된 패턴들을 깊이 있게 다룰 것이다.
