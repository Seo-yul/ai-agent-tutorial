RAG란 무엇인가

Retrieval-Augmented Generation의 개념, 필요성, 아키텍처를 이해한다

RAG란?

RAG(Retrieval-Augmented Generation)는 LLM에 외부 지식을 주입하여 더 정확하고 신뢰할 수 있는 답변을 생성하는 기법이다. 2020년 Meta AI(구 Facebook AI Research)의 Patrick Lewis 등이 발표한 논문 "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"에서 처음 제안되었다.

핵심 아이디어는 단순하다: "LLM이 모든 것을 암기하지 않아도 된다. 필요할 때 관련 정보를 찾아서 제공하면 된다."

사람이 시험을 볼 때 오픈북(open-book)으로 보는 것과 같은 원리이다. 모든 내용을 외우지 않아도, 필요한 정보를 빠르게 찾아서 참고하면 훨씬 정확한 답을 낼 수 있다. RAG는 LLM에게 이런 "오픈북"을 제공하는 기술이다. 

graph LR
    subgraph without ["기존 LLM (Closed-book)"]
        Q1["질문"] --> LLM1["LLM"]
        LLM1 --> A1["답변 (학습 데이터에만 의존)"]
    end

    subgraph with ["RAG (Open-book)"]
        Q2["질문"] --> R["Retriever (검색)"]
        R --> KB["외부 지식 베이스"]
        KB --> CTX["관련 문서"]
        CTX --> LLM2["LLM + 컨텍스트"]
        Q2 --> LLM2
        LLM2 --> A2["답변 (근거 기반)"]
    end

위 다이어그램에서 볼 수 있듯이, 기존 LLM은 질문을 받으면 학습 시점에 암기한 정보만으로 답변을 생성한다. 반면 RAG는 질문에 관련된 문서를 외부 지식 베이스에서 먼저 검색(Retrieve)한 뒤, 그 문서를 컨텍스트로 함께 제공하여 LLM이 근거 기반의 답변을 생성(Generate)하도록 한다.

INFO

RAG는 "검색(Retrieval)"과 "생성(Generation)"의 결합이라는 점에서, 전통적인 정보 검색(IR) 시스템과 생성형 AI의 장점을 모두 취하는 접근법이다. 검색 시스템의 정확성과 LLM의 자연어 이해/생성 능력을 동시에 활용한다.

왜 RAG가 필요한가?

LLM은 놀라운 능력을 가지고 있지만, 근본적인 한계도 존재한다. RAG가 왜 필요한지 이해하려면 먼저 이 한계들을 파악해야 한다.

1. 환각(Hallucination)

LLM의 가장 심각한 문제 중 하나는 환각(Hallucination)이다. LLM은 모르는 것을 "모른다"고 말하지 않고, 그럴듯하지만 사실이 아닌 정보를 자신있게 지어낸다. 존재하지 않는 논문을 인용하거나, 없는 법률 조항을 만들어내거나, 가상의 통계를 제시한다.

RAG를 사용하면 LLM이 실제 문서를 근거로 답변을 생성하므로 환각을 크게 줄일 수 있다. 답변에 출처를 명시할 수도 있어 사용자가 정보의 신뢰성을 직접 확인할 수 있다.

2. 지식 Cutoff

LLM은 학습 데이터의 시점까지만 세상을 알고 있다. 예를 들어 2024년 1월까지의 데이터로 학습된 모델은 2024년 2월 이후에 일어난 사건에 대해 전혀 알지 못한다. "오늘 주가가 얼마인지", "최신 라이브러리 버전이 무엇인지" 같은 질문에 답할 수 없다.

RAG는 최신 문서를 지식 베이스에 지속적으로 업데이트함으로써 이 문제를 해결한다. 모델을 재학습하지 않아도, 지식 베이스만 갱신하면 최신 정보를 반영할 수 있다.

3. 내부 지식 부재

LLM은 공개된 인터넷 데이터로 학습되므로, 기업의 내부 문서나 비공개 데이터에 대해서는 전혀 알지 못한다. 사내 위키, 내부 API 문서, 고객 데이터, 계약서 등 비공개 정보에 기반한 질문에는 답할 수 없다.

RAG를 통해 이러한 내부 문서를 지식 베이스로 구축하면, LLM이 조직 고유의 정보에 기반한 답변을 생성할 수 있게 된다.

Fine-tuning vs RAG

"LLM에 새로운 지식을 가르치려면 Fine-tuning을 하면 되지 않나?"라고 생각할 수 있다. 두 접근법의 차이를 비교해보자.

비교 항목 Fine-tuning RAG
비용 높음 (GPU 비용, 학습 시간) 낮음 (임베딩 한 번 + 검색 비용)

환각 제어 어려움 (학습해도 환각 발생) 용이 (출처 명시 가능)
데이터 보안 모델에 데이터가 내재화됨 데이터와 모델 분리, 접근 제어 가능
적합한 용도 말투/형식 변경, 특수 태스크 학습 지식 기반 질의응답, 문서 검색

결론적으로, Fine-tuning은 모델의 행동 방식을 바꿀 때 적합하고, RAG는 모델의 지식을 확장할 때 적합하다. 물론 두 기법을 함께 사용할 수도 있다. Fine-tuning으로 도메인 특화된 이해력을 높이고, RAG로 최신 지식을 공급하는 조합이 가장 강력하다.

TIP

실무에서는 대부분 RAG부터 시작하는 것이 좋다. Fine-tuning은 데이터 준비, 학습, 평가에 상당한 시간과 비용이 필요하다. RAG는 기존 LLM을 그대로 활용하면서도 조직 고유의 지식을 즉시 반영할 수 있다.

RAG 아키텍처 개요

RAG 시스템은 크게 두 개의 파이프라인으로 구성된다: Indexing Pipeline(색인 파이프라인)과 Query Pipeline(질의 파이프라인)이다.

Indexing Pipeline (오프라인)

문서를 검색 가능한 형태로 변환하여 저장하는 과정이다. 한 번 수행하면 여러 질의에 재사용할 수 있으므로 "오프라인" 파이프라인이라고도 부른다.

  1. 문서 로딩(Load): PDF, HTML, Markdown 등 다양한 형식의 원본 문서를 읽어들인다.
  2. 청킹(Chunk): 긴 문서를 의미 있는 작은 단위(chunk)로 분할한다. LLM의 컨텍스트 윈도우 제한과 검색 정밀도를 위해 필수적인 단계이다.
  3. 임베딩(Embed): 각 청크를 벡터(숫자 배열)로 변환한다. 의미적으로 유사한 텍스트는 벡터 공간에서 가까이 위치하게 된다.
  4. 저장(Store): 생성된 벡터를 벡터 데이터베이스에 저장한다.

Query Pipeline (온라인)

사용자의 질문을 받아 관련 문서를 검색하고 답변을 생성하는 과정이다. 실시간으로 수행되므로 "온라인" 파이프라인이라고 부른다.

  1. 질문 임베딩: 사용자의 질문을 동일한 임베딩 모델로 벡터화한다.
  2. 검색(Retrieve): 질문 벡터와 가장 유사한 벡터를 가진 청크들을 벡터 DB에서 검색한다.
  3. 프롬프트 구성: 검색된 청크(컨텍스트)와 원래 질문을 합쳐 프롬프트를 만든다.
  4. 생성(Generate): LLM이 컨텍스트를 참고하여 답변을 생성한다.

INFO

Indexing Pipeline에서 사용하는 임베딩 모델과 Query Pipeline에서 사용하는 임베딩 모델은 반드시 동일해야 한다. 서로 다른 모델을 사용하면 벡터 공간이 달라져 검색 품질이 급격히 저하된다.

RAG의 발전 단계

RAG 기술은 등장 이후 빠르게 발전해왔다. 크게 세 가지 단계로 나눌 수 있다: Naive RAG, Advanced RAG, Modular RAG.

Naive RAG

가장 기본적인 RAG 구현이다. 문서를 청킹하고 임베딩하여 벡터 DB에 저장한 뒤, 질문이 들어오면 단순 유사도 검색으로 관련 문서를 찾아 LLM에 전달하는 구조이다.

Naive RAG의 한계:

  • 검색 품질 문제: 사용자의 질문이 모호하거나 짧으면 관련 없는 문서가 검색된다.
  • 중복 문서: 유사한 내용의 청크가 여러 개 검색되어 컨텍스트가 낭비된다.
  • 컨텍스트 부족/과잉: 너무 적은 문서를 가져오면 답변이 불완전하고, 너무 많으면 LLM이 핵심을 놓친다.
  • 청킹 품질: 기계적으로 자른 청크가 의미 단위와 맞지 않아 정보가 잘리는 경우가 있다.

Advanced RAG

Naive RAG의 한계를 극복하기 위해 검색 전(Pre-retrieval), 검색(Retrieval), 검색 후(Post-retrieval) 각 단계에서 최적화를 추가한 접근법이다.

  • Pre-retrieval (검색 전 최적화):
    • 쿼리 변환(Query Transformation): 사용자의 원래 질문을 검색에 더 적합한 형태로 변환한다. HyDE(Hypothetical Document Embeddings), 쿼리 분해, 쿼리 확장 등의 기법이 있다.
    • 쿼리 라우팅(Query Routing): 질문 유형에 따라 적절한 검색 소스를 선택한다.
  • Retrieval (검색 최적화):
    • 하이브리드 검색(Hybrid Search): 벡터 검색(의미 기반)과 키워드 검색(BM25)을 결합하여 검색 정확도를 높인다.
    • Re-ranking: 검색된 문서를 Cross-encoder 등으로 재순위화하여 가장 관련성 높은 문서를 상위에 배치한다.
  • Post-retrieval (검색 후 최적화):
    • 컨텍스트 압축(Compression): 검색된 문서에서 질문과 관련 없는 부분을 제거하여 LLM에 핵심 정보만 전달한다.
    • 중복 제거(Deduplication): 의미적으로 중복되는 청크를 필터링한다.

Modular RAG

RAG의 각 컴포넌트를 독립적인 모듈로 구성하여, 필요에 따라 자유롭게 조합하는 접근법이다. 검색기, 생성기, 라우터, 평가기 등이 각각 독립 모듈로 존재하며, 파이프라인을 레고 블록처럼 조립할 수 있다. LangChain, LlamaIndex 같은 프레임워크가 이러한 모듈러 구조를 지원한다.

RAG vs Long Context

최신 LLM들은 컨텍스트 윈도우가 급격히 확대되고 있다. Claude는 200K 토큰, Gemini는 1M~2M 토큰을 지원한다. 이렇게 긴 컨텍스트를 사용할 수 있다면, 문서를 통째로 넣으면 되지 않을까? RAG가 정말 필요한 것일까?

Long Context의 한계

1. 비용 문제

LLM API는 입력 토큰 수에 비례하여 과금된다. 100페이지 분량의 문서를 매 질문마다 통째로 넣으면 토큰 비용이 폭증한다. RAG는 관련된 몇 개의 청크만 전달하므로 비용이 훨씬 적다.

2. "Needle in a Haystack" 문제

긴 컨텍스트에서 LLM이 중간에 위치한 정보를 놓치는 현상이 알려져 있다. 이를 "Lost in the Middle" 문제라고 한다. 컨텍스트의 시작과 끝 부분의 정보는 잘 활용하지만, 중간에 묻힌 핵심 정보를 무시하는 경향이 있다. RAG는 관련 정보만 추출하여 제공하므로 이 문제를 우회한다.

3. 규모의 한계

아무리 컨텍스트 윈도우가 크더라도, 수천 개의 문서나 수백 MB의 데이터를 한 번에 넣을 수는 없다. 기업의 전체 문서 저장소, 수년치 회의록, 대규모 코드베이스 등은 어떤 LLM의 컨텍스트 윈도우로도 감당할 수 없다.

4. 속도

입력 토큰이 많을수록 LLM의 응답 시간이 길어진다. 200K 토큰의 컨텍스트를 처리하는 것보다 2K 토큰의 핵심 정보를 처리하는 것이 훨씬 빠르다.

결론: 보완 관계

Long Context와 RAG는 서로를 대체하는 것이 아니라 보완하는 관계이다. 실무에서는 둘을 결합하여 사용하는 것이 가장 효과적이다.

  • RAG로 관련 문서를 먼저 검색하여 후보군을 좁힌다.
  • 검색된 문서가 여러 개이면 Long Context에 함께 넣어 LLM이 종합적으로 판단하게 한다.
  • 이렇게 하면 비용도 절감하면서 답변 품질도 높일 수 있다.
WARNING

"컨텍스트 윈도우가 크니까 RAG가 필요 없다"는 흔한 오해이다. Long Context는 만능이 아니며, 비용, 속도, 정확도 측면에서 RAG와 함께 사용해야 프로덕션 수준의 시스템을 구축할 수 있다.

이 튜토리얼에서 만들 것

이 튜토리얼을 통해 PDF 문서 기반 RAG 질의응답 시스템을 처음부터 끝까지 구축할 것이다. 단순한 프로토타입이 아니라, Kubernetes에 배포 가능한 프로덕션 수준의 시스템을 목표로 한다.
